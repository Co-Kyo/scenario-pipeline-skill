# Step ⑤: 四维评估

## 目的

用四维评估矩阵对每个命题打分，确定优先级。

## 前置条件

⛔ 加载 `core/scenario-matrix.md`（四维评估矩阵方法论）。

## 输入

- `decompositions.json`（Step ② 产出）
- `capability-graph.json`（Step ③ 产出，不含 highgrounds）
- `raw-materials.json`（Step ① 产出）

## 执行步骤

### 1. 逐命题四维打分

对每个命题，按以下四个维度打分（1-3 分）：

| 维度 | 含义 | 3 分标准 | 2 分标准 | 1 分标准 |
|------|------|---------|---------|---------|
| 跨栈耦合 | 涉及多少技术层 | ≥3 层 | 2 层 | 1 层 |
| 文档真空 | 官方文档覆盖度 | 几乎无文档 | 部分文档 | 文档完善 |
| 经验壁垒 | 需要多少实战经验 | 纯理论可理解 | 需少量实践 | 需大量实战 |
| 时事热度 | 当前面试/技术热点 | 高频面试题 | 偶尔出现 | 冷门 |

### 2. 计算总分 + 判定优先级

```
总分 = 四维之和（4-12）
优先级：10-12 → high, 7-9 → medium, 4-6 → rejected
```

**优先级决策记录**：每个命题必须记录 `priority_trace`，说明总分与阈值的对比：
```
"总分{X}（跨栈耦合{a}+文档真空{b}+经验壁垒{c}+时事热度{d}），阈值判定：{X}≥8 → high / 6≤{X}<8 → medium / {X}<6 → rejected"
```
如果触发一票入池条件，额外记录：`"触发一票入池：{具体条件}"`

### 3. 入池命题学习难度评估

对优先级为 `high` 的命题，基于原子能力评估其学习掌握难度。

**评估依据**（按权重从高到低）：

1. **能力依赖链深度**（主要权重）
   从 `capability-graph.json` 的 `dependency_graph` 出发，对命题涉及的每个能力计算依赖深度：
   - 无依赖的能力 → depth=0（如 A1、A19、A20）
   - 依赖 1 层的能力 → depth=1（如 A3 依赖 A1）
   - 依赖 2 层的能力 → depth=2（如 A5 依赖 A3→A1）
   - 命题的 max_depth = max(所有涉及能力的 depth)
   - max_depth 越高，学习阶梯越长，难度越大

2. **涉及能力数量与耦合度**
   - 能力数量多 → 知识面广
   - coupling 高（框架绑定度高）→ 换技术栈成本高

3. **AI 综合判断**
   基于上述数据，对每个命题给出 `low` / `medium` / `high` 三档难度评估。

**评估 prompt**（对每个入池命题执行）：

```
你是前端可视化领域的技术面试专家。请评估以下命题的学习掌握难度。

## 命题信息
- 命题：{proposition}
- 涉及原子能力：{capability_ids}
- 能力依赖链：{dependency_chain_description}

## 评估维度

1. **依赖链深度**：该命题涉及的能力中，最长的前置依赖链有几层？
   - 0 层（所有能力都无前置）→ 偏低
   - 1-2 层 → 中等
   - 3+ 层 → 偏高

2. **知识跨度**：需要同时理解多少个不同技术层的能力？
   - 1-2 个技术层 → 偏低
   - 3 个技术层 → 中等
   - 4+ 个技术层 → 偏高

3. **概念抽象度**：核心概念是否需要心智模型的跳跃？
   - 操作层面（API 用法、配置项）→ 偏低
   - 机制层面（渲染管线、事件模型）→ 中等
   - 设计层面（架构模式、引擎抽象）→ 偏高

## 输出格式
只输出一个 JSON：
{
  "difficulty": "low" | "medium" | "high",
  "reason": "一句话说明主要难度来源"
}
```

将评估结果写入 evaluations.json 每个命题的 `difficulty` 和 `difficulty_reason` 字段。

### 4. 学习顺序编排

根据难度评估结果，为入池命题编排推荐学习顺序：

```
排序规则：
1. low → medium → high（先易后难）
2. 同难度内，被其他命题依赖更多的排前面（prerequisite_count 降序）
3. 同难度同依赖，经验壁垒低的排前面
```

输出 `recommended_order`（从 1 开始的序号）。

### 5. 写入

按 `meta/output-contracts.md` §5 的示例格式，写入 `{workDir}/.meta/evaluations.json`。

## 输出

- 文件：`{workDir}/.meta/evaluations.json`
- 摘要（stdout）：命题评估表 + 优先级分布 + 难度分级 + 推荐学习顺序

## 校验清单

- [ ] 每个命题包含四个维度的评分
- [ ] 每个评分在 1-3 范围内
- [ ] 总分 = 四维之和
- [ ] 优先级判定与总分对应
- [ ] reasoning 字段解释了打分理由
- [ ] 每个命题包含 priority_trace（记录总分与阈值的对比判定过程）
- [ ] 入池命题包含 difficulty（low/medium/high）
- [ ] 入池命题包含 recommended_order
- [ ] recommended_order 按难度分级排序

## 异常处理

| 场景 | 处理 |
|------|------|
| 信息不足以打分 | 标记为 medium，reasoning 说明信息不足 |
| 所有命题 rejected | 提示用户调整搜索范围 |
| 难度评估信息不足 | 依赖链深度为主要依据，标记 difficulty_reason 说明信息不足 |

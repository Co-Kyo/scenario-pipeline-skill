# Workflow Core：复合工程场景评估框架

> 前处理与后处理的共享内核。前处理用它做筛选判别，后处理用它做结构化展开。

---

## 一、四维评估矩阵

对任意技术命题，从以下四个维度打分（每项 0-3 分，总分 12）：

| 维度 | 评估问题 | 0 分 | 1 分 | 2 分 | 3 分 |
|------|---------|------|------|------|------|
| **跨栈耦合** | 是否横跨 2 个以上技术层？ | 单一技术点 | 跨 2 层但主次分明 | 跨 3 层且相互影响 | 跨 4+ 层，链路复杂 |
| **文档真空** | 是否存在覆盖全貌的权威文档？ | 官方文档完备（MDN/规范有专页） | 有零散博客但无全貌（Google 结果 ≥ 3 篇相关） | 需拼凑多个来源（需跨 3+ 篇文章才能拼出全貌） | 几乎只能靠实践积累（Google 首页无直接相关结果） |
| **经验壁垒** | 不做能否凭空理解？ | 阅读即懂（看文档可复现） | 需跑一次 demo（有教程但需动手验证） | 需在真实项目中踩坑（问题在特定业务规模下才暴露） | 需要大规模业务场景才暴露（百万级 DAU / 复杂部署环境） |
| **时事热度** | 近期出现频率是否上升？ | 冷门稳定（近 6 月无新讨论） | 偶尔出现（季度内有 1-2 次讨论） | 多平台有讨论（近 1 月内 3+ 平台出现） | 高频出现在面试/社区/招聘（近 1 周内密集出现） |

### 打分校准规则

执行以下校准规则：

1. **文档真空**打分必须以 Google 搜索结果为客观依据：
   - 0 分：搜索命题关键词，首页有 MDN/规范/官方文档直接覆盖
   - 1 分：首页有博客但无完整链路说明
   - 2 分：需翻到第 2-3 页，跨 3+ 篇拼凑
   - 3 分：首页无直接相关结果

2. **经验壁垒**打分必须锚定具体场景规模：
   - 2 分及以上必须注明"在什么规模/条件下才会暴露"
   - 如无法描述具体暴露条件，最高给 1 分

3. **防虚高兜底**：若 4 个维度均为 2 分及以上，Agent 必须重新审视评分，至少 1 个维度应压低 1 分，除非能给出明确论据。

### 评分应用

| 总分 | 判定 | 前处理动作 | 后处理深度 |
|------|------|-----------|-----------|
| ≥ 6 | **高价值命题** | 直接入池 `high` | deepResearch 完整四象限 |
| 4-5 | **中等价值** | 入池 `medium`，待人工确认 | 可选：仅做链路解构 + trade-off |
| ≤ 3 | **不达标** | 记录理由后跳过 | 不触发后处理 |

### 一票入池条件

即使总分不达标，满足以下任一条件直接入池：

- 该命题在 **2 个以上不同信息源** 同时被讨论
- 包含明确的 **Trade-off 决策**（"用 A 还是用 B"）
- 涉及 **新兴技术与既有体系的碰撞**（如 AI + 前端、Rust 工具链 + JS 生态）

## 二、学习难度评估

> 四维评估筛选出"值得研究"的命题后，还需区分"从哪个开始学"。
> 基于原子能力的依赖链深度推断命题的学习掌握难度。

### 评估依据

| 维度 | 权重 | 说明 |
|------|------|------|
| 能力依赖链深度 | 主要 | 从 `dependency_graph` 计算命题涉及能力的最大依赖层数。depth 越高，学习阶梯越长 |
| 知识跨度 | 参考 | 涉及多少个不同技术层的能力。层数越多，需要的心智模型切换越多 |
| 概念抽象度 | 参考 | 操作层面（API）< 机制层面（渲染管线）< 设计层面（架构模式） |

### 难度分级

| 级别 | 标签 | 典型特征 |
|------|------|---------|
| 🟢 低 | `low` | 涉及能力依赖链 depth ≤ 1，操作/机制层面，有前端基础即可上手 |
| 🟡 中 | `medium` | 涉及能力依赖链 depth=2，或跨 3 个技术层，需额外学习特定 API 或机制 |
| 🔴 高 | `high` | 涉及能力依赖链 depth ≥ 3，或需全新知识体系（GLSL/引擎设计），或跨 4+ 技术层 |

### 学习顺序规则

1. 🟢 low → 🟡 medium → 🔴 high
2. 同难度内，被其他命题依赖更多的排前面
3. 同难度同依赖，经验壁垒低的排前面

## 三、四象限研究框架

对通过筛选的命题，按以下四个象限展开深度研究：

### 象限 I：链路解构 (Linkage Deconstruction)

```
穷举该场景落地的全链路节点。从构建时（Build-time）到运行时（Runtime），
从客户端（Client）到服务端及网关层（Server/Nginx），梳理各节点是如何共同
推动该场景生效的。

输出：有序列表，每个节点标注技术栈归属和关键配置项。
```

### 象限 II：极致场景与降级策略 (Extreme Scenarios & Fallbacks)

```
假设处于最复杂的极端业务场景（海量动态突变数据、极差网络、低端设备），
罗列常规方案必定会失效的坑点与边界 Case，给出从简单到极端的降级路线。

输出：每个坑点标注触发条件、表现症状、检测手段。
```

### 象限 III：Trade-off 权衡雷达 (Trade-off Radar)

```
列出 2-3 种技术路线，从以下维度对比：
- 性能（首屏/运行时/内存）
- 开发与维护成本
- 平台兼容性（浏览器/小程序/跨端）
- 侵入性（对现有架构的改造幅度）

输出：表格形式，每种方案标注其"牺牲点"。
```

### 象限 IV：最低成本实验指南 (Minimal Viable Experiment)

```
设计一个 15 分钟内本地跑通的最简代码环境。不需要业务逻辑，只需核心机制代码，
足以通过运行、抓包或断点直观感受该机制的核心难点。

【强制输出项】：

1. 环境基线声明（必须放在 README.md 顶部）
   - 运行时版本：如 Node v20.x / Chrome v121+
   - 包管理器：如 pnpm@8.x
   - 操作系统约束（如有）

2. 依赖锁定
   - package.json 中所有依赖必须使用精确版本号（禁止 ^ ~ latest）
   - 或提供 lockfile（pnpm-lock.yaml / package-lock.json）

3. 一键启动命令
   - 必须提供单条命令完成安装+启动（如 pnpm install && pnpm dev）
   - 禁止要求用户手动配置环境变量或修改文件

4. 验证检查点
   - 列出 3-5 个可观测的验证点（如"终端输出 XXX""浏览器 DevTools Network 面板可见 YYY"）
   - 每个检查点标注预期结果

5. 故障排除
   - 列出最常见的 2-3 个安装/运行失败原因及解决方法
```

## 四、命题命名规范

```
<技术域>：<具体场景描述>——<业务约束/触发背景>
```

示例：
- `长列表：AI 聊天窗口的虚拟列表渲染——10 万条消息，动态高度，滚动锚定`
- `缓存：HTTP 强缓存与协商缓存在 Nginx + CDN + Service Worker 三层的协作机制`
- `手势：小程序外卖 App 下拉二楼的实现与手势冲突处理`

粒度原则：
- ✅ 一个命题 = 一个可独立研究的工程问题
- ❌ 过粗："前端性能优化"（应拆分）
- ❌ 过细："Nginx expires 指令配置"（应归入更大命题）

# 核心能力：架构分词（Architecture Decomposition）

> 评估框架的前置动作。任何命题进入评估矩阵之前，必须先经过架构分词。
> 前处理用它标注命题的内部结构，后处理用它分配研究内容的权重。

---

## 一、何时执行

| 阶段 | 触发条件 | 作用 |
|------|---------|------|
| 前处理 · 命题提取后 | 每个候选命题 | 标注内部结构，修正跨栈耦合评分 |
| 后处理 · 场景解析时 | 每个研究命题 | 确定四象限研究的覆盖范围和内容权重 |

---

## 二、分词方法

### Step 1：识别限定词与技术栈关键词

从命题中区分两类词：

- **限定词**：指定框架/平台/工具的词（Vue、React、小程序、Nuxt...）
- **技术栈关键词**：指向底层工程机制的词（虚拟列表、缓存、沙箱、diff...）

```
命题："Vue3 虚拟列表：万级数据长列表渲染"
限定词：Vue3（框架层，指定实现载体）
技术栈关键词：虚拟列表（算法+浏览器层，指向通用工程机制）
```

**限定词决定搜索范围和技术视角，技术栈关键词决定工程深度。**

### Step 2：向上展开依赖链

问：技术栈关键词**依赖**什么底层能力？

```
虚拟列表
  ├── 算法层：二分查找 / 前缀和 / 双游标
  ├── 浏览器层：DOM 渲染管线（Layout → Paint → Composite）
  ├── 内存层：DOM 节点创建/回收、GC 策略
  ├── 交互层：滚动事件、惯性滚动、触控板兼容
  └── 框架层：响应式系统如何驱动重渲染 ← 限定词在这里介入
```

### Step 3：向下展开被依赖链

问：技术栈关键词**被**哪些上层场景使用？

```
虚拟列表
  ├── 聊天应用（万级消息列表）
  ├── 电商后台（万级订单表格）
  ├── AI 对话窗口（万级 token 流式输出）
  └── 地图 POI 列表
```

### Step 4：分层标注（不拆分）

将命题中的技术要素标注到对应层级，**保持为一个命题**：

```
命题："Vue3 虚拟列表：万级数据长列表渲染"

结构标注：
  [通用内核]
    · 算法层：虚拟化算法（前缀和 vs 双游标）
    · 浏览器层：渲染管线与 DOM 节点数的关系
    · 内存层：DOM 节点回收与 GC 策略
    · 交互层：滚动锚定、惯性滚动

  [Vue 特化]
    · 框架层：Composition API 响应式联动
    · 框架层：ResizeObserver + measureElement
    · 生态层：@tanstack/vue-virtual 的使用

  关系：通用内核是深度来源，Vue 特化是实现载体
```

### Step 5：确定内容权重

基于分词标注，确定后处理产出中各部分的权重：

**默认权重规则**：

| 通用内核占比 | 条件 |
|------------|------|
| ≥ 80% | 通用层跨 3+ 技术栈层级，深度远超框架层 |
| 60-80% | 通用层跨 2-3 层，框架层有一定深度 |
| 40-60% | 通用层和框架层深度相当 |
| < 40% | 框架层深度为主，通用层为辅（纯框架特化命题） |

```
"Vue3 虚拟列表"
  通用内核：算法+浏览器+内存+交互 = 4 层
  Vue 特化：Composition API + 生态库 = 2 层
  → 通用占比 ≥ 80%
  → 开篇从 Vue 开发者痛点切入，主体讲通用工程，收尾回到 Vue 生态实践
```

## 四、分词对评分的影响

架构分词直接影响四维评分中的**跨栈耦合**维度：

| 场景 | 未分词评分 | 分词后评分 | 原因 |
|------|----------|----------|------|
| "Vue3 虚拟列表" | 跨栈耦合 2 | 跨栈耦合 3 | 通用内核跨算法+浏览器+内存+交互 4 层 |
| "Vue 微前端" | 跨栈耦合 3 | 跨栈耦合 3 | 微前端本身就是跨栈场景，不因 Vue 标签改变 |
| "Vue keep-alive" | 跨栈耦合 2 | 跨栈耦合 2 | 通用层仅跨组件+路由+生命周期，无更深层内核 |

**规则**：分词后如果发现被限定词遮蔽的通用工程层，跨栈耦合分数应上调。命题本身不拆分，只是评分更准确。

## 五、分词对后处理产出的影响

### 内容结构模板

分词后，四象限研究和系列文章的结构应遵循：

```
开篇（10-15%）：
  从限定词（Vue）开发者的真实痛点切入
  "你用 Vue 做过万级列表吗？遇到过滚动卡顿吗？"
  → 引出问题，建立读者共鸣

主体（70-80%）：
  通用工程内核的深度展开
  · 算法原理（不是 Vue 特定的）
  · 浏览器渲染机制（不是 Vue 特定的）
  · 内存管理策略（不是 Vue 特定的）
  · 通用避坑清单
  → 这是核心价值所在

收尾（10-15%）：
  回到限定词（Vue）的生态实践
  · Vue 生态中的最佳工具选择
  · Vue 特定的 API 使用技巧
  · 完整 Vue 实现代码
  → 让读者知道"在 Vue 中怎么落地"
```

## 六、边界情况

### 不需要分词的命题

如果命题本身就是纯框架特化层，分词后无通用内核：

```
"Vue3 computed 与 watch 的区别"
  → 分词：纯框架层（Vue 响应式系统）
  → 无深层通用工程内核
  → 按框架特化命题处理，通用占比 < 40%
  → 产出以 Vue API 讲解为主
```

### 分词后通用层为空

```
"Vue3 Teleport 组件的使用"
  → 分词：Teleport（框架 API）→ DOM 操作（浏览器层）
  → 但 Teleport 的 DOM 操作不涉及复杂工程
  → 通用层为空，按纯框架特化命题处理
```

### 分词后发现命题过粗

```
"前端性能优化"
  → 分词：构建优化 + 运行时优化 + 网络优化 + 渲染优化 + 内存优化
  → 每一层都是独立的复合工程场景
  → 应拆分为多个独立命题（这是粒度问题，不是分词问题）
```

### 限定词和内核是同一层

```
"HTTP 缓存协商机制"
  → 分词：HTTP（协议层）是限定词，缓存协商（协议层）也是内核
  → 无框架特化层，通用占比 100%
  → 直接按通用工程命题处理
```

## 七、完整示例

### 输入

用户指令：`扫描：3-5 年经验前端开发者 Vue 深度场景问题`

### 分词过程（单个命题）

```
原始命题："Vue3 虚拟列表：万级数据渲染"
限定词：Vue3
技术栈关键词：虚拟列表

向上展开：
  虚拟列表 → 算法（二分/前缀和）
  虚拟列表 → 浏览器渲染管线（Layout/Paint/Composite）
  虚拟列表 → 内存管理（DOM 节点 GC）
  虚拟列表 → 滚动交互（passive/rAF/锚定）

向下展开：
  虚拟列表 → 聊天/电商/AI 对话/地图

分层标注：
  [通用内核] 算法 + 浏览器 + 内存 + 交互（4 层）
  [Vue 特化] Composition API + @tanstack/vue-virtual（2 层）

内容权重：通用 ≥ 80%

评分修正：跨栈耦合 2 → 3

入池：一个命题，带结构标注
  命题：Vue3 虚拟列表：万级数据渲染
  结构：通用内核（算法+浏览器+内存+交互）+ Vue 特化（Composition API + 生态库）
  通用占比：≥ 80%
  评分：9/12（跨栈耦合 3 + 文档真空 1 + 经验壁垒 3 + 时事热度 2）
```

### 后处理产出结构

```
系列文章：Vue3 虚拟列表解构

  第 1 篇（开篇）：从 Vue 开发者的真实痛点切入
    "你用 Vue 渲染过万级列表吗？scroll 卡成 PPT 的那种？"

  第 2-4 篇（主体）：通用工程内核
    算法原理 → 浏览器渲染机制 → 内存管理 → 通用避坑
    （不依赖 Vue 知识也能读懂）

  第 5 篇（收篇）：Vue 生态实践
    @tanstack/vue-virtual 最佳实践 + 完整实现代码
    （限定词回归，给出落地方案）
```

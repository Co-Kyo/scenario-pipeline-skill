# Core Web Vitals — Trade-offs（方案对比）

## LCP 优化路线对比

### 路线 A：资源加载优化
**思路**：缩短资源从请求到可用的时间。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| `<link rel="preload">` | LCP 关键资源（图片/字体/CSS） | 可缩短 0.5-2s | 需精确识别关键资源，preload 过多反而争抢带宽 |
| `fetchpriority="high"` | LCP 图片 | 同类资源内优先级提升 | 仅在同类资源间有效，滥用失效 |
| `preconnect` | 第三方域名（CDN/字体/API） | 省去 DNS+TCP+TLS 时间 | 每页 ≤6 个，过多浪费连接资源 |
| 图片格式优化（WebP/AVIF） | 所有图片 | 体积减少 25-50% | 需 `<picture>` 兼容旧浏览器 |
| 响应式图片 `srcset` | 不同设备 | 避免下载过大图片 | 需维护多套图片资源 |

**适用场景**：LCP 瓶颈在网络下载阶段（TTFB 正常但资源加载慢）。

### 路线 B：渲染阻塞消除
**思路**：减少阻止浏览器开始渲染的资源。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| 内联 Critical CSS | 首屏渲染 | 消除 CSS 阻塞，FCP 提前 | 需工具提取，HTML 体积增大 |
| `defer`/`async` JS | 非关键 JS | DOM 解析不被阻塞 | defer 保序但延迟执行，async 不保序 |
| 代码拆分 Code Splitting | 大型 SPA | 减少首屏 JS 体积 | 需构建工具配置，增加请求次数 |
| HTTP/2 Server Push | 关键资源 | 服务器主动推送 | 配置复杂，可能过度推送 |

**适用场景**：LCP 瓶颈在渲染阶段（资源已下载但浏览器等待阻塞资源）。

### 路线 C：服务端优化
**思路**：缩短 TTFB 和 HTML 文档传输时间。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| SSR/SSG | 动态/静态页面 | HTML 直出，LCP 可达 0.5-1s | SSR 增加服务器成本，SSG 不适合动态数据 |
| CDN 边缘缓存 | 全球用户 | TTFB 降至 50-100ms | 需处理缓存失效策略 |
| 流式 HTML 传输 | 大页面 | 边生成边传输 | 需框架支持（React 18 Streaming SSR） |

**适用场景**：LCP 瓶颈在 TTFB 阶段（服务器响应慢）。

### 推荐策略
```
LCP > 4s → 先检查 TTFB（路线 C）
2.5s < LCP < 4s → 资源加载优化（路线 A）+ 渲染阻塞消除（路线 B）
LCP < 2.5s → 维持并监控
```

## CLS 治理策略对比

### 策略 A：预留空间
**思路**：为所有动态内容预先声明尺寸。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| `width`/`height` 属性 | 图片/视频 | 消除图片加载偏移 | 需知道准确尺寸 |
| `aspect-ratio` CSS | 响应式图片/视频 | 自适应宽度时保持比例 | 不支持 IE |
| 固定高度容器 | 广告位/通知 | 预留空间避免推移 | 内容过多时溢出 |
| `contain: layout` | 独立组件 | 限制布局影响范围 | 创建新 containing block，影响 fixed 定位 |
| `contain-intrinsic-size` | content-visibility 元素 | 预估屏幕外元素尺寸 | 估算不准仍会偏移 |

**适用场景**：CLS 来源是图片/广告/动态注入内容。

### 策略 B：字体优化
**思路**：消除 Web Font 加载导致的布局偏移。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| `font-display: optional` | 首屏文本 | 完全消除 FOUT | 字体可能不显示 |
| `font-display: swap` | 非首屏文本 | 快速显示 fallback | FOUT 明显 |
| `size-adjust` + `ascent-override` | 高保真场景 | fallback 字体 metrics 匹配 | 配置复杂 |
| `preload` 字体 | 关键字体 | 提前加载减少 swap 时间 | 需精确识别关键字体 |

**适用场景**：CLS 来源是 Web Font 加载。

### 策略 C：动画与过渡
**思路**：避免动画过程中产生布局偏移。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| `transform` 动画 | 位移/缩放/旋转 | 仅触发 Composite，无布局偏移 | 不改变文档流位置 |
| `opacity` 动画 | 淡入淡出 | 仅触发 Composite | 不改变布局 |
| `will-change` 提示 | 即将动画的元素 | 预创建合成层 | 滥用导致内存暴涨 |
| CSS `transition` | 状态切换 | 平滑过渡减少视觉跳动 | 需声明过渡属性和时长 |

**适用场景**：CLS 来源是动画实现方式不当。

### 推荐策略
```
CLS > 0.25 → 全面排查，优先修复图片/广告空间预留（策略 A）
0.1 < CLS < 0.25 → 字体优化（策略 B）+ 动画优化（策略 C）
CLS < 0.1 → 维持并监控
```

## INP 改善手段对比

### 手段 A：任务分片
**思路**：将长任务切分为多个短任务，让出主线程。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| `scheduler.yield()` | 通用长任务 | 让出主线程，允许输入事件插入 | 新 API，需 polyfill |
| `requestIdleCallback` | 低优先级任务 | 空闲时执行 | Safari 不支持，高负载下不触发 |
| `setTimeout(fn, 0)` | 兼容性兜底 | 切分到下一个宏任务 | 精度低（≥4ms），频繁调用有开销 |
| `MessageChannel` | 精确切分 | 立即调度到下一个宏任务 | 比 setTimeout 精确 |

**适用场景**：INP 瓶颈是主线程长任务阻塞。

### 手段 B：计算卸载
**思路**：将计算密集任务从主线程移走。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| Web Worker | 大数据计算/解析 | 完全不阻塞主线程 | 无法访问 DOM，通信有序列化开销 |
| `requestAnimationFrame` 分帧 | 渲染相关计算 | 与渲染同步，每帧 ≤16ms | 不适合非渲染计算 |
| WASM | CPU 密集计算 | 接近原生性能 | 开发成本高，调试困难 |

**适用场景**：INP 瓶颈是计算密集型任务。

### 手段 C：事件处理优化
**思路**：减少事件处理函数本身的耗时。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| 事件委托 | 大量同类事件 | 减少监听器数量 | 事件冒泡路径变长 |
| 防抖 Debounce | 高频事件(input/scroll) | 减少处理次数 | 有延迟感 |
| 节流 Throttle | 持续触发事件(scroll) | 固定频率处理 | 丢失部分事件 |
| 避免强制同步布局 | 事件处理中读写布局 | 消除同步 Layout | 需重构代码结构 |

**适用场景**：INP 瓶颈在特定事件处理函数。

### 手段 D：第三方脚本治理
**思路**：减少第三方脚本对主线程的占用。

| 技术 | 适用场景 | 效果 | 代价 |
|------|---------|------|------|
| `defer`/`async` 加载 | 非关键第三方脚本 | 不阻塞 DOM 解析 | 执行时机不可控 |
| iframe 隔离 | 广告/聊天插件 | 独立线程，不阻塞主页面 | 跨 iframe 通信复杂 |
| 按需加载 | 低频使用的第三方功能 | 首屏不加载 | 首次使用有延迟 |
| Partytown | 第三方脚本 | Web Worker 中执行 | 需要配置，兼容性待验证 |

**适用场景**：INP 瓶颈是第三方脚本。

### 推荐策略
```
INP > 500ms → Long Task 拆分（手段 A）+ 第三方治理（手段 D）
200ms < INP < 500ms → 事件处理优化（手段 C）+ 计算卸载（手段 B）
INP < 200ms → 维持并监控
```

## 指标间冲突与权衡

### LCP vs CLS 冲突
- **场景**：为优化 LCP，延迟加载非首屏图片（`loading="lazy"`）；但图片未加载时高度为 0，加载后产生 CLS。
- **解法**：`loading="lazy"` 的图片必须声明 `width`/`height` 或 `aspect-ratio`。

### LCP vs INP 冲突
- **场景**：为优化 LCP，预加载大量资源；但资源解析和执行可能阻塞主线程，恶化 INP。
- **解法**：preload 控制在 5 个以内，JS 使用 `defer` 而非同步执行。

### CLS vs INP 冲突
- **场景**：为避免 CLS，使用 CSS transition 做平滑过渡；但 transition 期间浏览器需持续合成，可能影响输入响应。
- **解法**：使用 `transform`/`opacity` 动画（仅 Composite），不影响主线程。

### 通用 vs 框架特定
- **场景**：React 的 hydration 过程可能产生 Long Task（>50ms），恶化 INP；但完全避免 hydration 需要 SSR+流式架构。
- **解法**：React 18 `startTransition` 标记低优先级更新，避免阻塞输入事件。

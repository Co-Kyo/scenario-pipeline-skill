# P2-首屏白屏 — 方案对比

---

## 1. 渲染策略对比：SSR vs SSG vs CSR

| 维度 | SSR (服务端渲染) | SSG (静态生成) | CSR (客户端渲染) |
|------|------------------|----------------|------------------|
| **首屏 HTML 来源** | 服务端实时生成 | 构建时预生成 | 空 HTML，JS 生成 |
| **FCP** | ⭐⭐⭐ 快（HTML 已含内容） | ⭐⭐⭐ 最快（CDN 静态文件） | ⭐ 慢（需等 JS 下载执行） |
| **TTI** | ⭐⭐ 中（需 Hydration） | ⭐⭐ 中（需 Hydration） | ⭐⭐⭐ 最快（无 SSR 开销） |
| **TTFB** | ⭐ 较慢（服务端渲染耗时） | ⭐⭐⭐ 最快（CDN 直接返回） | ⭐⭐⭐ 最快（返回空壳） |
| **SEO** | ⭐⭐⭐ 优秀 | ⭐⭐⭐ 优秀 | ⭐ 差（需等 JS 渲染） |
| **服务端成本** | 高（每次请求都要渲染） | 低（构建时一次性） | 无 |
| **数据新鲜度** | ⭐⭐⭐ 实时 | ⭐ 构建时快照 | ⭐⭐⭐ 实时 |
| **适用场景** | 动态内容、个性化页面 | 博客、文档、营销页 | 后台管理系统、SPA |
| **典型框架** | Next.js / Nuxt SSR | Next.js SSG / Astro | Create React App / Vue SPA |

## 2. SSR 渲染模式对比：流式 vs 同步

| 维度 | Streaming SSR (流式) | Synchronous SSR (同步) |
|------|----------------------|----------------------|
| **首字节时间** | ⭐⭐⭐ 极快（先发 shell） | ⭐ 慢（等全部渲染完） |
| **首屏完整度** | ⭐⭐ 渐进式（Suspense 分段） | ⭐⭐⭐ 一次性完整 |
| **SEO** | ⭐⭐ 需确保爬虫拿到完整 HTML | ⭐⭐⭐ 天然完整 |
| **实现复杂度** | 高（Suspense boundary 设计） | 低（renderToString） |
| **Slow Data 场景** | ⭐⭐⭐ 优秀（不阻塞其他区域） | ⭐ 差（最慢数据阻塞全部） |
| **框架支持** | React 18 renderToReadableStream / Vue 3.5 | React renderToString / Vue renderToString |
| **适用场景** | 数据源延迟不均、长页面 | SEO 严格、页面简单 |

## 3. Hydration 策略对比：全量 vs Selective vs Lazy

| 维度 | 全量 Hydration | Selective Hydration | Lazy Hydration |
|------|---------------|--------------------|----|
| **交互就绪时间** | ⭐ 慢（全量 JS 执行完） | ⭐⭐ 分段就绪 | ⭐⭐⭐ 按需就绪 |
| **首屏交互区** | 全部一次性激活 | Suspense boundary 分组激活 | onInteraction/onVisible 激活 |
| **JS 执行量** | 100% | 按区域分批 | 最小化（仅交互时执行） |
| **实现复杂度** | 低 | 中（需设计 boundary） | 高（需标识交互区域） |
| **SEO 兼容** | ⭐⭐⭐ 完美 | ⭐⭐⭐ 完美 | ⭐⭐ 需确保静态 HTML 完整 |
| **框架支持** | React hydrateRoot / Vue app.mount | React Suspense + startTransition | Vue 3.5 onInteraction/onVisible/whenIdle |
| **适用场景** | 简单页面、强交互 | 长页面、混合交互密度 | 内容型页面、首屏优先 |

## 4. CSS 加载策略对比

| 维度 | 外部 <link> 全量 | 内联 Critical CSS + 异步 | CSS-in-JS |
|------|------------------|--------------------------|-----------|
| **渲染阻塞** | ⚠️ 阻塞 FCP | ⭐ 不阻塞 | ⭐ 不阻塞（运行时注入） |
| **首屏样式完整性** | ⭐⭐⭐ 完整 | ⭐⭐ 首屏完整，非首屏延迟 | ⭐⭐⭐ 运行时完整 |
| **缓存效率** | ⭐⭐⭐ 可独立缓存 | ⭐⭐ Critical CSS 内联无法缓存 | ⭐ 无法缓存 |
| **构建复杂度** | 低 | 高（需提取 Critical CSS） | 中 |
| **适用场景** | 通用 | 首屏性能敏感 | 组件化项目 |

## 5. JS 加载策略对比

| 维度 | 同步 <script> | defer | async | type="module" |
|------|--------------|-------|-------|---------------|
| **DOM 解析阻塞** | ⚠️ 阻塞 | ⭐ 不阻塞 | ⭐ 不阻塞 | ⭐ 不阻塞 |
| **执行时机** | 立即 | DOM 解析完成后 | 下载完立即 | 类似 defer |
| **执行顺序** | 文档顺序 | 文档顺序 | 不保证 | 文档顺序 |
| **首屏影响** | ⚠️ 延迟 FCP | ⭐ 无影响 | ⚠️ 可能中断解析 | ⭐ 无影响 |
| **适用场景** | 几乎不用 | 通用推荐 | 独立脚本 | ESM 项目 |

## 6. 资源加载优先级对比

| 维度 | preload | prefetch | preconnect | dns-prefetch |
|------|---------|----------|------------|--------------|
| **优先级** | 高 | 低 | 中 | 低 |
| **加载内容** | 当前导航关键资源 | 未来导航资源 | DNS+TCP+TLS 连接 | 仅 DNS 解析 |
| **带宽竞争** | 与首屏资源同优先级 | 不与首屏争抢 | 无（仅连接） | 无（仅 DNS） |
| **数量限制** | ≤5 个 | 适量 | ≤6 个 | 适量 |
| **首屏收益** | ⭐⭐⭐ LCP/FCP | ⭐ 下一屏 | ⭐⭐ TTFB | ⭐ TTFB |
| **误用风险** | CORS 不匹配→双重下载 | 过多→争抢带宽 | 过多→浪费资源 | 低 |

## 7. Code Splitting 粒度对比

| 维度 | 粗粒度（少量大 chunk） | 路由级中等粒度 | 细粒度（大量小 chunk） |
|------|----------------------|---------------|----------------------|
| **首屏请求数** | ⭐⭐⭐ 少 | ⭐⭐ 适中 | ⚠️ 多 |
| **缓存命中率** | ⭐ 低（改一处全量更新） | ⭐⭐ 中 | ⭐⭐⭐ 高（局部更新） |
| **HTTP 请求开销** | ⭐⭐⭐ 低 | ⭐⭐ 适中 | ⚠️ 高（HTTP/1.1 尤甚） |
| **首屏体积** | ⚠️ 可能包含非首屏代码 | ⭐⭐ 合理 | ⭐⭐⭐ 最小 |
| **推荐场景** | 小型项目 | 通用推荐 | HTTP/2+大型项目 |

## 8. CDN 部署架构对比

| 维度 | 单 CDN | Multi-CDN | Edge Computing |
|------|--------|-----------|----------------|
| **管理复杂度** | ⭐ 低 | ⚠️ 高 | ⭐⭐ 中 |
| **成本** | ⭐ 低 | ⚠️ 高（翻倍+） | ⭐⭐ 中 |
| **可用性** | ⭐⭐ 有单点风险 | ⭐⭐⭐ 高可用 | ⭐⭐⭐ 高可用 |
| **首屏延迟** | ⭐⭐ 区域优化 | ⭐⭐⭐ 全球最优 | ⭐⭐⭐ 全球最优+动态处理 |
| **动态能力** | ⭐ 无 | ⭐ 无 | ⭐⭐⭐ 路由/鉴权/转换 |
| **适用场景** | 中小项目 | 大型/金融/跨国 | 需要边缘逻辑 |

## 9. Node.js vs 边缘运行时对比

| 维度 | Node.js SSR | 边缘运行时 (Vercel Edge / Cloudflare Workers) |
|------|-------------|----------------------------------------------|
| **冷启动** | ⚠️ 慢（100ms+） | ⭐⭐⭐ 极快（<10ms） |
| **数据库访问** | ⭐⭐⭐ 完整支持 | ⚠️ 有限（需 HTTP API） |
| **API 兼容** | ⭐⭐⭐ Node.js 全量 API | ⚠️ Web API 子集 |
| **地理分布** | ⭐ 单区域部署 | ⭐⭐⭐ 全球边缘 |
| **TTFB** | ⭐⭐ 取决于部署位置 | ⭐⭐⭐ 用户就近 |
| **适用场景** | 数据密集、需数据库 | 延迟敏感、轻量逻辑 |

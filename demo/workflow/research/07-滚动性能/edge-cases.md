# 边界情况清单 — 滚动性能：长页面大量图片的懒加载与滚动优化（电商商品列表）

## EC-01: 快速滑动时懒加载图片大面积闪烁（P0）

**来源能力**: A9-IntersectionObserver 与懒加载 × B1 懒加载图片闪烁（状态跃迁）

**场景**: 用户在电商商品列表中快速上下滑动，大量商品卡片同时进出视口。由于 IntersectionObserver 的回调批量触发特性，图片 src 从占位符 → 加载中 → 完成 的状态跃迁在短时间内密集发生，导致页面出现大面积闪烁和视觉跳动。

**预期行为**: 图片平滑过渡，无明显闪烁或布局偏移。

**失败模式**: 用户快速滑动时，商品图片反复闪烁（占位符与真实图交替出现），列表出现明显卡顿。

**通用高地引用**:
- **G4-状态跃迁管理**: 懒加载图片处于"未加载 → 加载中 → 已加载"多状态机，快速滚动导致状态跃迁频繁且不完整，需要状态锁或防抖机制保证跃迁原子性。
- **G7-时序竞争**: IntersectionObserver 回调与图片加载完成事件之间存在时序窗口，回调可能在上一张图片尚未渲染完成时就触发下一批加载。

**验证步骤**:
1. 构建包含 500+ 商品卡片的列表页，每张图片使用 IntersectionObserver 懒加载。
2. 使用 Puppeteer 模拟快速滑动（速度 > 3000px/s），持续 10 秒。
3. 录制页面截图序列（每帧），统计连续帧中同一商品图片出现 src 切换的次数。
4. 预期：单张图片 src 切换次数 ≤ 2（占位→真实），无反向切换（真实→占位→真实）。
5. 如果出现反向切换或每帧多于 5 张图片同时处于加载中状态，则判定失败。

## EC-02: IntersectionObserver 回调阻塞主线程导致滚动卡顿（P1）

**来源能力**: A9-IntersectionObserver 与懒加载 × B2 IO 回调阻塞（时序竞争）

**场景**: 视口内同时存在大量商品卡片，当用户快速滚动到新区域时，IntersectionObserver 一次性回调大量 entries（例如 50+ 条），回调函数内部执行 DOM 查询、样式计算或同步布局操作，阻塞主线程超过 16ms。

**预期行为**: 滚动帧率保持 ≥ 55fps，IO 回调不阻塞主线程超过单帧预算。

**失败模式**: IO 回调中执行了 `getBoundingClientRect()`、`offsetHeight` 等触发强制同步布局的代码，导致主线程阻塞 > 50ms，用户感知到明显卡顿。

**通用高地引用**:
- **G4-状态跃迁管理**: 批量回调中对每条 entry 执行同步 DOM 操作，放大了状态跃迁的开销。
- **G7-时序竞争**: 回调执行期间滚动事件仍在排队，回调越长排队越严重，形成滚雪球效应。

**验证步骤**:
1. 在商品列表中注册 IntersectionObserver（threshold=0, rootMargin='200px'）。
2. 在回调中模拟同步操作：`entries.forEach(e => e.target.getBoundingClientRect())`。
3. 使用 Chrome DevTools Performance 面板录制快速滚动 5 秒。
4. 检查主线程长任务（Long Task > 50ms）的数量。
5. 预期：长任务数量为 0。如果出现 ≥ 1 个长任务，判定失败。
6. 修复方向：将回调中的 DOM 操作推迟到 `requestIdleCallback` 或分批处理（每帧最多处理 5 条 entry）。

## EC-03: 长列表强制同步布局导致布局抖动（P0）

**来源能力**: A1-浏览器渲染管线 × B1 强制同步布局（时序竞争）

**场景**: 电商商品列表采用瀑布流或等高网格布局。在懒加载图片完成后，代码同步读取新加载图片的尺寸（`offsetHeight`、`clientWidth`）以更新容器高度或触发重新布局，此时浏览器被迫执行强制同步布局（Forced Synchronous Layout），阻塞渲染管线。

**预期行为**: 图片加载完成后不触发强制同步布局，布局更新与下一帧渲染合并。

**失败模式**: 每张图片 `onload` 回调中立即读取 `offsetHeight` 并设置容器 `style.height`，导致 N 次强制同步布局（N = 当前批次加载的图片数），主线程阻塞时间与 N 线性相关。

**通用高地引用**:
- **G4-状态跃迁管理**: 图片从"加载中"跃迁到"已加载"时触发尺寸读取，状态跃迁与布局计算耦合。
- **G7-时序竞争**: `onload` 在微任务队列中执行，与 `requestAnimationFrame` 的渲染帧产生时序竞争。

**验证步骤**:
1. 构建 200 个商品卡片的网格布局，图片使用懒加载。
2. 在每个图片 `onload` 中执行 `console.log(el.offsetHeight)` 和 `container.style.height = ...`。
3. 使用 Performance Observer 监听 `layout-shift` 事件和 `longtask` 事件。
4. 快速滚动到未加载区域，等待图片批量加载。
5. 预期：单次批量加载（5 张图片）期间，强制同步布局次数 ≤ 1。
6. 如果出现 ≥ 3 次强制同步布局（可通过 Performance 面板 Layout 事件确认），判定失败。

## EC-04: WebP/AVIF 不兼容浏览器回退时图片长时间空白（P1）

**来源能力**: A12-图片格式与压缩策略 × B1 格式兼容性（输入变异）

**场景**: 电商平台统一使用 WebP/AVIF 格式以节省带宽。部分旧版浏览器（如 iOS Safari < 14、IE11、部分国产浏览器内核）不支持这些格式，`<picture>` 标签的 `<source>` 匹配失败后回退到 `<img>` 的 fallback src，但 fallback 资源未正确配置或不存在。

**预期行为**: 不支持 WebP/AVIF 的浏览器自动回退到 JPEG/PNG，用户看到正常图片。

**失败模式**: 回退 src 指向 404 或仍为 WebP 格式，图片长时间显示占位符或破裂图标。在 Safari 13 上商品图片大面积缺失。

**通用高地引用**:
- **G1-输入变异**: 浏览器 UA、格式支持能力、网络环境等输入高度变异，单一格式策略无法覆盖全场景。
- **G3-资源边界**: CDN 配置未正确设置 Content-Type 和 Vary 头，导致缓存的 WebP 被错误返回给不支持的浏览器。

**验证步骤**:
1. 使用 BrowserStack 测试以下环境：iOS Safari 13、Chrome 32、Samsung Internet 12。
2. 加载使用 `<picture>` + WebP/AVIF 的商品列表页。
3. 检查每张商品图片是否成功渲染（非破裂图标、非空白占位）。
4. 预期：所有测试环境中图片 100% 正常显示。
5. 如果任一环境中 > 0% 的图片显示异常，判定失败。
6. 额外检查：CDN 响应头中 `Vary: Accept` 是否正确设置。

## EC-05: 未压缩大图直接加载导致首屏渲染严重延迟（P0）

**来源能力**: A12-图片格式与压缩策略 × B2 图片过大未压缩（资源边界）

**场景**: 运营上传商品图片时未经过压缩处理，单张图片原始尺寸达 3-5MB（4000×3000 像素的相机直出图）。虽然使用了懒加载，但用户滚动到该区域时，大图加载耗时过长（3G 网络下 > 5 秒），期间图片区域显示空白或极低质量占位。

**预期行为**: 图片加载时间在 3G 网络下 ≤ 2 秒，展示适当的加载态。

**失败模式**: 单张商品图片 5MB，3G 网络（750kbps）加载耗时 > 50 秒，用户滚动到该区域后长时间等待，触发浏览器图片解码 OOM 或内存警告。

**通用高地引用**:
- **G3-资源边界**: 图片文件大小超出合理带宽预算，需要设定上限并强制压缩。
- **G1-输入变异**: 用户上传的图片来源多样（手机、相机、截图），尺寸和质量差异极大。

**验证步骤**:
1. 准备测试图片集：包含 100KB、500KB、2MB、5MB 四档大小的商品图。
2. 使用 Chrome DevTools Network 节流至 Slow 3G（400kbps, 2000ms RTT）。
3. 逐一加载不同大小的图片，记录 LCP 时间和图片完全加载时间。
4. 预期：所有图片文件大小 ≤ 200KB（WebP）或 ≤ 500KB（JPEG），加载时间 ≤ 3 秒。
5. 如果存在单张图片 > 1MB 且加载时间 > 5 秒，判定失败。
6. 修复方向：服务端强制裁剪至最大 1200px 宽，WebP 压缩质量 75-80。

## EC-06: CSS contain 使用不当导致合成层爆炸（P1）

**来源能力**: A13-CSS contain 与合成层 × B1 层爆炸（资源边界）

**场景**: 为优化滚动性能，对商品卡片批量添加 `contain: layout paint style`，同时卡片内图片使用了 `will-change: transform` 或 `transform: translateZ(0)` 强制提升为合成层。当列表中有 500+ 张图片时，GPU 合成层数量暴增，显存占用超限，触发浏览器层合并或直接回退到软件渲染。

**预期行为**: 合成层数量控制在合理范围（< 50 个活跃层），GPU 内存占用 < 256MB。

**失败模式**: 500 个商品卡片全部提升为独立合成层，Chrome 任务管理器显示 GPU 内存 > 1GB，页面出现严重卡顿甚至白屏。

**通用高地引用**:
- **G3-资源边界**: 合成层数量受 GPU 显存限制，需要设定上限并动态管理。
- **G4-状态跃迁管理**: 图片加载完成后需要从"占位层"跃迁到"真实层"，层的创建和销毁需要有序管理。

**验证步骤**:
1. 构建 500 商品卡片列表，每张卡片设置 `contain: layout paint`。
2. 为所有卡片内的图片添加 `will-change: transform`。
3. 打开 Chrome DevTools → Layers 面板，统计活跃合成层数量和 GPU 内存占用。
4. 快速滚动完整列表一遍。
5. 预期：活跃合成层数 ≤ 50，GPU 内存 ≤ 256MB。
6. 如果合成层数 > 100 或 GPU 内存 > 512MB，判定失败。
7. 修复方向：仅对视口内 ±3 屏的图片提升为合成层，滚动出视口后移除 `will-change`。

## EC-07: 懒加载占位符尺寸不一致导致 CLS 飙升（P0）

**来源能力**: A9-IntersectionObserver 与懒加载 × A1-浏览器渲染管线

**场景**: 商品卡片使用懒加载，但占位符（placeholder）未设置固定宽高比。当真实图片加载完成后，图片实际宽高比与占位符不同，导致卡片高度变化，后续所有卡片位置偏移，触发大规模布局偏移（Layout Shift）。

**预期行为**: CLS（Cumulative Layout Shift）< 0.1，图片加载不引起周围元素位移。

**失败模式**: 占位符高度为 200px，但商品图片实际高度为 350px（竖版商品图），加载完成后卡片高度突增 150px，下方所有卡片向下偏移，CLS > 0.5。

**通用高地引用**:
- **G4-状态跃迁管理**: 占位态 → 真实态的尺寸差异是布局偏移的根源，需要在占位态就预留正确空间。
- **G1-输入变异**: 商品图片宽高比不统一（正方形、竖版、横版），占位策略需要适配不同比例。

**验证步骤**:
1. 准备宽高比分别为 1:1、3:4、16:9 的商品图片混排列表。
2. 占位符使用固定 `height: 200px`（不设置 `aspect-ratio`）。
3. 使用 `web-vitals` 库记录 CLS 值。
4. 从页面顶部滚动到底部，记录完整 CLS。
5. 预期：CLS < 0.1。
6. 如果 CLS > 0.25，判定失败。
7. 修复方向：使用 `aspect-ratio` CSS 属性或服务端返回图片尺寸信息，在占位符阶段就设定正确比例。

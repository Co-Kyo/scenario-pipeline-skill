# 图片优化：WebP/AVIF 格式选择与加载策略 —— 边界用例

## B1 格式兼容性（输入变异，P1）

### EC-1.1：AVIF 在旧版 Safari 不显示

- **触发条件**：用户使用 Safari < 16.4（iOS < 16.4 / macOS < Ventura 13.3）访问仅提供 AVIF 的页面
- **预期行为**：回退到 WebP 或 JPEG，图片正常显示
- **实际风险**：无 `<picture>` 元素或无 JS 回退 → 图片完全不显示，出现破碎图标
- **复现路径**：iPhone 8 升级到 iOS 15 后无法再升级 → 访问纯 AVIF 站点 → 白屏/破损图
- **加固方向**：始终使用 `<picture>` 元素提供多格式源；服务端基于 `Accept` 头做格式协商（Content Negotiation）

### EC-1.2：WebP 在 IE 及极旧浏览器不支持

- **触发条件**：企业内网仍存在 IE 11 或更旧浏览器，WebP 解码失败
- **预期行为**：回退到 JPEG/PNG
- **实际风险**：仅提供 WebP 的页面在 IE 上全部图片丢失
- **加固方向**：`<picture>` 最后一个 `<source>` 使用 JPEG/PNG 作为兜底

### EC-1.3：AVIF 动画在部分 WebView 中解码异常

- **触发条件**：App 内嵌 WebView（特别是国产浏览器内核 < Chromium 85）加载 AVIF 动图
- **预期行为**：显示动画或至少显示首帧
- **实际风险**：解码崩溃、黑屏、或显示残影
- **加固方向**：对动画场景优先使用 WebP 动图；AVIF 动画仅作增强层

## B2 图片过大未压缩（资源边界，P0）

### EC-2.1：用户上传未压缩原始图直接展示

- **触发条件**：CMS/后台允许用户上传未经处理的原始图片（如 5MB+ 的 PNG/HEIC），前端直接引用原始 URL
- **预期行为**：自动转码压缩后展示
- **实际风险**：LCP 飙升至 5s+；低端设备内存溢出导致页面崩溃
- **复现路径**：运营人员上传 8MB 产品原图 → 首页轮播直接引用 → 移动端 3G 下白屏 10s
- **加固方向**：上传时触发异步转码管线（原图 → WebP/AVIF 多尺寸）；前端始终引用转码后的 URL

### EC-2.2：CDN 未开启图片优化导致原始尺寸下发

- **触发条件**：CDN 配置遗漏或图片优化规则未生效，请求 `/image.png` 直接返回原始文件
- **预期行为**：CDN 自动返回 WebP/AVIF 并根据设备宽度裁剪
- **实际风险**：带宽浪费 3-10 倍；流量费用飙升
- **加固方向**：CDN 层配置图片优化规则；部署后自动化校验返回 Content-Type 和文件大小

### EC-2.3：图片格式标记错误（JPEG 文件标记为 PNG）

- **触发条件**：文件后缀名与实际编码不一致，浏览器按错误格式解码
- **预期行为**：正常渲染
- **实际风险**：解码失败或渲染扭曲；部分浏览器直接拒绝加载
- **加固方向**：上传时校验文件魔数（magic bytes）与扩展名一致性

### EC-2.4：极高分辨率图片导致设备内存溢出

- **触发条件**：用户提供 10000×10000 以上的原始图片，未做尺寸限制直接加载
- **预期行为**：自动缩放到设备可承受的最大尺寸
- **实际风险**：移动端 GPU 内存不足 → 页面闪退或白屏
- **加固方向**：服务端限制最大输出尺寸（如 4096×4096）；`<img>` 标签使用 `srcset` 按需加载

## B3 srcset/sizes 配置错误（输入变异，P1）

### EC-3.1：移动端加载桌面尺寸图片

- **触发条件**：`srcset` 定义了桌面尺寸但 `sizes` 属性缺失或写死为 `100vw`，在移动端浏览器选择了最大宽度源
- **预期行为**：移动端加载 375-768px 宽度的图片
- **实际风险**：3G 网络下多下载 2-4 倍数据；LCP 延迟 2-5s
- **复现路径**：`<img srcset="hero-1200.jpg 1200w, hero-600.jpg 600w" sizes="100vw">` → 移动端视口 375px → 浏览器选 1200w
- **加固方向**：`sizes` 必须与实际 CSS 布局匹配；使用 Lighthouse CI 检测响应式图片评分

### EC-3.2：高 DPI 屏幕加载 1x 图片导致模糊

- **触发条件**：Retina 屏设备（2x/3x）但 `srcset` 未提供高分辨率源，浏览器只能选 1x 图
- **预期行为**：加载 2x 或 3x 图片
- **实际风险**：图片明显模糊，视觉质量差
- **加固方向**：`srcset` 至少包含 1x 和 2x 两个密度描述符；关键图片（首屏 hero）提供 3x 源

### EC-3.3：`w` 描述符与 `sizes` 不匹配导致选错源

- **触发条件**：`srcset` 使用 `w` 描述符但 `sizes` 中的媒体查询与实际 CSS 布局矛盾
- **预期行为**：浏览器基于 `sizes` 计算出正确的 `currentSrcSize` 并选择最合适的源
- **实际风险**：选中过大或过小的图片；特别是在 CSS Grid/Flex 弹性布局下容易出错
- **加固方向**：使用 Chrome DevTools 的 Network 面板验证实际加载的 `currentSrc`；CI 中集成响应式图片回归测试

### EC-3.4：`picture` 元素中 `<source>` 媒体查询与 `srcset` 冲突

- **触发条件**：`<picture>` 内同时使用媒体查询条件和 `srcset`/`sizes`，逻辑交叉导致不可预测的源选择
- **预期行为**：每个断点清晰地指向一个确定的图片源
- **实际风险**：不同浏览器选择不同源；调试困难
- **加固方向**：`<picture>` 中的 `<source>` 用媒体查询做格式/艺术指导切换；尺寸自适应交给 `srcset`+`sizes`，避免混用

## 通用高地检查清单

| 高地维度 | 检查项 |
|---------|--------|
| 格式兜底 | 所有 `<picture>` 均有 JPEG/PNG 兜底 `<source>` |
| 尺寸限制 | 服务端限制最大输出尺寸 ≤ 4096px |
| 压缩验证 | CI 流水线检测上传图片是否已压缩（>200KB 标记警告） |
| 响应式覆盖 | srcset 至少包含 3 个断点（mobile/tablet/desktop） |
| DPI 覆盖 | 关键图片提供 ≥2x 密度源 |
| 性能预算 | LCP 图片 ≤ 200KB（WebP）/ ≤ 150KB（AVIF） |
| 兼容性测试 | Safari 15/16、Chrome、Firefox、Samsung Internet 覆盖测试 |
| 回归监控 | Lighthouse CI 响应式图片评分 ≥ 90 |

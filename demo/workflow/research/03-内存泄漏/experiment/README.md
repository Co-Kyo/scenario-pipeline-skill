# 实验：内存泄漏 — 长时间运行页面的性能退化排查

## 实验目标

通过 4 个动手实验，直观理解浏览器内存泄漏的常见模式，掌握使用 DevTools 排查内存问题的基本方法。

## 实验清单

| # | 实验名称 | 核心知识点 | 对应素材 |
|---|---------|-----------|---------|
| 1 | Map vs WeakMap 内存对比 | 强引用阻止 GC，弱引用自动释放 | A16 WeakMap |
| 2 | 事件监听器泄漏演示 | 组件切换时未清理监听器导致内存增长 | A14 事件监听泄漏 |
| 3 | setInterval 闭包引用泄漏 | 定时器持有已删除 DOM 的引用 | A15 闭包引用 |
| 4 | Detached DOM 检测 | 移除节点后仍被 JS 引用 | A13 堆快照 / Detached DOM |

## 使用方法

1. 在浏览器中打开 `index.html`（推荐 Chrome）
2. 页面顶部显示实时内存使用量（需 `performance.memory` 支持，Chrome 专属）
3. 每个实验区域独立操作，按步骤点击按钮并观察结果
4. 打开 DevTools → Memory 面板配合观察堆快照变化

## 验证检查点

### 检查点 1：Map vs WeakMap
- **操作**：分别用 Map 和 WeakMap 存储 10000 个对象，然后删除引用
- **预期**：Map 版本内存不降（对象仍被 Map 持有），WeakMap 版本内存可被 GC 回收
- **验证方法**：点击「手动 GC」后观察内存数值变化

### 检查点 2：事件监听器泄漏
- **操作**：反复切换「组件」10 次以上
- **预期**：每次切换后事件监听器数量持续增长
- **验证方法**：查看页面显示的监听器计数；DevTools → Event Listeners 面板确认

### 检查点 3：setInterval 闭包泄漏
- **操作**：启动定时器 → 销毁组件 → 查看定时器状态
- **预期**：组件销毁后定时器仍在运行，闭包中持有的 DOM 引用无法释放
- **验证方法**：控制台持续输出日志；堆快照中可见 Detached DOM 节点

### 检查点 4：Detached DOM 检测
- **操作**：创建 DOM 节点 → 移除 → 保持 JS 引用
- **预期**：节点从 DOM 树移除但未被 GC（Detached 状态）
- **验证方法**：堆快照按 "Detached" 过滤可找到该节点

## 技术说明

- 纯 HTML/CSS/JS，无需构建工具，浏览器直接打开
- 内存数据通过 `performance.memory` API 获取（仅 Chrome/Edge 支持）
- 其他浏览器会显示「不支持」提示，但实验逻辑仍可观察
- 建议使用 Chrome 无痕模式减少扩展干扰

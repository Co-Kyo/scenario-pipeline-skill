## 阶段一：理解 GC 机制

### 你将理解什么
内存泄漏的本质是"不再需要的对象仍然被引用，GC 无法回收"。要理解泄漏，先理解 GC 怎么判断"不再需要"。

### Step 1：分代回收基础
**做**：读 `capabilities/A12-V8垃圾回收机制.md`。
**你会看到什么**：V8 将堆分为新生代（Scavenge）和老生代（Mark-Sweep-Compact）。新生代存活过两次 GC 的对象晋升老生代。
**这说明了什么**：老生代回收成本高（全堆扫描），所以泄漏的对象一旦晋升到老生代，影响会越来越大。
**接下来去哪**：读 `overview.md` 第一节"内存生命周期"。
**做到才算过**：能解释"新生代 Scavenge"和"老生代 Mark-Sweep"的区别。

- [ ] 能解释 Mark-Sweep-Compact 的三个阶段

## 阶段二：常见泄漏模式

### 你将理解什么
90% 的前端内存泄漏来自三种模式：未清理的事件监听、未清理的定时器、意外的闭包引用。

### Step 2：事件监听泄漏
**做**：读 `capabilities/A14-事件监听清理.md`。
**你会看到什么**：`addEventListener` 后不 `removeEventListener`，回调持有对 DOM 元素的引用，元素 remove 后仍无法回收。
**这说明了什么**：SPA 路由切换时，旧组件的事件监听必须在 `unmount`/`onUnmounted` 中清理。
**接下来去哪**：读 `edge-cases.md` 关于"全局事件监听"的部分。
**做到才算过**：能在一个 React 组件中正确清理 window 上的 scroll 监听。

### Step 3：定时器与闭包
**做**：读 `capabilities/A15-定时器与闭包管理.md`。
**你会看到什么**：`setInterval` 不清理会一直执行；闭包引用了大对象，即使函数不再使用，大对象也无法回收。
**这说明了什么**：`clearInterval`/`clearTimeout` 是必须的，不是"可选的清理"。
**接下来去哪**：读 `capabilities/A16-WeakMap-WeakRef弱引用.md`。
**做到才算过**：能解释"为什么闭包会导致内存泄漏"。

### Step 4：弱引用的正确使用
**做**：读 `capabilities/A16-WeakMap-WeakRef弱引用.md`。
**你会看到什么**：WeakMap 的 key 是弱引用，key 对象被 GC 后整个 entry 自动消失。
**这说明了什么**：缓存场景用 WeakMap 代替 Map，可以避免"缓存导致对象无法回收"。
**接下来去哪**：读 `trade-offs.md` 中"WeakMap vs Map"的权衡。
**做到才算过**：能用 WeakMap 实现一个不阻止 GC 的缓存。

- [ ] 能用 WeakMap/WeakRef 避免缓存泄漏

## 阶段三：诊断工具

### 你将理解什么
发现了"页面越来越卡"，怎么定位是内存泄漏？用 DevTools Memory 面板拍堆快照对比。

### Step 5：堆快照对比
**做**：读 `capabilities/A13-堆快照分析.md`。
**你会看到什么**：拍两次堆快照（操作前/后），对比"新增对象"中哪些没被回收。
**这说明了什么**：堆快照能看到每个对象的引用链（Retainers），顺着引用链找到"谁阻止了 GC"。
**接下来去哪**：读 `capabilities/A17-DevTools Performance-Memory面板.md`。
**做到才算过**：能用 DevTools 拍堆快照并找到一个泄漏对象的引用链。

### Step 6：Performance 面板看内存曲线
**做**：读 `capabilities/A17-DevTools Performance-Memory面板.md`。
**你会看到什么**：Performance 面板录制时，Memory 曲线持续上升 = 泄漏；锯齿形 = 正常 GC。
**这说明了什么**：内存曲线是最快的"是否泄漏"判断依据，堆快照是定位"哪里泄漏"的工具。
**接下来去哪**：读 `experiment/index.html` 中的内存泄漏模拟实验。
**做到才算过**：能通过 Memory 曲线判断页面是否存在泄漏。

- [ ] 能通过 Memory 曲线判断泄漏

## 阶段四：综合排查

### 你将理解什么
真实场景中，泄漏往往不是单一原因，而是多种模式的组合。

### Step 7：完整排查流程
**做**：读 `edge-cases.md` 全文。
**你会看到什么**：隐蔽变体——事件监听在全局对象上、闭包引用了整个组件实例、WeakMap 使用不当等。
**这说明了什么**：排查不能只看"有没有 removeEventListener"，要顺着引用链找到根因。
**接下来去哪**：读 `experiment/index.html` 的防御策略速查表。
**做到才算过**：能针对一个 SPA 应用写出完整的内存泄漏排查 checklist。

📦 模块系统对比 A28 ESM vs CJS

ESM 的静态结构是 Tree Shaking 的前提。CJS 的动态 require 无法在构建时确定依赖。

🟢 ESM 模块 — math.mjs
// ESM：静态声明，构建时可分析 export function add(a, b) { return a + b; } export function multiply(a, b) { return a * b; } export function divide(a, b) { return a / b; } // 导入侧 — 静态绑定 import { add } from './math.mjs'; console.log(add(1, 2));
🟠 CJS 模块 — math.cjs
// CJS：动态执行，构建时无法确定 const math = {}; math.add = function(a, b) { return a + b; }; math.multiply = function(a, b) { return a * b; }; math.divide = function(a, b) { return a / b; }; module.exports = math; // 导入侧 — 动态执行 const m = require('./math.cjs'); console.log(m.add(1, 2));

⚖️ Hash 策略对比

❌ [hash] 全局

任意文件改动 → 所有产物 hash 变化 → 缓存全部失效

⚠️ [chunkhash]

chunk 内容变化 → 该 chunk hash 变化，但可能影响依赖链

✅ [contenthash]

仅内容变化的 chunk hash 改变 → 最大化缓存命中

✅ 验证检查点

完成实验后，确认以下检查点全部通过。

⏳

CP-1：ESM 静态分析

ESM 的 export 语句在解析阶段即可确定导出列表，无需执行代码。

⏳

CP-2：Code Splitting 按需加载

点击非首屏功能时，只有对应 chunk 被请求加载，首屏体积保持不变。

⏳

CP-3：Tree Shaking 移除未引用导出

取消勾选某些导出后执行 Tree Shaking，未使用的导出被标记为已删除。

⏳

CP-4：contenthash 变化范围正确

仅修改某个模块时，只有该模块及其依赖链的 chunk hash 发生变化。

⏳

CP-5：产物分析面板数据自洽

面板中显示的体积、chunk 数量、缓存命中率数据相互一致。

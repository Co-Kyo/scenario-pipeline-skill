# 方案对比象限：内存泄漏排查中的关键权衡

## 开篇：选择的困惑

SPA 应用长时间运行后逐渐卡顿，几乎所有前端开发者都遇到过。排查方向很明确——事件监听、定时器、Detached DOM——但真正在代码层面做决策时，一个根本性的困惑会反复出现：**到底该用哪种内存管理策略？**

用 `Map` 存缓存还是 `WeakMap`？性能分析交给 Performance 面板还是 Lighthouse？资源清理逐个手写 `removeEventListener` 还是统一封装 `AbortController`？这些问题没有绝对正确答案，只有场景下的最优权衡。以下三个象限帮你理清思路。

---

## 象限一：缓存引用策略——Map vs WeakMap

### 核心矛盾

- **Map**：强引用，存入的对象永远不会被 GC 回收，数据"稳如磐石"——但这也正是泄漏的温床。SPA 中组件销毁后，如果 Map 仍持有对 DOM 节点或组件实例的引用，这些对象就成了 Detached DOM，内存只增不减。
- **WeakMap**：弱引用，键对象没有其他引用时，GC 自动回收对应的条目。缓存数据"随用随丢"——但你无法遍历、无法获知大小，调试时几乎是个黑盒。

### 权衡原理

| 维度 | Map（强引用） | WeakMap（弱引用） |
|------|-------------|------------------|
| 泄漏风险 | 高：必须手动 delete，遗漏即泄漏 | 低：GC 自动兜底 |
| 调试可见性 | 高：可遍历、可检查 size | 低：无法枚举，内存快照中才能看到 |
| 适用场景 | 需要长期持有的全局配置、路由表 | 绑定 DOM 元素/组件实例的临时缓存 |
| 性能开销 | 无额外开销 | GC 扫描弱引用有微量成本 |

### 决策指南

- 你的缓存键是 **DOM 节点或组件实例**？→ `WeakMap`，让 GC 跟随元素生命周期自动清理。
- 你的缓存需要 **跨模块共享、可查询、可统计**？→ `Map`，但必须配套显式清理逻辑（组件卸载时 `map.delete(key)`）。
- 拿不准时，先用 `WeakMap`——宁可数据被提前回收，也不要让它永远驻留。

## 象限二：性能分析工具——Performance 面板 vs Lighthouse

### 核心矛盾

- **Performance 面板**：手动录制、手动分析，能看到每一次 GC 事件、每一个内存分配的调用栈。精度极高，但对使用者要求也高——你需要知道在录制期间模拟用户操作，需要能看懂火焰图，需要耐心逐帧排查。
- **Lighthouse**：一键审计，自动给出 Performance 评分和优化建议。门槛极低，但它的内存分析粒度粗，无法告诉你"哪个函数在泄漏"，只能告诉你"页面性能不好"。

### 权衡原理

| 维度 | Performance 面板 | Lighthouse |
|------|-----------------|------------|
| 分析精度 | 极高：逐帧、逐 GC、逐分配 | 粗粒度：综合评分 + 通用建议 |
| 操作门槛 | 高：需要手动录制和解读 | 低：一键运行，结论直给 |
| 泄漏定位能力 | 强：可追踪 JS Heap 增长曲线和 Retainers | 弱：只能反映结果，不能定位原因 |
| 适用阶段 | 深度排查、确认泄漏点 | 快速摸底、CI 集成监控 |

### 决策指南

- 你怀疑有泄漏但 **不知道在哪**？→ 先跑 Lighthouse 快速摸底，确认是否有内存相关警告。
- 你已经观察到卡顿/内存增长，需要 **精确定位**？→ Performance 面板录制 + Memory 面板的 Heap Snapshot / Allocation Timeline。
- CI 流水线中想 **自动拦截回归**？→ Lighthouse 设定 Performance 分数阈值，低于则阻断。
- 两者不是互斥的：Lighthouse 做体检，Performance 做手术。

## 象限三：资源清理方式——手动逐个清理 vs AbortController 统一取消

### 核心矛盾

- **手动清理**：在 `useEffect` 的 return 里逐个调用 `removeEventListener`、`clearInterval`、`cancelAnimationFrame`。逻辑清晰，每一条清理都对得上一条注册——但代码冗长，遗漏一条就多一个泄漏点，且新增监听时容易忘记同步更新清理逻辑。
- **AbortController**：创建一个 controller，把 `signal` 传给所有可中止的操作（fetch、addEventListener），组件卸载时 `controller.abort()` 一刀切。简洁优雅——但需要封装，且不是所有 API 都原生支持 signal（定时器、第三方库回调需要额外适配）。

### 权衡原理

| 维度 | 手动逐个清理 | AbortController 统一取消 |
|------|------------|------------------------|
| 代码量 | 多：N 个注册对应 N 条清理 | 少：一个 abort() 全部中止 |
| 遗漏风险 | 高：新增操作容易忘加清理 | 低：signal 传递即可，结构化保障 |
| 灵活性 | 高：可选择性清理某些资源 | 中：abort 是全量中止，无法部分取消 |
| 适配成本 | 无：原生 API 直接调用 | 需要封装：定时器、自定义回调需手动适配 signal |
| 可读性 | 低：清理逻辑分散，增删改需同步 | 高：集中在一处，意图明确 |

### 决策指南

- 组件内监听/定时器 **少于 3 个**？→ 手动清理足够，不必引入额外抽象。
- 组件内资源 **多且杂**（fetch + 多个事件 + 定时器）？→ `AbortController` 统一管理，封装一个 `useAbortController` hook。
- 团队规模大、Code Review 流程松？→ 优先 `AbortController`，用结构化手段降低人为遗漏概率。
- 涉及第三方库不支持 signal？→ 包一层适配器，在 abort 时调用库的 dispose/destroy 方法。

## 收尾：具体推荐

针对 SPA 内存泄漏排查这一命题，推荐的默认策略组合：

1. **缓存引用**：默认用 `WeakMap` 存储与 DOM/组件绑定的数据。仅在确实需要全局可查询的场景（如路由配置、国际化字典）才用 `Map`，并强制要求在模块卸载时显式清理。
2. **分析工具链**：开发阶段用 `Performance 面板 + Memory 面板` 做深度定位；CI 阶段用 `Lighthouse` 做自动化拦截。两者互补，不互斥。
3. **资源清理**：封装一个 `useAbortController` 自定义 hook，组件内所有可中止操作统一走 signal。对不支持 signal 的 API（如 `setInterval`），在 abort 回调中手动清除。

**一句话原则**：能自动回收的交给 GC（WeakMap），能统一取消的交给 AbortController，能自动检测的交给 CI（Lighthouse）。手动操作只在必要时介入，并且用代码审查兜底。

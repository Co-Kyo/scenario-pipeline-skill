# 事件循环机制

> ID: A2 | 扇出: 3/8 | 耦合度: 1 | 战略价值: 3.0 | 累积价值: 5.0 | ⛰️ 二级高地

## 核心机制

JavaScript 是单线程的，事件循环（Event Loop）是其并发模型的核心：

```
调用栈（Call Stack）
    ↓ 执行完毕
微任务队列（Microtask Queue）← Promise.then / MutationObserver / queueMicrotask
    ↓ 清空
渲染机会（Rendering Opportunity）← 每 16.6ms（60fps）
    ↓ 浏览器决定是否渲染
宏任务队列（Macrotask Queue）← setTimeout / setInterval / I/O / MessageChannel
```

**关键规则**：
1. 调用栈清空后，先清空所有微任务，再考虑渲染
2. 渲染前执行 `requestAnimationFrame` 回调
3. 如果浏览器决定不渲染（如 tab 不可见），直接取下一个宏任务
4. `requestIdleCallback` 在帧末尾的空闲时间执行

**setTimeout(cb, 0) 不等于立即执行**：它把回调放入宏任务队列，必须等当前调用栈 + 所有微任务 + 可能的渲染完成后才执行。

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|---------|---------|---------|---------|
| 1 | 微任务风暴 | Promise 链过长或递归产生大量微任务 | 主线程被微任务独占，渲染被阻塞，页面卡死 | Performance 面板 → Long Task 标记 | 拆分微任务为多个宏任务（setTimeout 包裹） |
| 2 | 宏任务饥饿 | 每个宏任务执行时间 > 16ms | 帧率持续低于 60fps，输入延迟 | Performance 面板 → 红色 Long Task | 任务切片（requestIdleCallback / 时间分片） |
| 3 | rAF 时序错误 | 在 rAF 回调中执行重计算逻辑 | rAF 回调超时，错过渲染窗口 | Performance 面板 → rAF 任务过长 | rAF 只做轻量 DOM 更新，重计算放 Web Worker |
| 4 | setTimeout 精度 | 嵌套 setTimeout 超过 5 层 | 浏览器强制最小延迟 4ms，实际间隔不可控 | 代码审查 | 需要精确计时用 requestAnimationFrame + 时间戳 |

## 调试工具

| 工具 | 用法 |
|------|------|
| Chrome DevTools Performance 面板 | 录制后查看 Task 类型（Macro/Micro/Render），识别 Long Task |
| `performance.eventCounts` | 统计各类型事件触发频率 |
| Performance Observer（Long Tasks） | `new PerformanceObserver(cb).observe({entryTypes: ['longtask']})` 监测 >50ms 的任务 |

## 典型权衡

| 维度 | 方案 A | 方案 B | 选择建议 |
|------|--------|--------|---------|
| 任务调度 | 同步批量执行（简单，但阻塞渲染）| 切片执行（rAF/rIC，但复杂度高）| 数据量大（>1000 项）时切片，小量同步即可 |
| 微任务 vs 宏任务 | Promise（微任务，响应快但可能独占）| setTimeout（宏任务，允许渲染插入）| 状态更新用 Promise，需要渲染插入用 MessageChannel |

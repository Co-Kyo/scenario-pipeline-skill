# A31 - 事件循环与宏微任务

## 核心机制

事件循环（Event Loop）是浏览器协调 JavaScript 执行、用户交互、渲染和网络事件的核心调度机制。JavaScript 是单线程语言，事件循环通过**任务队列**机制让多类任务有序执行。

### 1. 事件循环的单次迭代

每次事件循环迭代（one iteration / one turn）按以下顺序执行：

```
┌─────────────────────────────────────────────┐
│  1. 从 macrotask queue 取最旧的一个任务执行    │
│  2. 执行并清空 microtask queue（全部）         │
│  3. 渲染阶段（如果需要）：                     │
│     a. 处理 input events                      │
│     b. 执行 rAF callbacks                     │
│     c. 执行 ResizeObserver / IntersectionObserver │
│     d. Style → Layout → Paint → Composite     │
│  4. 检查是否需要渲染，如需要则执行渲染          │
│  5. 回到步骤 1                                │
└─────────────────────────────────────────────┘
```

### 2. Macrotask（宏任务）

Macrotask 是事件循环每轮迭代的基本调度单元。

**常见的 macrotask 来源**：

| 来源 | API |
|------|-----|
| 定时器 | `setTimeout()`, `setInterval()` |
| 用户交互 | click, scroll, input 等事件回调 |
| 网络 | `fetch()` 的 `.then()` 后续回调（注：首次 .then 是 microtask） |
| I/O | `MessageChannel.onmessage` |
| 渲染 | `requestAnimationFrame` 回调（在渲染阶段执行） |
| 脚本 | `<script>` 标签的执行 |

**关键规则**：
- 每轮事件循环**只执行一个** macrotask
- 新加入的 macrotask 要等下一轮迭代
- 不同 macrotask 来源可能有**优先级差异**（浏览器实现相关）

### 3. Microtask（微任务）

Microtask 在每个 macrotask 执行完毕后、渲染之前**全部清空**。

**常见的 microtask 来源**：

| 来源 | API |
|------|-----|
| Promise | `.then()`, `.catch()`, `.finally()` 的回调 |
| MutationObserver | DOM 变化观察回调 |
| queueMicrotask | 显式入队微任务 |
| `await` | `async/await` 中 await 后的代码 |

**关键规则**：
- microtask queue **必须清空**才进入渲染或下一个 macrotask
- microtask 中可以产生新的 microtask（递归清空）
- **风险**：如果 microtask 不断产生新 microtask，会**阻塞渲染**，导致页面无响应

### 4. 渲染时机

浏览器**不一定每轮迭代都渲染**。渲染触发条件：

- DOM 变更导致需要重绘
- CSS 动画帧
- rAF 回调存在
- 浏览器判断需要更新（如滚动）

**渲染频率通常受显示器刷新率限制**（60Hz = 最多每秒 60 次渲染）。

### 5. Long Task 识别

Long Task 是指主线程被**连续占用超过 50ms** 的任务。它会导致：

- 输入延迟（用户点击后 > 50ms 无响应）
- 动画卡顿（渲染被延迟）
- TTI（Time to Interactive）延长

**识别方法**：
- Performance 面板中 Main 区域的红色长条
- Long Tasks API（PerformanceObserver + `entryType: "longtask"`）
- Lighthouse 的 TBT（Total Blocking Time）指标

## 工程瓶颈

### 瓶颈1：微任务递归导致渲染饥饿

- **触发条件**：在 Promise.then 回调中不断产生新的 Promise
- **症状**：页面冻结，无法响应用户输入，无法渲染
- **检测**：Performance 面板中 Microtask Queue 持续占用主线程
- **缓解**：
  - 使用 `setTimeout` 或 `requestAnimationFrame` 将部分任务移到 macrotask
  - 限制微任务递归深度
  - 使用 `queueMicrotask` 而非 Promise 链来控制执行

### 瓶颈2：setTimeout 最小延迟导致调度不准

- **触发条件**：`setTimeout(fn, 0)` 实际延迟通常为 4ms（HTML 规范规定嵌套超过 5 层后最小 4ms）
- **症状**：定时器回调间隔不可靠
- **检测**：对比 `setTimeout` 实际延迟与预期
- **缓解**：
  - 使用 `MessageChannel.postMessage()` 实现 0ms 延迟的 macrotask
  - 视觉更新使用 `requestAnimationFrame`
  - 精确定时使用 `performance.now()` + 循环检查

### 瓶颈3：事件处理回调阻塞主线程

- **触发条件**：click/scroll/input 回调中执行耗时操作
- **症状**：用户操作后页面无响应；滚动卡顿
- **检测**：Performance 面板中事件回调耗时 > 50ms
- **缓解**：
  - 将耗时逻辑移到 Web Worker
  - 使用时间切片（time slicing）：将任务分解为多个小任务
  - 使用 `requestIdleCallback` 处理非紧急任务

### 瓶颈4：宏任务队列积压

- **触发条件**：大量事件同时触发（如快速滚动产生大量 scroll 事件）
- **症状**：队列中的任务处理延迟越来越大
- **检测**：Performance 面板中任务间隔逐渐增大
- **缓解**：
  - 使用 `requestAnimationFrame` 节流视觉更新
  - 使用 `AbortController` 取消过时的 fetch 请求
  - 合并/去重事件处理

### 瓶颈5：async/await 的隐式微任务

- **触发条件**：`await` 后的代码会创建微任务，大量连续 await 产生大量微任务
- **症状**：看似同步的代码实际被拆成大量微任务，影响渲染时机
- **检测**：Performance 面板中微任务执行时间异常长
- **缓解**：理解 `await` 的微任务语义；在需要渲染的场景使用 `setTimeout` 或 `rAF` 穿插

## 典型权衡

### 权衡1：微任务 vs 宏任务调度

| 维度 | Microtask | Macrotask |
|------|-----------|-----------|
| 执行时机 | 当前任务结束后立即 | 下一轮事件循环 |
| 渲染影响 | 可能阻塞渲染（递归时） | 不阻塞（每轮一个） |
| 优先级 | 高 | 低 |
| 适用场景 | 状态一致性（如 Promise 链） | 延迟执行、定时任务 |
| **建议** | 需要立即一致的用 microtask；允许延迟的用 macrotask |

### 权衡2：requestAnimationFrame vs requestIdleCallback

| 维度 | requestAnimationFrame | requestIdleCallback |
|------|----------------------|---------------------|
| 执行时机 | 渲染前（高优先级） | 渲染后空闲时（低优先级） |
| 帧率影响 | 占用帧预算 | 不影响帧率 |
| 适用场景 | 动画、视觉更新 | 日志上报、预取、分析 |
| **建议** | 视觉相关用 rAF；后台任务用 rIC |

### 权衡3：同步 vs 异步事件处理

| 维度 | 同步处理 | 异步处理（setTimeout/rAF） |
|------|----------|--------------------------|
| 响应延迟 | 无 | 有（至少一帧延迟） |
| 主线程占用 | 立即占用 | 分散到后续帧 |
| 代码复杂度 | 低 | 高（需要状态管理） |
| **建议** | 简单操作同步处理；复杂计算异步切片 |

## 最小验证实验

```html
<!DOCTYPE html>
<html>
<head>
<style>
  body { font-family: monospace; padding: 20px; background: #1a1a2e; color: #eee; }
  #log { height: 500px; overflow-y: auto; background: #0d1117; padding: 10px; border: 1px solid #333; }
  .log-item { margin: 2px 0; padding: 2px 4px; }
  .sync { color: #7ee787; }
  .micro { color: #79c0ff; }
  .macro { color: #d2a8ff; }
  .render { color: #ffa657; }
  .rAF { color: #ff7b72; }
  button { padding: 8px 16px; margin: 4px; cursor: pointer; }
  #box { width: 50px; height: 50px; background: #58a6ff; position: relative; }
</style>
</head>
<body>
<h3>事件循环执行顺序验证</h3>
<button onclick="runTest1()">测试1: 同步+微任务+宏任务</button>
<button onclick="runTest2()">测试2: 微任务递归</button>
<button onclick="runTest3()">测试3: rAF 与渲染时机</button>
<button onclick="clearLog()">清除日志</button>
<div id="log"></div>
<div id="box"></div>
<script>
const logEl = document.getElementById('log');
let counter = 0;

function log(msg, type = '') {
  const div = document.createElement('div');
  div.className = `log-item ${type}`;
  div.textContent = `[${(performance.now() / 1000).toFixed(3)}s] ${msg}`;
  logEl.appendChild(div);
  logEl.scrollTop = logEl.scrollHeight;
}

function clearLog() { logEl.innerHTML = ''; }

// 测试1: 验证 同步 → 微任务 → 宏任务 的执行顺序
function runTest1() {
  clearLog();
  log('=== 测试1: 执行顺序 ===');

  // 同步代码
  log('1. 同步代码 (sync)', 'sync');

  // 微任务 (Promise)
  Promise.resolve().then(() => {
    log('3. Promise.then (microtask)', 'micro');
  });

  // 微任务 (queueMicrotask)
  queueMicrotask(() => {
    log('4. queueMicrotask (microtask)', 'micro');
  });

  // 宏任务 (setTimeout)
  setTimeout(() => {
    log('5. setTimeout (macrotask)', 'macro');

    // 宏任务执行后，新的微任务
    Promise.resolve().then(() => {
      log('6. Promise.then inside setTimeout (microtask)', 'micro');
    });
  }, 0);

  // 同步代码（在微任务之前执行）
  log('2. 同步代码 continued (sync)', 'sync');

  log('观察：同步 → 微任务 → 宏任务');
}

// 测试2: 微任务递归可能阻塞渲染
function runTest2() {
  clearLog();
  log('=== 测试2: 微任务递归 (可能阻塞渲染!) ===');

  const box = document.getElementById('box');
  box.style.left = '0px';
  box.style.background = '#58a8ff';

  // 先用 rAF 移动方块
  requestAnimationFrame(() => {
    box.style.left = '200px';
    box.style.transition = 'left 0.5s';
  });

  // 然后启动微任务递归
  let count = 0;
  function recursiveMicrotask() {
    count++;
    if (count < 1000) {
      log(`微任务 #${count}`, 'micro');
      queueMicrotask(recursiveMicrotask);
    } else {
      log(`微任务递归完成，共 ${count} 次`, 'micro');
      log('注意：方块的动画可能被延迟（渲染被微任务阻塞）', 'render');
    }
  }
  queueMicrotask(recursiveMicrotask);
}

// 测试3: rAF 与渲染时机
function runTest3() {
  clearLog();
  log('=== 测试3: rAF 与渲染 ===');

  const box = document.getElementById('box');
  box.style.transition = 'none';
  box.style.left = '0px';
  box.style.background = '#ff7b72';

  let frameCount = 0;

  function animate(timestamp) {
    frameCount++;
    const x = Math.min(frameCount * 2, 400);
    box.style.left = x + 'px';
    log(`rAF 帧 #${frameCount}: x=${x}`, 'rAF');

    if (x < 400) {
      requestAnimationFrame(animate);
    } else {
      log(`动画完成，共 ${frameCount} 帧`, 'rAF');
    }
  }

  // 在 rAF 之前添加微任务，观察执行时机
  Promise.resolve().then(() => {
    log('微任务在首次 rAF 之前执行', 'micro');
  });

  setTimeout(() => {
    log('宏任务在 rAF 之间穿插', 'macro');
  }, 0);

  requestAnimationFrame(animate);
}
</script>
</body>
</html>
```

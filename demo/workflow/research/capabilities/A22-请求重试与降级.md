# A22 - 请求重试与降级

## 核心机制

网络请求不可避免会失败（超时、5xx、网络中断等），重试与降级策略是保证前端应用韧性的关键手段。

### 指数退避重试（Exponential Backoff）

核心思想：每次重试的等待时间指数增长，避免大量客户端同时重试导致服务端雪崩。

```
等待时间 = min(baseInterval * 2^attempt + random_jitter, maxInterval)
```

典型参数：
- `baseInterval`：1000ms（1 秒）
- `maxAttempts`：3-5 次
- `maxInterval`：30000ms（30 秒）
- `jitter`：随机抖动（0 ~ baseInterval 的 0.5-1 倍），防止惊群效应

```javascript
async function fetchWithRetry(url, options = {}) {
  const { maxRetries = 3, baseDelay = 1000 } = options;
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch(url);
      if (response.ok) return response;
      if (response.status >= 500) throw new Error(`Server Error: ${response.status}`);
      if (response.status === 429) {
        // Rate limited - use Retry-After header
        const retryAfter = response.headers.get('Retry-After');
        const delay = retryAfter ? parseInt(retryAfter) * 1000 : baseDelay * 2 ** attempt;
        await sleep(delay);
        continue;
      }
      throw new Error(`Client Error: ${response.status}`); // 4xx 不重试
    } catch (err) {
      if (attempt === maxRetries) throw err;
      const jitter = Math.random() * baseDelay;
      const delay = baseDelay * 2 ** attempt + jitter;
      await sleep(Math.min(delay, 30000));
    }
  }
}
```

### 重试策略的关键决策

**何时重试**：
- 网络错误（`TypeError: Failed to fetch`）
- 5xx 服务端错误
- 429 Too Many Requests（使用 `Retry-After` 头）
- 超时（需设置 `AbortController`）

**何时不重试**：
- 4xx 客户端错误（400、401、403、404）— 重试无意义
- POST/PUT 等非幂等请求 — 重试可能导致重复操作（需先确认幂等性）
- 请求已被用户取消

### Fallback 缓存（Stale Cache Fallback）

当网络请求失败时，使用本地缓存的旧数据作为降级：

```javascript
async function fetchWithFallback(url, cacheKey) {
  try {
    const response = await fetchWithRetry(url);
    const data = await response.json();
    // 更新本地缓存
    localStorage.setItem(cacheKey, JSON.stringify({ data, timestamp: Date.now() }));
    return { data, fromCache: false };
  } catch (err) {
    // 网络失败，使用缓存
    const cached = localStorage.getItem(cacheKey);
    if (cached) {
      const { data, timestamp } = JSON.parse(cached);
      return { data, fromCache: true, staleFor: Date.now() - timestamp };
    }
    throw err; // 无缓存可用
  }
}
```

### Graceful Degradation（优雅降级）

当核心功能不可用时，提供替代体验而非直接报错：

- **内容降级**：API 数据加载失败 → 展示缓存数据 + "数据可能不是最新" 提示
- **功能降级**：实时消息推送断开 → 降级为轮询
- **视觉降级**：自定义字体加载失败 → 系统字体 fallback
- **交互降级**：图片 CDN 不可用 → 展示占位图或纯文字描述

### 其他重试策略

- **固定间隔重试**：简单但不推荐（容易雪崩）
- **线性退避**：`delay = base * attempt`，增长较慢
- **Fibonacci 退避**：增长速度介于线性和指数之间
- **断路器模式（Circuit Breaker）**：当失败率超过阈值时自动停止请求，一段时间后半开尝试

## 工程瓶颈

1. **重试放大故障**：大量客户端同时重试可能导致已过载的服务端进一步恶化（retry storm）。必须加 jitter 和限制重试次数。
2. **幂等性保证**：非幂等请求（如扣款、下单）重试可能导致重复执行。需要服务端支持幂等 key 或客户端确认请求未到达服务端再重试。
3. **超时设置困难**：不同接口响应时间差异大，统一超时可能导致正常请求被误杀，需要分接口配置。
4. **降级数据的用户感知**：展示过期缓存数据时需要明确提示用户，避免信息误导。
5. **断路器状态管理**：在前端维护断路器状态需要考虑多 Tab 共享、页面刷新重置等问题。

## 调试工具

- **Chrome DevTools → Network**：观察重试请求、状态码、耗时
- **Chrome DevTools → Network → Throttling**：模拟慢网络和断网测试降级
- **Console 日志**：记录重试次数、降级原因
- **Sentry / 网络监控**：统计请求失败率和重试率
- **Performance Observer**：监控 `resource` 条目中的请求失败

## 典型权衡

1. **重试次数 vs 用户等待时间**：更多重试提高成功率但增加延迟。通常 3 次指数退避是平衡点。关键接口可更多，非关键接口可更少。
2. **缓存降级 vs 数据新鲜度**：使用 stale 缓存提升体验但可能展示过期数据。需要根据业务场景决定是否接受。
3. **客户端重试 vs 服务端限流**：客户端重试配合服务端 `Retry-After` 头是最优组合；仅客户端重试没有服务端配合可能加剧问题。

# 资源加载策略

> ID: A7 | 扇出: 3/8 | 耦合度: 1 | 战略价值: 3.0

资源加载策略是前端性能优化的核心原子能力，通过浏览器原生 Resource Hints 机制（preload / prefetch / preconnect / dns-prefetch）和现代加载属性（lazy loading / fetchpriority），实现资源的优先级调度、预获取和延迟加载，直接影响 LCP、FCP、CLS 等 Core Web Vitals。

## 核心机制

### 1. preload — 当前导航关键资源预加载

```html
<link rel="preload" href="critical.css" as="style" />
<link rel="preload" href="main.js" as="script" />
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin />
```

- **作用**：告诉浏览器当前页面**必定需要**该资源，立即以高优先级发起请求
- **必须指定 `as` 属性**：决定请求优先级、CSP 策略、Accept 头，支持值包括 `style`、`script`、`font`、`image`、`fetch`、`document`、`audio`、`video` 等
- **CORS 资源**：font 和 fetch 类型必须设置 `crossorigin` 属性（即使是同源）
- **响应式预加载**：可通过 `media` 属性实现按视口条件预加载
- **MIME 类型提示**：`type` 属性让浏览器判断是否支持，不支持则跳过下载
- **脚本动态创建**：可通过 JS 动态创建 `<link rel="preload">` 实现运行时预加载

### 2. prefetch — 下一页可能需要的资源预获取

```html
<link rel="prefetch" href="next-page-chunk.js" />
```

- **作用**：提示浏览器在空闲时获取**未来导航可能需要**的资源
- **优先级低**：不会与当前页面的关键资源竞争带宽
- **适用场景**：SPA 路由预加载、下一页数据、可能用到的图片
- **与 preload 区别**：preload 用于当前页必需资源（高优先级），prefetch 用于未来可能资源（低优先级）

### 3. preconnect / dns-prefetch — 连接层预处理

```html
<link rel="preconnect" href="https://cdn.example.com" />
<link rel="dns-prefetch" href="https://cdn.example.com" />
```

- **preconnect**：提前完成 DNS 解析 + TCP 握手 + TLS 协商（约节省 100-300ms）
- **dns-prefetch**：仅提前完成 DNS 解析，兼容性更广，开销更小
- **使用建议**：preconnect 限制在 2-6 个域名，过多会浪费资源；dns-prefetch 可用于更多域名

### 4. lazy loading — 延迟加载

```html
<img src="photo.jpg" loading="lazy" alt="..." />
<iframe src="embed.html" loading="lazy"></iframe>
```

- **原生 `loading="lazy"`**：浏览器原生支持，对 `<img>` 和 `<iframe>` 生效
- **`eager`**（默认）：立即加载
- **`auto`**：由浏览器自行决定
- **Intersection Observer**：JS 层面更灵活的懒加载实现，可自定义阈值和触发条件
- **注意**：首屏关键图片**不应**使用 lazy loading，会导致 LCP 延迟

### 5. fetchpriority — 获取优先级提示

```html
<img src="hero.jpg" fetchpriority="high" />
<img src="below-fold.jpg" fetchpriority="low" />
<link rel="preload" href="critical.css" as="style" fetchpriority="high" />
<script src="analytics.js" fetchpriority="low"></script>
```

- **属性值**：`high` | `low` | `auto`（默认）
- **适用元素**：`<img>`、`<link>`、`<script>`、SVG `<image>`
- **作用**：在同类资源内部微调优先级，不影响跨类型优先级
- **Baseline 2024**：2024 年 10 月起跨主流浏览器可用
- **最佳实践**：结合 preload 使用，为 LCP 图片同时设置 preload + fetchpriority="high"

### 6. 资源优先级调度模型

浏览器内部优先级层次（Chromium 为例）：

| 优先级 | 资源类型 |
|--------|---------|
| Highest | 当前页面 `<script>`（parser-blocking） |
| High | CSS、font（preload）、LCP 图片 |
| Medium | `<script defer/async>`、图片 |
| Low | `<link rel="prefetch">`、非首屏图片 |
| Lowest | idle 预获取 |

`fetchpriority` 在同层级内做 ±1 微调。

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|---------|---------|---------|---------|
| 1 | preload 未被使用 | preload 了资源但实际未在页面中引用 | 浏览器 Console 警告 "resource was preloaded but not used"；带宽浪费 | Chrome DevTools Console；Network 面板 | 确保 preload 的 href 与后续引用一致；移除无用 preload |
| 2 | preload 与实际请求不匹配 | font preload 缺少 crossorigin；CORS 模式不匹配 | 预加载的资源被重复请求（双重下载） | Network 面板对比 | 确保 crossorigin 属性与最终请求的 CORS 模式一致 |
| 3 | prefetch 竞争带宽 | 过多 prefetch 与当前页面关键资源争抢带宽 | 首屏加载变慢，LCP 退化 | Lighthouse；Performance 面板 | 限制 prefetch 数量；使用 Speculation Rules API 替代 |
| 4 | 过度 preconnect | preconnect 超过 6 个域名 | DNS/TCP 连接占用带宽和 CPU，反而拖慢主资源 | Net-internals (chrome://net-internals) | 限制 2-6 个关键域名；其余用 dns-prefetch |
| 5 | lazy loading 首屏图片 | LCP 图片设置了 loading="lazy" | LCP 显著延迟；CLS 增加（图片未预留空间） | Lighthouse LCP 审计 | 首屏图片使用 loading="eager" 或不设 loading；设置 width/height |
| 6 | fetchpriority 滥用 | 所有资源都设为 high | 优先级提示无效，等同于 auto | Performance 面板请求优先级列 | 仅对关键资源（LCP、关键 CSS）设 high；非关键资源设 low |
| 7 | 资源提示发现延迟 | preload 写在 CSS @import 内或动态注入太晚 | preload 提示在资源已被发现后才生效，失去意义 | Coverage 面板；Waterfall 时间线 | preload 放在 `<head>` 最前面；使用 HTTP 103 Early Hints |

## 调试工具

| 工具 | 用法 |
|------|------|
| Chrome DevTools Network 面板 | 查看请求优先级列（Priority）、Waterfall 时间线、是否命中 preload 缓存 |
| Chrome DevTools Console | 检测 preload 未使用警告、prefetch 状态 |
| Lighthouse | 自动审计 "Preload key requests"、"Preconnect to required origins"、"Defer offscreen images" |
| WebPageTest | 多地域真实设备测试；查看资源加载瀑布图和优先级 |
| chrome://net-internals | DNS 解析时间、TCP/TLS 连接详情、preconnect 生效状态 |
| Coverage 面板 (DevTools) | 检测未使用的 CSS/JS 以决定是否需要 preload |
| Performance 面板 (DevTools) | 录制加载过程，查看关键渲染路径和资源调度时序 |
| Speculation Rules API | Chrome 新一代预渲染机制，替代部分 prefetch 场景 |

## 典型权衡

| 维度 | 方案A：preload 关键资源 | 方案B：prefetch 下一页资源 | 选择建议 |
|------|----------------------|--------------------------|---------|
| 目标 | 当前页渲染关键路径 | 未来导航体验优化 | 首屏优先用 preload，交互后用 prefetch |
| 优先级 | 高，与 CSS/JS 同级 | 低，空闲时加载 | preload 过多会阻塞首屏；prefetch 过多浪费带宽 |
| 代价 | 占用关键带宽，需精准识别 | 可能预取不需要的资源 | preload 数量控制在 5 以内；prefetch 按用户行为概率排序 |

| 维度 | 方案A：原生 lazy loading | 方案B：Intersection Observer JS 懒加载 | 选择建议 |
|------|------------------------|--------------------------------------|---------|
| 实现成本 | 零 JS，一个属性 | 需要编写和维护 JS 代码 | 优先用原生 lazy；需要自定义阈值时用 IO |
| 兼容性 | Chrome 77+、Firefox 75+、Safari 15.4+ | 所有现代浏览器 | 老浏览器回退方案：不设 loading 属性即 eager |
| 精细控制 | 浏览器决定阈值 | 完全自定义 rootMargin、threshold | 需要精确控制加载时机（如轮播图）用 IO |

| 维度 | 方案A：preconnect | 方案B：dns-prefetch | 选择建议 |
|------|-----------------|-------------------|---------|
| 预处理范围 | DNS + TCP + TLS | 仅 DNS | 关键第三方域名用 preconnect；其余用 dns-prefetch |
| 开销 | 较高（维持连接） | 微量（仅一次 DNS 查询） | 每页 preconnect 不超过 6 个 |
| 兼容性 | IE 不支持 | IE 9+ | 需兼容旧浏览器时优先 dns-prefetch |

## 最小验证实验

```html
<!-- 实验：对比有无 preload 对 LCP 的影响 -->
<!DOCTYPE html>
<html>
<head>
  <!-- 实验组：预加载关键图片 -->
  <link rel="preload" href="hero.webp" as="image" fetchpriority="high" />
  
  <!-- 对照组：不添加上面的 preload 行 -->
  
  <!-- 连接预处理 -->
  <link rel="preconnect" href="https://cdn.example.com" crossorigin />
  
  <!-- 非首屏图片懒加载 -->
  <style>
    .hero { width: 100%; aspect-ratio: 16/9; }
    .below-fold { width: 100%; aspect-ratio: 4/3; }
  </style>
</head>
<body>
  <!-- LCP 候选 -->
  <img src="hero.webp" class="hero" alt="Hero" 
       width="1200" height="675" />
  
  <!-- 以下内容需要滚动才可见 -->
  <div style="height: 2000px"></div>
  
  <!-- 非首屏图片使用懒加载 -->
  <img src="photo-2.jpg" class="below-fold" alt="Below fold" 
       loading="lazy" width="800" height="600" />
  
  <!-- 验证步骤：
    1. 打开 Chrome DevTools → Network 面板
    2. 勾选 "Priority" 列
    3. 刷新页面，观察 hero.webp 的优先级和加载时间
    4. 运行 Lighthouse，对比有无 preload 的 LCP 差异
    5. 检查 Console 是否有 preload 未使用警告
  -->
</body>
</html>
```

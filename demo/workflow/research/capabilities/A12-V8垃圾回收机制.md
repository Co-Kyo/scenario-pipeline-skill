# A12 - V8 垃圾回收机制

## 核心机制

### 分代回收（Generational GC）

V8 将堆内存分为两个代（Generation）：

- **Young Generation（新生代）**：存放新分配的对象，空间较小（通常 1-8 MB）。进一步细分为 **Nursery**（首次分配区域）和 **Intermediate**（经历过一次 GC 仍存活的对象）。
- **Old Generation（老生代）**：存放经过多次 GC 仍然存活的对象，空间较大。

**分代假说（Generational Hypothesis）**：大多数对象生命周期极短，很快变为不可达。V8 的分代策略利用这一特性，对新生代执行频繁、快速的 GC，对老生代执行低频、更彻底的 GC。

### Scavenge（Minor GC / 新生代回收）

新生代采用 **Semi-Space（半空间）** 策略：

1. 新生代被分为两个等大的半空间：**From-Space**（活动区域）和 **To-Space**（空闲区域）。
2. 新对象在 From-Space 中分配。
3. 当 From-Space 满时触发 Scavenge：
   - 从根集合（stack、global）和 **old-to-new 引用列表** 出发，标记存活对象。
   - 将存活对象复制到 To-Space。
   - Nursery 对象若存活则升级为 Intermediate；Intermediate 对象若再次存活则晋升到 Old Generation。
   - From-Space 和 To-Space 互换角色。
4. 使用 **Write Barrier** 维护 old-to-new 引用表，避免每次 Scavenge 都遍历整个老生代。

**性能特征**：当大多数对象短命时，Scavenge 通常 < 1ms。若存活对象过多，耗时显著增加。

### Mark-Sweep-Compact（Major GC / 老生代回收）

老生代使用 Mark-Sweep-Compact 算法，分三个阶段：

1. **Marking（标记）**：从根集合出发，递归遍历所有可达对象并标记。采用**增量标记（Incremental Marking）** 和**并发标记（Concurrent Marking）** 减少主线程暂停。
2. **Sweeping（清除）**：扫描老生代页面，将死对象占用的内存加入 **Free-List**（按大小分类），供后续分配使用。由并发 Sweeper 线程执行。
3. **Compaction（压缩/整理）**：对碎片化严重的页面执行压缩，将存活对象移动到连续内存区域。仅在碎片化达到阈值时执行，开销较大。

### 增量标记与并发优化（Orinoco 项目）

V8 的 Orinoco 项目将传统 stop-the-world GC 改造为：

- **并发标记（Concurrent Marking）**：标记工作在后台 GC 线程执行，主线程继续运行 JavaScript。使用 **Snapshot-at-the-beginning** 写屏障确保正确性。
- **增量标记（Incremental Marking）**：将标记工作拆分为小步（每步 < 5ms），插入主线程空闲间隙执行。
- **并发清除（Concurrent Sweeping）**：清除阶段完全在后台线程完成。
- **Idle-time GC**：利用 Chrome 的任务调度器，在浏览器空闲期间执行 GC 任务（如动画帧间剩余时间），实现"免费"回收。

## 工程瓶颈

1. **新生代 Scavenge 停顿**：若大量对象存活（如缓存未及时释放），Scavenge 耗时显著增加，导致帧丢失。
2. **老生代 Full GC 停顿**：大型应用的老生代标记可能超过 100ms，导致明显卡顿。
3. **内存碎片化**：长期运行后老生代产生碎片，压缩操作开销大且可能触发额外 GC。
4. **Write Barrier 开销**：维护 old-to-new 引用列表和 Snapshot-at-the-beginning 有运行时性能成本。
5. **晋升风暴**：大量短生命周期对象因被意外引用而晋升到老生代，增加 Major GC 频率。
6. **Hidden Class 退化**：频繁动态增删对象属性会导致 Hidden Class 链过长，影响 GC 扫描效率。

## 调试工具

| 工具 | 用途 |
|------|------|
| Chrome DevTools → Performance 面板 | 观察 GC 事件时间线、停顿时长 |
| Chrome DevTools → Memory 面板 | Heap Snapshot、Allocation Timeline |
| `--trace-gc` (V8 flag) | 打印 GC 事件日志（类型、耗时、回收量） |
| `--trace-gc-verbose` | 打印详细 GC 信息（包括每代回收细节） |
| `--expose-gc` + `global.gc()` | 手动触发 GC，用于性能测试 |
| PerformanceObserver (LCP/INP) | 监控 GC 对用户可感知指标的影响 |
| `performance.measureUserAgentSpecificMemory()` | 测量 JavaScript 堆内存使用量 |

## 典型权衡

| 维度 | 权衡 |
|------|------|
| 新生代大小 | 更大 → 减少 Scavenge 频率，但每次 Scavenge 耗时更长 |
| 并发标记 vs 增量标记 | 并发标记吞吐量更高，但写屏障更复杂；增量标记延迟更低 |
| 压缩频率 | 频繁压缩 → 减少碎片但增加 CPU 开销；不压缩 → 内存浪费 |
| 对象晋升策略 | 过早晋升 → 老生代膨胀；过晚晋升 → 频繁 Scavenge |
| Write Barrier 粒度 | 粗粒度 → 记录冗余引用增加内存；细粒度 → 增加运行时开销 |

## 最小验证实验

### 实验 1：观察 GC 事件

```javascript
// 在 Chrome DevTools Console 中运行
// 打开 Performance 面板，录制后观察 GC 事件（蓝色标记）

// 创建大量短生命周期对象触发 Scavenge
for (let i = 0; i < 100000; i++) {
  const obj = { data: new Array(100).fill(i) };
}

// 创建长生命周期对象触发 Major GC
const longLived = [];
for (let i = 0; i < 1000000; i++) {
  longLived.push({ data: new Array(100).fill(i) });
}
// 等待后手动 GC（需 --expose-gc）
// global.gc();
```

### 实验 2：通过 --trace-gc 观察

```bash
node --trace-gc script.js
# 输出示例：
# [12345:0x...] 1234 ms: Scavenge 2.1 (3.0) -> 1.8 (4.0) MB, 1.2 / 0.0 ms
# [12345:0x...] 5678 ms: Mark-sweep 10.5 (12.0) -> 8.2 (14.0) MB, 15.3 / 0.0 ms
```

### 实验 3：Performance 面板观察 Idle GC

```javascript
// 打开 Performance 面板 → 录制 → 触发动画
// GC 事件会显示为蓝色竖线，观察是否在帧间隙执行
requestAnimationFrame(function loop() {
  // 创建临时对象
  const temp = new Float64Array(1000);
  requestAnimationFrame(loop);
});
```

# A1 - 浏览器渲染管线

## 概述

浏览器渲染管线（Rendering Pipeline）是浏览器将 HTML/CSS/JavaScript 转换为屏幕像素的完整流程。核心路径为：**CRP → Style → Layout → Paint → Composite**。理解这条管线是所有前端性能优化的基础。

## 核心机制

### 1. 构建 DOM 树（DOM Construction）

HTML 解析器（HTML Parser）将接收到的字节流转换为 Token → Node → DOM Tree。这是一个**增量式**过程——解析器边接收数据边构建，不需要等待整个 HTML 下载完成。

- 每个 HTML 元素对应一个 DOM 节点
- 节点按标签层级关系组织为树结构
- 遇到 `<script>`（无 async/defer）时解析阻塞

### 2. 构建 CSSOM 树（CSSOM Construction）

CSS 解析器将 CSS 规则转换为 CSSOM（CSS Object Model）。与 DOM 不同，CSSOM 的构建是**非增量式**的——CSS 是**渲染阻塞（render-blocking）**资源。

- CSS 规则可以被后续规则覆盖，因此必须等全部 CSS 解析完才能使用
- CSSOM 节点继承父节点的样式（级联 Cascade）
- 选择器特异性（Specificity）决定最终应用的样式

### 3. 构建渲染树（Render Tree Construction）

DOM + CSSOM → Render Tree。渲染树只包含**可见内容**：

- `display: none` 的元素及其子树**不会**出现在渲染树中
- `visibility: hidden` 的元素**会**出现在渲染树中（占据布局空间）
- `<head>` 等不可见元素不包含在渲染树中
- 每个可见节点关联计算后的样式（computed style）

### 4. 布局 / 回流（Layout / Reflow）

渲染树构建完成后，浏览器计算每个节点的**几何信息**（位置和尺寸）：

- 盒模型计算：content、padding、border、margin
- 视口（viewport）尺寸影响布局——`<meta name="viewport">` 设置布局宽度
- 布局是递归的：从根节点开始，计算每个子节点
- 设备旋转、窗口缩放会触发重新布局

### 5. 绘制（Paint / Repaint）

将渲染树的节点转换为屏幕上的**像素**：

- 绘制是分层的——浏览器会创建多个绘制层（Paint Layers）
- 绘制顺序遵循 CSS 的层叠上下文（Stacking Context）
- 首次加载时绘制整个屏幕；后续只重绘受影响的区域
- 绘制过程：`fillRect` → `drawImage` → `fillText` 等 Canvas-like 操作

### 6. 合成（Composite）

现代浏览器将页面分为多个**合成层（Compositor Layers）**，由 GPU 并行处理：

- `transform`、`opacity`、`will-change` 等属性可以触发独立合成层
- 合成操作在 GPU 上执行，不阻塞主线程
- 合成层越多，GPU 内存消耗越大
- 分层可以让某些动画跳过 Layout 和 Paint 阶段

### 管线流程图

```
HTML Bytes → DOM Tree ──┐
                        ├→ Render Tree → Layout → Paint → Composite → Screen
CSS Bytes  → CSSOM  ───┘
```

## 工程瓶颈

### 瓶颈1：CSS 渲染阻塞

- **触发条件**：外部 CSS 文件未加载完成时，浏览器不会渲染任何内容
- **症状**：白屏时间长、FCP 延迟
- **检测**：Chrome DevTools → Network 面板查看 CSS 阻塞情况；Lighthouse 的 "Eliminate render-blocking resources" 审计
- **缓解**：
  - 内联关键 CSS（Critical CSS）
  - 非关键 CSS 使用 `media` 属性延迟加载：`<link rel="stylesheet" href="print.css" media="print">`
  - 使用 `<link rel="preload">` 提前加载关键 CSS

### 瓶颈2：JavaScript 阻塞解析

- **触发条件**：`<script>` 标签未使用 `async` 或 `defer`，浏览器在执行脚本前暂停 HTML 解析
- **症状**：DOM 构建延迟、首屏渲染推迟
- **检测**：Performance 面板查看 "Recalculate Style" 和 "Parse HTML" 的耗时
- **缓解**：
  - `<script async>`：异步加载，下载完立即执行
  - `<script defer>`：异步加载，DOM 解析完成后按顺序执行
  - 将脚本放在 `</body>` 之前

### 瓶颈3：布局抖动（Layout Thrashing）

- **触发条件**：JavaScript 交替读取和修改布局属性（如先读 `offsetHeight` 再改 `style.width`），强制浏览器同步执行布局计算
- **症状**：帧率骤降、动画卡顿
- **检测**：Performance 面板中密集的 "Recalculate Style" + "Layout" 交替出现；Chrome DevTools 的 "Forced reflow" 警告
- **缓解**：
  - 批量读取 DOM 布局属性，再批量修改
  - 使用 `requestAnimationFrame` 在帧开始时处理 DOM 操作
  - 使用 `ResizeObserver` 替代手动读取尺寸

### 瓶颈4：过多合成层

- **触发条件**：大量使用 `will-change`、`transform`、`opacity` 创建独立合成层
- **症状**：GPU 内存暴涨、层管理开销增大
- **检测**：Chrome DevTools → Layers 面板查看合成层数量和内存
- **缓解**：
  - 仅对需要动画的元素使用 `will-change`
  - 动画结束后移除 `will-change`
  - 控制合成层总数在合理范围

### 瓶颈5：强制同步布局（Forced Synchronous Layout）

- **触发条件**：在修改样式后立即读取布局属性，浏览器被迫中断优化进行同步布局
- **症状**：额外的布局计算开销、帧时间不稳定
- **检测**：Performance 面板中的 "Layout" 事件带有红色三角警告标记
- **缓解**：
  - 先读所有布局值，再写所有样式变更
  - 使用 `requestAnimationFrame` 分离读写操作
  - 使用 CSS containment（`contain: layout`）限制布局影响范围

## 调试工具

| 工具 | 用途 |
|------|------|
| Chrome DevTools → Performance | 录制并分析完整渲染管线各阶段耗时 |
| Chrome DevTools → Layers | 可视化合成层、层内存使用 |
| Chrome DevTools → Rendering | 开启 "Paint flashing"（绿色闪烁表示重绘）、"Layer borders" |
| Lighthouse | 自动审计 CRP 性能，给出优化建议 |
| Performance API (`performance.timing`) | 精确测量各阶段时间戳 |
| WebPageTest | 真实设备多地点的渲染性能测试 |
| `getComputedStyle()` | 检查元素的计算样式 |

## 典型权衡

### 权衡1：CSS 内联 vs 外链

- **内联**：减少网络请求，消除 CSS 阻塞，但无法利用缓存，HTML 体积增大
- **外链**：可缓存、可并行下载，但阻塞渲染
- **实践**：首屏关键 CSS 内联，非关键 CSS 外链 + preload

### 权衡2：合成层多 vs 少

- **多合成层**：动画更流畅（GPU 加速），但内存消耗大，层管理有开销
- **少合成层**：内存友好，但复杂动画可能掉帧
- **实践**：只为做动画的元素创建合成层，动画结束后清理

### 权衡3：DOM 节点数量 vs 功能复杂度

- **节点多**：功能完整，但 Layout/Paint 耗时增加
- **节点少**：渲染快，但可能牺牲功能或需要虚拟滚动等复杂方案
- **实践**：使用虚拟列表（Virtual List）处理大量列表项；使用 `content-visibility: auto` 跳过视口外内容的渲染

## 最小验证实验

### 实验：观察 CRP 各阶段耗时

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>CRP Pipeline Demo</title>
  <style>
    /* 阻塞渲染的 CSS */
    .box {
      width: 100px;
      height: 100px;
      background: #4CAF50;
      margin: 10px;
      display: inline-block;
    }
    .highlight { background: #FF5722; }
  </style>
  <script>
    // 阻塞解析的脚本 — 模拟耗时操作
    const start = performance.now();
    while (performance.now() - start < 100) {} // 阻塞 100ms
    console.log('Script executed at:', performance.now().toFixed(2), 'ms');
  </script>
</head>
<body>
  <h1>CRP Demo</h1>
  <div id="container"></div>

  <script>
    // 生成 DOM 节点
    const container = document.getElementById('container');
    const frag = document.createDocumentFragment();
    for (let i = 0; i < 500; i++) {
      const div = document.createElement('div');
      div.className = 'box';
      div.textContent = i;
      frag.appendChild(div);
    }
    container.appendChild(frag);

    // 测量布局耗时
    const layoutStart = performance.now();
    const height = container.offsetHeight; // 触发同步布局
    const layoutEnd = performance.now();
    console.log(`Layout took: ${(layoutEnd - layoutStart).toFixed(2)}ms`);

    // 批量修改 vs 布局抖动对比
    const boxes = document.querySelectorAll('.box');

    // ❌ 布局抖动：交替读写
    const thrashStart = performance.now();
    boxes.forEach(box => {
      box.style.width = (box.offsetWidth + 1) + 'px'; // 读 → 写
    });
    const thrashEnd = performance.now();
    console.log(`Layout thrashing: ${(thrashEnd - thrashStart).toFixed(2)}ms`);

    // ✅ 批量操作：先读后写
    const batchStart = performance.now();
    const widths = Array.from(boxes).map(b => b.offsetWidth); // 批量读
    boxes.forEach((box, i) => {
      box.style.width = (widths[i] + 1) + 'px'; // 批量写
    });
    const batchEnd = performance.now();
    console.log(`Batch operation: ${(batchEnd - batchStart).toFixed(2)}ms`);
  </script>
</body>
</html>
```

**实验步骤**：
1. 打开 Chrome DevTools → Performance 面板
2. 点击录制，然后刷新页面
3. 观察火焰图中的 Parse HTML、Evaluate Script、Recalculate Style、Layout 各阶段
4. 开启 Rendering → Paint flashing，观察重绘区域
5. 对比 Layout thrashing 和 Batch operation 的耗时差异

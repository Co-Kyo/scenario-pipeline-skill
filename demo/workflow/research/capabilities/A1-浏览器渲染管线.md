# 浏览器渲染管线

> ID: A1 | 扇出: 6/8 | 耦合度: 1 | 战略价值: 6.0 | 累积价值: 15.0 | 🏔️ 一级高地

## 核心机制

关键渲染路径（Critical Rendering Path, CRP）是浏览器将 HTML/CSS 转换为屏幕像素的完整流程：

```
HTML → DOM Tree ─┐
                 ├→ Render Tree → Layout → Paint → Composite
CSS  → CSSOM  ──┘
```

**DOM 构建**：HTML 解析是增量的（incremental）。浏览器逐字节接收 HTML，将其转为 token → node → DOM Tree。节点数量直接影响后续所有阶段耗时。

**CSSOM 构建**：CSS 解析**不是**增量的——CSS 是渲染阻塞的（render-blocking）。浏览器必须接收并解析完全部 CSS 才能构建 Render Tree，因为后续规则可能覆盖前面的（级联特性）。

**Render Tree**：DOM + CSSOM 合并，只包含可见内容。`display: none` 的节点不进入 Render Tree。

**Layout（布局/回流）**：计算每个可见节点的确切位置和尺寸。任何 Render Tree 的修改（添加节点、改变内容、更新盒模型样式）都会触发 Layout。Layout 是性能瓶颈的重灾区——在滚动或动画中触发 Layout 会导致 jank。

**Paint（绘制）**：将 Layout 计算结果转化为屏幕上的像素。浏览器会优化为只重绘受影响的最小区域。

**Composite（合成）**：多个图层（layers）合并为最终画面。`transform` 和 `opacity` 动画只触发 Composite，跳过 Layout 和 Paint，是最高效的动画方式。

**关键性能规则**：
- 修改 `transform`/`opacity` → 只触发 Composite（最高效）
- 修改 `width`/`height`/`margin`/`padding` → 触发 Layout → Paint → Composite（最昂贵）
- 修改 `color`/`background`/`visibility` → 触发 Paint → Composite

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|---------|---------|---------|---------|
| 1 | 强制同步布局（Layout Thrashing） | JS 中交替读写布局属性（如读 offsetHeight 后立即写 style.width） | 单帧 Layout 次数暴增，帧率骤降至 15fps | Performance 面板 → 红色 Layout 任务 | 批量读完再批量写，或用 requestAnimationFrame 统一处理 |
| 2 | 过深 DOM 嵌套 | DOM 树超过 32 层或节点数超过 1500 个 | Layout 计算时间随节点数线性增长 | Elements 面板 → 右下角节点计数 | 扁平化 DOM 结构，虚拟列表减少实际节点 |
| 3 | CSS 选择器复杂度 | 使用深层后代选择器 `.a .b .c .d` | CSSOM 构建和 Style 计算耗时增加 | Performance 面板 → Style Recalculate | 使用 BEM 命名，减少选择器嵌套层级 |
| 4 | 触发重排的动画 | 用 `width`/`height`/`margin` 做动画 | 每帧触发 Layout + Paint，CPU 占用高 | Performance 面板 → 连续 Layout 任务 | 改用 `transform: scale()` 或 `transform: translate()` |
| 5 | 图层爆炸 | 过度使用 `will-change` 创建独立图层 | GPU 内存暴涨，Composite 耗时增加 | Layers 面板 → 图层数量 | 仅对频繁重绘的元素使用 `will-change`，动画结束后移除 |

**版本相关性**：
- 瓶颈 1（强制同步布局）：Chrome LayoutNG（v85+）优化了布局性能，但 Layout Thrashing 仍然是跨版本的通用问题（**弱相关**）
- 瓶颈 2（DOM 深度）：Chrome 持续优化大 DOM 性能，但根本复杂度不变（**弱相关**）

## 调试工具

| 工具 | 用法 |
|------|------|
| Chrome DevTools Performance 面板 | 录制页面操作，查看 Layout/Paint/Composite 任务的时间分布 |
| Chrome DevTools Layers 面板 | 3D 可视化图层结构，检查图层爆炸 |
| Chrome DevTools Rendering 面板 | 开启 Paint Flashing（绿色闪烁=重绘区域）、Layout Shift Regions |
| `performance.getEntriesByType('paint')` | 获取 FP/FCP 时间戳 |
| `performance.getEntriesByType('layout-shift')` | 获取 CLS 布局偏移记录 |

## 典型权衡

| 维度 | 方案 A | 方案 B | 选择建议 |
|------|--------|--------|---------|
| 动画属性 | `transform`/`opacity`（Composite only）| `width`/`height`/`margin`（Layout + Paint）| 优先用 transform/opacity，只在无法避免时触发 Layout |
| 图层策略 | `will-change` 创建独立图层（GPU 加速）| 不提升图层（CPU 整体重绘）| 对频繁重绘的动画元素提升，静态元素不要提升 |
| DOM 粒度 | 虚拟列表（节点少，内存低）| 全量渲染（节点多，但无复用复杂度）| 数据量 > 1000 时虚拟列表，否则全量更简单 |

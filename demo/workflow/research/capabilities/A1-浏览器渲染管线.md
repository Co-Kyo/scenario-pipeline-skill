# 浏览器渲染管线

> ID: A1 | 扇出: 4/8 | 耦合度: 1 | 战略价值: 4.0

## 核心机制

浏览器渲染管线（Critical Rendering Path, CRP）是指浏览器将 HTML、CSS 和 JavaScript 转换为屏幕像素的完整流程。该管线是所有前端性能优化的基础模型。

### 1. 构建对象模型

**DOM 构建（Parsing → DOM Tree）**
浏览器接收 HTML 字节流，通过分词器（Tokenizer）将其转为 Token（startTag/endTag），再构建为 DOM 节点树。DOM 构建是**增量式**的——边解析边构建，遇到外部资源（`<script>`、`<link>`）时会发起请求。**同步 `<script>` 会阻塞 DOM 解析**，因为浏览器需要等待 JS 下载并执行完毕才能继续。

**CSSOM 构建（CSS → CSSOM Tree）**
CSS 解析生成 CSS 对象模型。与 DOM 不同，CSSOM 构建是**非增量式**的——CSS 具有**渲染阻塞**特性，浏览器必须接收并处理完所有 CSS 后才能进行渲染，因为后续规则可能覆盖前面的规则（Cascade 机制）。

### 2. 样式计算（Style Calculation）

主线程将 DOM 和 CSSOM 合并，为每个 DOM 节点计算最终的计算样式（Computed Style）。浏览器遍历 DOM 树，匹配 CSS 选择器规则，生成 Render Tree（渲染树）。Render Tree 只包含**可见内容**——`display: none` 的节点及其子节点不会出现在渲染树中；`head` 等非可视区域同样被排除。

### 3. 布局（Layout / Reflow）

Render Tree 构建完成后，主线程遍历计算每个节点的几何信息：位置（x, y）和尺寸（width, height）。布局结果称为 Layout Tree。布局依赖视口尺寸，`<meta name="viewport" content="width=device-width">` 会影响布局视口宽度。每次设备旋转或窗口 resize 都会触发重新布局。

**关键 API：**
- `element.getBoundingClientRect()` — 触发布局查询
- `element.offsetWidth / offsetHeight / clientWidth` — 强制同步布局
- `ResizeObserver` — 监听尺寸变化

### 4. 分层（Layer Tree / Update Layer Tree）

主线程遍历 Layout Tree 创建 Layer Tree（层树），确定哪些元素需要独立图层。浏览器默认对特定元素（如具有 `transform`、`opacity`、`will-change`、`position: fixed`、`<video>`、`<canvas>` 等）创建独立图层。可通过 CSS 属性 `will-change: transform` 显式提示浏览器分层。

### 5. 绘制（Paint）

主线程遍历 Layout Tree 生成一系列**绘制记录（Paint Records）**，记录绘制顺序和操作指令。绘制将布局信息转化为具体的绘制指令（如"在坐标 (x, y) 绘制一个红色矩形"）。

### 6. 光栅化（Raster）

合成线程（Compositor Thread）将图层切分为图块（Tiles），将每个图块发送给光栅线程（Raster Thread）进行光栅化——将矢量绘制指令转为位图像素，存储在 GPU 内存中。合成器进程可对视口内或附近的图块进行优先级排序，优先光栅化用户可见区域。

### 7. 合成（Composite）

合成线程收集所有已光栅化图块的绘制四边形信息（Draw Quads），构建合成帧（Compositor Frame），通过 IPC 提交给浏览器进程，再由 GPU 进程渲染到屏幕上。

**合成阶段的性能优势：** 合成操作在独立的合成线程上执行，无需主线程参与，因此不阻塞 JS 执行和样式计算。**仅触发 Composite 的 CSS 属性**（如 `transform`、`opacity`）可实现最优动画性能，避免 Layout 和 Paint 开销。

### 帧生命周期总览

```
HTML → DOM Tree
CSS  → CSSOM Tree
       ↓
   Render Tree (Style)
       ↓
     Layout (Reflow)
       ↓
   Layer Tree
       ↓
     Paint
       ↓
   Raster (GPU)
       ↓
   Composite → 屏幕像素
```

每一步都可能因 JS 介入而被重新触发，形成**强制同步布局（Forced Synchronous Layout）**或**布局抖动（Layout Thrashing）**。

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|----------|----------|----------|----------|
| P0-1 | **Layout Thrashing（布局抖动）** | JS 循环中交替读取布局属性（offsetWidth 等）和修改样式，导致反复强制同步布局 | 帧率骤降，长任务 >50ms，Main 火焰图中出现密集 Layout 事件 | Chrome DevTools Performance → Main 中查找密集的 Layout 事件；`getComputedStyle()` 同步调用 | 读写分离（FastDom / 虚拟 DOM）；批量 DOM 操作（DocumentFragment）；使用 `requestAnimationFrame` 调度 |
| P0-2 | **CSS 渲染阻塞** | `<link rel="stylesheet">` 阻塞首屏渲染，尤其是关键 CSS 体积大或网络慢时 | FCP 延迟，白屏时间长 | Lighthouse → Render-blocking resources；Network 面板查看 CSS 阻塞时长 | 内联关键 CSS（Critical CSS）；非关键 CSS 使用 `media` 属性延迟加载；CSS minify |
| P0-3 | **JS 阻塞解析** | `<script>` 标签无 `async`/`defer`，阻塞 DOM 解析 | DOM 构建延迟，TTFB 后长时间无内容渲染 | Performance → Main 中 Parsing 被 Script 执行打断 | 添加 `async`/`defer` 属性；`<script type="module">` 默认 defer；代码拆分与懒加载 |
| P1-1 | **Paint 开销过大** | 复杂 CSS 效果（`box-shadow`、`border-radius`、`filter`、大图 `background`）导致绘制区域大 | 掉帧，Paint 时间长 | Performance → Paint 事件耗时；`Rendering` 面板 → Paint flashing | 减少绘制区域；使用 `transform` 替代 `top/left` 动画；`contain: paint` 限制绘制范围 |
| P1-2 | **图层爆炸（Layer Explosion）** | 过度使用 `will-change` 或浏览器隐式创建过多图层 | 内存暴涨，合成耗时增加，低端设备卡顿 | Chrome DevTools → Layers 面板查看图层数量和内存占用 | 仅对动画元素使用 `will-change`；定期检查 Layers 面板；避免过多 `transform` 触发隐式分层 |
| P1-3 | **强制同步布局（Forced Reflow）** | 读取 `getBoundingClientRect()`、`offsetWidth` 等属性后立即修改样式 | 帧率下降，Main 中出现红色标记的长任务 | Performance → 查找 Layout 事件紧跟在 Script 事件之后 | 使用 `ResizeObserver` 替代轮询读取；使用 `IntersectionObserver` 替代滚动位置计算 |
| P1-4 | **大图解码阻塞** | 高分辨率图片（即使压缩后体积小）在 Raster 阶段解码耗时长 | 主线程阻塞，页面卡死 | Performance → Raster 中 Image decode 耗时异常；Memory 面板 | 使用适当分辨率的图片；`loading="lazy"` 延迟加载；WebP/AVIF 格式；`decoding="async"` |

## 调试工具

| 工具 | 用法 |
|------|------|
| **Chrome DevTools Performance 面板** | 录制页面加载/运行时性能，查看 FPS、CPU、NET、Main 火焰图、Frames、Raster、GPU 时间线。支持模拟 CPU 降速和弱网。关键视图：Summary（Loading/Scripting/Rendering/Painting）、Call Tree（根活动分析）、Bottom-up（Self Time 排序） |
| **Chrome DevTools Rendering 面板** | 开启 Paint Flashing（高亮重绘区域）、Layout Shift Regions（高亮布局偏移）、Layer Borders（显示图层边界） |
| **Chrome DevTools Layers 面板** | 3D 可视化页面图层结构，检查图层数量、内存占用、图层创建原因 |
| **Lighthouse** | 自动化性能审计，输出 LCP、FID、CLS 等 Web Vitals 指标和优化建议 |
| **Performance Observer API** | `new PerformanceObserver(callback).observe({entryTypes: ['largest-contentful-paint','layout-shift','paint']})` 编程式采集性能指标 |
| **`performance.mark()` / `performance.measure()`** | 自定义性能标记，测量特定代码段执行时间 |
| **`console.time()` / `console.timeEnd()`** | 快速测量同步代码块耗时 |
| **Chrome Task Manager** | 查看每个标签页的 CPU、内存、GPU 内存使用量 |

## 典型权衡

| 维度 | 方案A | 方案B | 选择建议 |
|------|-------|-------|----------|
| **动画实现** | `top/left` 改变位置 | `transform: translate()` | 方案 B。`transform` 仅触发 Composite，跳过 Layout 和 Paint，性能最优 |
| **CSS 加载** | 外部 `<link>` 全量加载 | 内联 Critical CSS + 异步加载剩余 | 方案 B。减少渲染阻塞时间，但增加 HTML 体积和维护成本，适合首屏性能敏感场景 |
| **图层管理** | 浏览器自动分层 | `will-change` 手动分层 | 混合策略。仅对高优先级动画元素使用 `will-change`，避免图层爆炸 |
| **JS 加载** | `<script>` 同步加载 | `<script defer>` | 方案 B。defer 不阻塞解析且保持执行顺序；async 适合独立脚本（如分析 SDK） |
| **布局查询** | 同步读取 `offsetWidth` | `ResizeObserver` 异步监听 | 方案 B。避免强制同步布局，但在需要精确值时仍需同步读取 |
| **图片加载** | 全量加载所有图片 | `loading="lazy"` 延迟加载首屏外图片 | 方案 B。但首屏图片不应 lazy，避免 LCP 劣化 |
| **重绘控制** | 频繁修改多个样式属性 | 批量修改 + `requestAnimationFrame` | 方案 B。合并到下一帧处理，减少不必要的中间重绘 |

## 最小验证实验

```html
<!DOCTYPE html>
<html lang="zh">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>渲染管线验证实验</title>
  <style>
    .box {
      width: 100px; height: 100px;
      background: #3498db;
      margin: 20px;
      will-change: transform;
    }
    .bad-animate {
      /* 方案A：触发 Layout + Paint + Composite */
      animation: bad-move 2s infinite alternate;
    }
    .good-animate {
      /* 方案B：仅触发 Composite */
      animation: good-move 2s infinite alternate;
    }
    @keyframes bad-move {
      from { left: 0; }
      to   { left: 300px; }
    }
    @keyframes good-move {
      from { transform: translateX(0); }
      to   { transform: translateX(300px); }
    }
    .info {
      font-family: monospace;
      padding: 10px;
      background: #f0f0f0;
      margin: 20px;
      font-size: 14px;
    }
  </style>
</head>
<body>
  <h2>渲染管线验证实验</h2>
  <div class="info">
    <p>打开 Chrome DevTools → Performance → 录制 → 观察两种动画的性能差异</p>
    <p>方案A（bad-animate）：使用 left 属性 → 触发 Layout + Paint + Composite</p>
    <p>方案B（good-animate）：使用 transform → 仅触发 Composite</p>
  </div>
  <div style="position:relative;">
    <div class="box bad-animate" style="position:relative;">Layout</div>
    <div class="box good-animate" style="position:relative;">Composite</div>
  </div>

  <script>
    // 实验：强制同步布局 vs 读写分离
    const boxes = document.querySelectorAll('.box');

    // ❌ 布局抖动示例：读写交替
    function badPattern() {
      for (let i = 0; i < 100; i++) {
        const w = boxes[0].offsetWidth; // 读 → 强制布局
        boxes[0].style.width = (w + 1) + 'px'; // 写 → 脏标记
      }
    }

    // ✅ 读写分离示例
    function goodPattern() {
      const widths = [];
      for (let i = 0; i < 100; i++) {
        widths.push(boxes[0].offsetWidth); // 批量读
      }
      for (let i = 0; i < 100; i++) {
        boxes[0].style.width = (widths[i] + 1) + 'px'; // 批量写
      }
    }

    // 使用 Performance API 测量
    performance.mark('bad-start');
    badPattern();
    performance.mark('bad-end');
    performance.measure('布局抖动', 'bad-start', 'bad-end');

    performance.mark('good-start');
    goodPattern();
    performance.mark('good-end');
    performance.measure('读写分离', 'good-start', 'good-end');

    // 输出结果
    const measures = performance.getEntriesByType('measure');
    measures.forEach(m => {
      console.log(`${m.name}: ${m.duration.toFixed(2)}ms`);
    });

    // Performance Observer 监控 LCP
    if ('PerformanceObserver' in window) {
      const lcpObserver = new PerformanceObserver((list) => {
        const entries = list.getEntries();
        const lastEntry = entries[entries.length - 1];
        console.log('LCP:', lastEntry.startTime.toFixed(0), 'ms', lastEntry.element);
      });
      lcpObserver.observe({ type: 'largest-contentful-paint', buffered: true });

      const layoutShiftObserver = new PerformanceObserver((list) => {
        let cls = 0;
        list.getEntries().forEach(entry => {
          if (!entry.hadRecentInput) cls += entry.value;
        });
        console.log('CLS:', cls.toFixed(4));
      });
      layoutShiftObserver.observe({ type: 'layout-shift', buffered: true });
    }
  </script>
</body>
</html>
```

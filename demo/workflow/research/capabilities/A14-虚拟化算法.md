# 虚拟化算法

> ID: A14 | 扇出: 1/8 | 耦合度: 1 | 战略价值: 1.0

虚拟化算法（Virtualization / Windowing）是一类通过仅渲染可视区域内可见元素来优化大规模列表/表格渲染性能的技术。核心思想：**DOM 节点数与数据总量解耦**，使浏览器只需维护少量活跃 DOM 节点，从而将内存占用和重排/重绘开销控制在常数级别。

---

## 核心机制

### 1. 可视区域计算（Viewport Calculation）

- 根据容器的 `clientHeight`（或 `offsetHeight`）和 `scrollTop` 计算当前可视窗口的起止偏移量
- 公式：`startIndex = binarySearch(scrollTop, itemOffsets)`，`endIndex = binarySearch(scrollTop + viewportHeight, itemOffsets)`
- 对于等高列表可直接除法得出索引；对于变高列表需要前缀和 + 二分查找

### 2. 动态高度预估（Dynamic Height Estimation）

- 当列表项高度不确定时，采用"预估高度 + 渲染后修正"策略
- **预估阶段**：使用 `preHeight`（预设平均高度）计算初始 startIndex / endIndex
- **修正阶段**：DOM 渲染完成后，通过 `element.clientHeight` 获取真实高度并缓存到 `allHeightList[]`
- **总高度公式**：`ListHeight = knownHeight + (totalCount - knownCount) × preHeight`
- 已知高度用前缀和数组加速查询：`top(i) = Σ(allHeightList[0..i-1])`

### 3. 滚动锚定（Scroll Anchoring）

- 快速滚动时防止内容"跳动"——当已知高度不足以覆盖 scrollTop 时，回退到预估高度平滑过渡
- 某些实现使用 `requestAnimationFrame` 节流滚动回调，确保每帧最多触发一次重算
- React 场景下可用 `scrollToIndex` + `alignment`（start/center/end/auto）精确控制锚定位置

### 4. DOM 复用池（DOM Recycling / Object Pool）

- 不销毁离开可视区域的 DOM 节点，而是放入回收池
- 进入可视区域的元素从池中取用并更新数据，避免反复 `createElement` / `removeChild`
- vue-virtual-scroller 的 `RecycleScroller` 即采用此策略：组件实例被复用，仅更新 props
- React 系（react-window / react-virtuoso）通常依赖 diff 算法天然复用，但高端实现也维护 DOM 池

### 5. IntersectionObserver

- 浏览器原生 API，异步观察目标元素与根元素（root）的交叉状态
- 核心参数：
  - `root`：观察容器（默认 viewport）
  - `rootMargin`：扩展/收缩检测区域（正值扩大，可实现"预加载"）
  - `threshold`：交叉比例阈值数组 `[0, 0.25, 0.5, 0.75, 1.0]`
  - `trackVisibility` + `delay`：追踪实际可见性（考虑遮挡、opacity、transform），但计算开销大
- 回调在主线程执行，应配合 `requestIdleCallback` 处理耗时逻辑
- 适用于懒加载、无限滚动、广告可见性追踪等场景
- 局限：不提供精确像素级重叠信息，只提供比例区间

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|---------|---------|---------|---------|
| 1 | 变高列表滚动跳动 | 数据项高度差异大 + 快速滚动 | 滚动条位置突变、内容闪烁 | Performance 面板观察 Layout Thrashing | 预渲染缓冲区（overscan）+ 高度缓存 + 平滑滚动锚定 |
| 2 | 首屏白屏 / 布局偏移 | 预估高度与实际偏差大 | 初始渲染后内容整体偏移（CLS 升高） | Lighthouse CLS 指标、DevTools Layout Shift Regions | 预估高度取中位数、SSR 预渲染首屏、骨架屏占位 |
| 3 | 滚动回调性能瓶颈 | 高频 scroll 事件 + 复杂计算 | 60fps 掉帧、滚动卡顿 | Chrome Performance 录制 → Long Task 检测 | `requestAnimationFrame` 节流、IntersectionObserver 替代 scroll 事件、计算移到 Web Worker |
| 4 | 内存泄漏（未回收） | DOM 池未清理 / Observer 未 disconnect | 内存持续增长 | DevTools Memory → Heap Snapshot 对比 | 组件卸载时 disconnect Observer、限制池大小上限 |
| 5 | 搜索 / 跳转定位失败 | `scrollToIndex` 时高度缓存不完整 | 跳转到错误位置 | 手动测试 + 打印计算的 offset | 二分查找前缀和数组、跳转前强制刷新高度缓存 |
| 6 | 动态插入/删除项 | 数据源变更导致索引偏移 | 渲染错乱、重复项 | 对比数据源与渲染 DOM 数量 | 使用唯一 key 绑定、变更时重算 offset 映射 |
| 7 | IntersectionObserver 回调延迟 | `trackVisibility: true` + 复杂遮挡计算 | 可见性判断滞后 | 对比 scroll 事件与 IO 回调时间差 | 非必要不开启 trackVisibility、增大 delay 阈值 |

## 典型权衡

| 维度 | 方案A | 方案B | 选择建议 |
|------|-------|-------|---------|
| 等高 vs 变高 | 等高列表：简单除法定位，性能最优 | 变高列表：前缀和 + 二分查找，复杂度 O(log n) | 能固定高度就固定；无法固定时用预估 + 修正方案 |
| 固定容器 vs 滚动容器 | 固定高度容器：`overflow: auto`，计算简单 | `window` 滚动：需监听 window.scroll，根元素为 null | 内嵌场景用容器；全页列表用 window |
| DOM 回收 vs 纯销毁 | DOM 池复用：减少 GC 压力，适合频繁滚动 | 纯销毁重建：实现简单，适合低频场景 | 高频滚动（聊天列表）用回收池；低频展示用销毁 |
| IntersectionObserver vs scroll 事件 | IO：异步、浏览器优化、不阻塞主线程 | scroll：实时性高、可获取精确 scrollTop | 预加载/懒加载用 IO；精确定位用 scroll |
| Overscan（缓冲区）大小 | 小 overscan（1-3 项）：DOM 节点少、内存低 | 大 overscan（5-10 项）：滚动更平滑、减少白屏 | 快速滚动场景加大 overscan；内存敏感场景减小 |
| 同步渲染 vs 异步渲染 | 同步：scroll 回调内立即更新 DOM | 异步：`requestIdleCallback` / `requestAnimationFrame` 延迟更新 | 优先 rAF 节流；非关键更新可 idle |

## 最小验证实验

```html
<!DOCTYPE html>
<html>
<head>
<style>
  #container { height: 400px; overflow-y: auto; position: relative; }
  .item { position: absolute; width: 100%; box-sizing: border-box; padding: 8px; border-bottom: 1px solid #eee; }
</style>
</head>
<body>
<div id="container"></div>
<script>
const TOTAL = 100000;
const ITEM_HEIGHT = 40;
const OVERSCAN = 5;
const container = document.getElementById('container');
const totalHeight = TOTAL * ITEM_HEIGHT;

// 创建占位元素
const spacer = document.createElement('div');
spacer.style.height = totalHeight + 'px';
container.appendChild(spacer);

// 池化 DOM 节点
const pool = [];
function getItem() {
  return pool.pop() || (() => {
    const el = document.createElement('div');
    el.className = 'item';
    return el;
  })();
}
function recycleItem(el) {
  pool.push(el);
}

let activeItems = [];

function render() {
  const scrollTop = container.scrollTop;
  const viewHeight = container.clientHeight;
  const startIdx = Math.max(0, Math.floor(scrollTop / ITEM_HEIGHT) - OVERSCAN);
  const endIdx = Math.min(TOTAL, Math.ceil((scrollTop + viewHeight) / ITEM_HEIGHT) + OVERSCAN);

  // 回收不在范围内的节点
  activeItems.forEach(el => {
    const idx = parseInt(el.dataset.idx);
    if (idx < startIdx || idx >= endIdx) {
      el.remove();
      recycleItem(el);
    }
  });
  activeItems = activeItems.filter(el => {
    const idx = parseInt(el.dataset.idx);
    return idx >= startIdx && idx < endIdx;
  });

  // 渲染新进入范围的节点
  const existing = new Set(activeItems.map(el => parseInt(el.dataset.idx)));
  for (let i = startIdx; i < endIdx; i++) {
    if (!existing.has(i)) {
      const el = getItem();
      el.dataset.idx = i;
      el.style.top = (i * ITEM_HEIGHT) + 'px';
      el.style.height = ITEM_HEIGHT + 'px';
      el.textContent = 'Item #' + i;
      spacer.appendChild(el);
      activeItems.push(el);
    }
  }
}

container.addEventListener('scroll', () => requestAnimationFrame(render));
render();
</script>
</body>
</html>
```

**验证要点**：
1. 打开 Chrome DevTools → Elements，确认 DOM 中 `.item` 节点始终只有 ~15 个（视口 + overscan），而非 10 万
2. Performance 面板录制快速滚动，确认无 Long Task、帧率稳定 60fps
3. Memory 面板确认节点池大小有上限、无持续增长

# A15 - 定时器与闭包管理

## 核心机制

### setTimeout / setInterval 基本机制

- **setTimeout(callback, delay)**：在 delay 毫秒后执行一次 callback，返回 timer ID
- **setInterval(callback, interval)**：每 interval 毫秒重复执行 callback，返回 timer ID
- **clearTimeout(id)** / **clearInterval(id)**：取消对应的定时器

浏览器内部维护一个 **Task Queue**，定时器到期后将 callback 作为宏任务（Macrotask）推入队列。

### 定时器与内存的关系

定时器回调函数及其闭包上下文会被浏览器保留，直到：
1. 定时器被 `clearTimeout` / `clearInterval` 显式取消
2. 定时器执行完毕（仅 setTimeout）
3. 页面卸载

**关键问题**：只要定时器未被清除，回调函数闭包中引用的所有对象都不会被 GC 回收。

### 闭包引用链

```javascript
function setup() {
  const hugeData = new Array(1000000).fill('x'); // 大对象

  const timerId = setInterval(() => {
    // 这个回调闭包引用了 setup 的作用域
    // 因此 hugeData 无法被 GC 回收
    console.log(hugeData.length);
  }, 1000);

  // 忘记 clearInterval(timerId) → 泄漏！
}
```

### 闭包引用链断开策略

1. **显式清除定时器**：在组件卸载/页面离开时 `clearTimeout` / `clearInterval`
2. **置空引用**：在不再需要时将大对象设为 `null`
3. **最小闭包原则**：回调函数只引用必要的数据
4. **WeakRef 模式**：用 WeakRef 包装大对象，回调中通过 `deref()` 访问

### requestAnimationFrame 与 requestIdleCallback

- **requestAnimationFrame(callback)**：下一帧渲染前执行，需在回调中重新注册以持续执行。取消用 `cancelAnimationFrame(id)`。
- **requestIdleCallback(callback)**：浏览器空闲时执行，取消用 `cancelIdleCallback(id)`。同样存在闭包引用问题。

### setTimeout vs setInterval 的差异

| 特性 | setTimeout | setInterval |
|------|-----------|-------------|
| 执行次数 | 一次 | 重复 |
| 间隔保证 | 从回调结束到下次触发的间隔 ≥ delay | 从开始到下次触发的间隔 ≈ interval |
| 堆叠风险 | 无 | 回调执行时间 > interval 时会堆叠 |
| 替代方案 | — | 用 setTimeout 递归替代更可控 |

```javascript
// setTimeout 递归替代 setInterval（推荐）
function poll(fn, interval) {
  let timerId;
  function tick() {
    fn();
    timerId = setTimeout(tick, interval);
  }
  timerId = setTimeout(tick, interval);
  return () => clearTimeout(timerId); // 返回清理函数
}
```

## 工程瓶颈

1. **SPA 组件卸载遗漏**：组件销毁时未清除在 `useEffect` 或生命周期中设置的定时器。
2. **setInterval 堆叠**：回调执行时间超过间隔时，回调不断堆积导致内存和 CPU 压力。
3. **闭包持有 DOM 引用**：定时器回调闭包持有已移除 DOM 节点的引用，导致 Detached DOM 泄漏。
4. **第三方库定时器**：某些库（如轮询库、监控 SDK）内部启动定时器但不暴露清理接口。
5. **Web Worker 中的定时器**：Worker 中的定时器在 Worker terminate 后可能残留。
6. **递归 setTimeout 累积**：错误的递归 setTimeout 模式可能导致多条并行链路。

## 典型权衡

| 维度 | 权衡 |
|------|------|
| setInterval vs setTimeout 递归 | setInterval 简单但易堆叠；setTimeout 递归更可控但代码稍复杂 |
| 清理粒度 | 每个定时器单独清理更安全；批量管理（如定时器池）更方便 |
| 延迟精度 | setTimeout/setInterval 最小延迟约 4ms（嵌套后递增），需要精确时序用 requestAnimationFrame |
| 闭包最小化 | 拆分闭包减少引用链 → 代码可读性降低 |
| 轮询 vs WebSocket | 定时器轮询简单但浪费资源；WebSocket 推送高效但需服务端支持 |

## 最小验证实验

### 实验 1：闭包泄漏

```javascript
// 1. 拍摄 Heap Snapshot 1
// 2. 执行
function startPolling() {
  const cache = new Array(100000).fill({ data: 'x'.repeat(1000) });

  setInterval(() => {
    // 闭包引用了 cache，即使 cache 本可释放
    if (cache.length > 0) {
      console.log('polling...');
    }
  }, 5000);
  // 未存储 timer ID → 无法清除
}
startPolling();

// 3. GC → 拍摄 Heap Snapshot 2
// 4. 搜索 "setInterval" 或 "cache"，观察引用链
```

### 实验 2：定时器清理模式

```javascript
// React useEffect 清理模式
function usePolling(url, interval) {
  useEffect(() => {
    const timerId = setInterval(() => {
      fetch(url).then(/* ... */);
    }, interval);

    return () => clearInterval(timerId); // 清理
  }, [url, interval]);
}

// 通用清理函数工厂
function createTimer(fn, interval) {
  const id = setInterval(fn, interval);
  return {
    clear: () => clearInterval(id),
    // 替换回调（断开旧闭包引用）
    replace: (newFn) => {
      clearInterval(id);
      return createTimer(newFn, interval);
    }
  };
}
```

### 实验 3：setInterval 堆叠检测

```javascript
let lastRun = Date.now();
const id = setInterval(() => {
  const now = Date.now();
  const drift = now - lastRun - 100; // 期望 100ms 间隔
  if (drift > 50) {
    console.warn(`Timer drift: ${drift}ms (callback too slow or stacking)`);
  }
  lastRun = now;
  // 模拟慢回调
  for (let i = 0; i < 1e7; i++) Math.random();
}, 100);
```

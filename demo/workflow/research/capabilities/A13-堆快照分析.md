# A13 - 堆快照分析

## 核心机制

### Heap Snapshot 基本概念

Chrome DevTools Memory 面板的 Heap Snapshot 功能可以拍摄 V8 堆内存的完整快照，记录所有 JavaScript 对象、DOM 节点及其引用关系。

**快照内容包含**：
- 所有 JavaScript 对象（含闭包、数组、函数等）
- DOM 节点（含 detached 状态的节点）
- 对象之间的引用关系（构成对象图）
- 每个对象的内存大小（Shallow Size + Retained Size）

### 快照拍摄方式

1. **Heap Snapshot**：拍摄当前堆的瞬时快照
2. **Allocation instrumentation on timeline**：在时间线上记录对象分配
3. **Allocation sampling**：通过采样方式记录分配（开销更低）

### 核心分析视图

#### Summary 视图
按构造函数名称分组显示对象：
- **Constructor**：对象类型（如 `Array`、`Object`、`Closure`）
- **Distance**：到 GC Root 的最短路径长度
- **Shallow Size**：对象自身占用的内存
- **Retained Size**：释放该对象后可回收的总内存（含其独占的子对象）

#### Comparison 视图（关键）
拍摄两个快照进行对比：
- **#Delta**：对象数量变化
- **Size Delta**：内存大小变化
- **Alloc. Size**：新增分配的内存
- **Freed Size**：释放的内存

**典型流程**：快照 1 → 执行操作 → 手动 GC → 快照 2 → 对比

#### Containment 视图
展示对象之间的引用树结构，可追踪：
- GC Root → ... → 目标对象的完整引用链
- 识别哪些对象阻止了目标对象被回收

#### Dominators 视图
展示支配者树（Dominator Tree）：
- 若 A 是 B 的支配者，释放 A 必然释放 B
- 用于快速定位最大的内存占用者

### Detached DOM 节点

当 DOM 节点从文档树移除但仍有 JavaScript 引用时，成为 **Detached DOM 节点**。这是最常见的内存泄漏模式之一。在 Heap Snapshot 中搜索 `Detached` 可快速定位。

## 工程瓶颈

1. **快照体积巨大**：大型应用快照可达数百 MB，DevTools 加载和分析非常慢。
2. **GC 时机干扰**：拍摄快照前未手动 GC 会导致大量"假阳性"死对象出现在快照中。
3. **Comparison 误读**：两个快照间的 GC 时机不同，Delta 可能产生误导。
4. **Closure 追踪困难**：闭包的引用链可能非常深且隐晦，难以定位泄漏源。
5. **跨框架/跨 Worker**：Heap Snapshot 仅覆盖当前 V8 Isolate，无法分析 Worker 或 iframe 中的内存。
6. **Retained Size 计算**：对于共享引用的对象，Retained Size 的计算基于支配者假设，可能不完全准确。
7. **符号化与 Source Map**：压缩后的代码在快照中难以对应源码位置。

## 调试工具

| 工具 | 用途 |
|------|------|
| Chrome DevTools → Memory 面板 | Heap Snapshot 拍摄、对比、分析 |
| `HeapSnapshot.format` (Chrome flag) | 导出 `.heapsnapshot` JSON 文件 |
| `chrome://inspect` → Memory | 远程调试 Node.js 堆快照 |
| `--heapsnapshot-signal=SIGUSR2` (Node) | 向 Node 进程发送信号生成快照 |
| `v8.writeHeapSnapshot()` (Node 14+) | 程序化生成堆快照文件 |
| `heapdump` / `heap-profile` npm 包 | Node.js 堆快照生成工具 |
| Chrome DevTools → Allocation Timeline | 对象分配时间线，追踪分配热点 |
| DevTools → Allocation Sampling | 低开销的分配采样分析 |

## 典型权衡

| 维度 | 权衡 |
|------|------|
| Snapshot vs Timeline | Snapshot 精确但开销大；Timeline 采样开销小但可能遗漏 |
| 手动 GC 前后 | 不 GC → 看到更多"噪声"但更真实；GC 后 → 更干净但可能遗漏瞬态泄漏 |
| Summary vs Comparison | Summary 适合全局概览；Comparison 适合定位增量泄漏 |
| Shallow vs Retained | Shalone Size 大不一定是问题；Retained Size 大说明该对象阻止了大量内存释放 |
| 生产环境快照 | 精确但对性能影响大；采样模式更适合生产 |

## 最小验证实验

### 实验 1：Detached DOM 节点泄漏

```javascript
// 1. 打开 Chrome DevTools → Memory → 拍摄 Heap Snapshot 1
// 2. 执行以下代码
const detachedNodes = [];
function createLeak() {
  const div = document.createElement('div');
  div.innerHTML = 'test'.repeat(1000);
  document.body.appendChild(div);
  document.body.removeChild(div); // 从 DOM 移除
  detachedNodes.push(div); // 但 JS 仍持有引用 → 泄漏
}
for (let i = 0; i < 100; i++) createLeak();

// 3. 点击 "Collect garbage" 按钮
// 4. 拍摄 Heap Snapshot 2
// 5. 切换到 Comparison 视图，对比 Snapshot 1 vs 2
// 6. 搜索 "Detached"，观察 detached HTMLDivElement 数量
```

### 实验 2：闭包导致的内存泄漏

```javascript
// 1. 拍摄 Heap Snapshot 1
// 2. 执行
let leakRef = null;
function heavyOperation() {
  const largeData = new Array(100000).fill('x'.repeat(100));
  leakRef = function() {
    // 闭包捕获了 largeData 的引用
    return largeData.length;
  };
}
heavyOperation();

// 3. GC → 拍摄 Heap Snapshot 2
// 4. 在 Containment 视图中找到 leakRef → 追踪引用链 → 定位 largeData
```

### 实验 3：程序化堆快照（Node.js）

```javascript
// Node.js 14+
const v8 = require('v8');
const fs = require('fs');

const snapshotStream = v8.writeHeapSnapshot();
console.log(`Heap snapshot written to: ${snapshotStream}`);
// 用 Chrome DevTools Memory 面板加载 .heapsnapshot 文件进行分析
```

# 请求并发与数据预取

> ID: A15 | 扇出: 2/8 | 耦合度: 1 | 战略价值: 1.0 | 🏕️ 三级能力

## 核心机制

**并发控制**：
- `Promise.all([req1, req2, req3])` — 全部并发，任一失败整体失败
- `Promise.allSettled([...])` — 全部并发，等待全部完成（不管成功失败）
- `Promise.race([...])` — 竞速，取最快的结果

**并发限制**（避免浏览器 6 连接限制 / 服务器压力）：
```javascript
async function parallelLimit(tasks, limit = 4) {
  const results = [];
  const executing = new Set();
  for (const [i, task] of tasks.entries()) {
    const p = task().then(r => { executing.delete(p); return r; });
    executing.add(p);
    results[i] = p;
    if (executing.size >= limit) await Promise.race(executing);
  }
  return Promise.all(results);
}
```

**BFF 聚合层**：在 Backend-for-Frontend 层将多个微服务接口聚合为一个前端请求，减少网络往返。

**数据预取**：在用户可能操作前预加载数据（hover 预取、路由预取）。

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|---------|---------|---------|---------|
| 1 | 请求瀑布流 | 多个接口串行调用 | 首屏加载时间 = 所有请求时间之和 | Network 面板 → 请求瀑布图 | Promise.all 并行化 |
| 2 | 重复请求 | 同一数据被多个组件同时请求 | 浪费带宽 + 竞态条件 | Network 面板 → 重复 URL | 请求去重（SWR/React Query） |
| 3 | 请求竞态 | 快速切换页面，旧请求的响应覆盖新数据 | 显示过期数据 | 代码审查 | AbortController 取消旧请求 |

## 典型权衡

| 维度 | 方案 A | 方案 B | 选择建议 |
|------|--------|--------|---------|
| 数据获取 | 客户端并发请求（灵活但请求数多）| BFF 聚合（减少请求但增加服务端复杂度）| 接口 > 3 个时用 BFF 聚合 |
| 缓存策略 | React Query/SWR（自动缓存+去重+重试）| 手动管理（完全可控但复杂）| 推荐 React Query/SWR |

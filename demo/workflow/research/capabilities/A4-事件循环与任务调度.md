## 核心机制

### 1. 事件循环（Event Loop）基本模型

JavaScript 是单线程语言，通过 **事件循环** 实现异步非阻塞。每个浏览器 tab 对应一个 agent（可理解为主线程），agent 内部维护三个核心数据结构：

| 结构 | 作用 | 特性 |
|------|------|------|
| **Heap** | 存储对象的内存区域 | 无结构，GC 管理 |
| **Queue（Job Queue）** | 待执行任务的排队区 | FIFO，分为宏任务队列和微任务队列 |
| **Stack（Call Stack）** | 执行上下文调用栈 | LIFO，函数调用/返回 |

**事件循环的每一轮迭代（tick）流程：**

```
1. 从宏任务队列取出一个 task 执行
2. 执行完毕后，清空所有微任务队列（microtask checkpoint）
3. 如果需要渲染：
   a. 触发 resize/scroll 事件
   b. 触发 IntersectionObserver 回调
   c. 执行 requestAnimationFrame 回调
   d. 执行 IdleDeadline 回调（requestIdleCallback）
   e. 执行渲染更新（Layout → Paint → Composite）
4. 回到步骤 1，取出下一个宏任务
```

### 2. 宏任务（Macrotask）与微任务（Microtask）

**宏任务** —— 事件循环每轮迭代取出一个执行：

| 来源 | 示例 |
|------|------|
| Script | `<script>` 标签执行 |
| Timer | `setTimeout`, `setInterval` |
| I/O | 网络请求回调、文件读写 |
| UI Rendering | 浏览器渲染步骤 |
| MessageChannel | `port.postMessage` |
| `requestAnimationFrame` | 每帧回调（浏览器渲染阶段） |
| `requestIdleCallback` | 空闲期回调 |

**微任务** —— 每个宏任务结束后立即清空整个微任务队列：

| 来源 | 示例 |
|------|------|
| `Promise.then/catch/finally` | Promise 链式回调 |
| `MutationObserver` | DOM 变更观察 |
| `queueMicrotask()` | 手动入队微任务 |
| `process.nextTick` | Node.js 专有，优先级最高 |
| `await` 后续代码 | async/await 暂停后的恢复 |

**关键执行顺序：** `同步代码 → 全部微任务 → 下一个宏任务 → 全部微任务 → ...`

### 3. requestAnimationFrame（rAF）

```js
requestAnimationFrame(callback)  // callback 接收 DOMHighResTimeStamp
```

- **调用时机：** 在浏览器下一次重绘之前调用，通常与显示刷新率同步（60Hz → ~16.67ms/帧）
- **一次性：** 需要在回调中再次调用 rAF 才能持续动画
- **后台节流：** 页面隐藏或 iframe 不可见时，浏览器暂停 rAF 调用
- **时间戳共享：** 同一 agent 内所有 rAF 回调收到相同的时间戳，保证同步动画一致性
- **用例：** UI 动画、与视觉刷新同步的计算

### 4. requestIdleCallback（rIC）

```js
requestIdleCallback(callback, { timeout: 1000 })
// callback 接收 IdleDeadline 对象
//   deadline.timeRemaining() → 当前帧剩余空闲时间(ms)
//   deadline.didTimeout → 是否因 timeout 触发
```

- **调用时机：** 浏览器空闲期间，不阻塞关键渲染和输入响应
- **FIFO 顺序：** 通常按入队顺序执行，但带 timeout 的可能提前执行
- **建议设置 timeout：** 否则可能数秒不触发回调
- **兼容性有限：** Safari 不支持，需 polyfill 或 `MessageChannel` + `rAF` 模拟
- **用例：** 非紧急后台任务：日志上报、预计算、空闲期 DOM 批量更新

### 5. 长任务（Long Task）检测

**Long Tasks API**（`PerformanceObserver` + `longtask` entry type）：

```js
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.warn('Long task:', entry.duration, 'ms', entry.attribution);
  }
});
observer.observe({ entryTypes: ['longtask'] });
```

- **阈值：** 单个 task 执行超过 **50ms** 即被记录为 long task
- **归因信息：** `entry.attribution` 包含任务来源（containerType, containerName, containerId）
- **INP 关联：** Long task 是影响 Interaction to Next Paint（INP）指标的主要原因

## 工程瓶颈

| # | 瓶颈名 | 触发条件 | 表现症状 | 检测手段 | 缓解策略 |
|---|--------|----------|----------|----------|----------|
| 1 | **主线程长任务阻塞** | 单个 task > 50ms（复杂计算/大量 DOM 操作） | 页面卡顿、输入无响应、INP 劣化 | Performance 面板 Long Tasks 标记；`PerformanceObserver({entryTypes:['longtask']})` | ① 拆分为 < 50ms 的小任务 ② 用 `requestIdleCallback` 延迟非关键工作 ③ Web Worker 卸载计算 |
| 2 | **微任务队列饥饿** | 同步创建大量 Promise/microtask，微任务队列过长 | 宏任务（包括渲染）被延迟，页面假死 | Performance 面板查看 microtask checkpoint 耗时 | ① 分批处理 Promise 链 ② 用 `setTimeout(fn, 0)` 将部分工作降级为宏任务 ③ 避免无限微任务递归 |
| 3 | **rAF 回调过重** | rAF 回调内执行复杂计算/大量 DOM 读写 | 掉帧、动画卡顿、CPU 持续高占用 | Performance 面板 rAF 区间 > 16ms；FPS meter | ① 只在 rAF 中做轻量状态更新 ② 计算密集工作移到 rIC 或 Worker ③ 使用 `will-change` 提升合成层 |
| 4 | **rIC 回调不触发** | 网页持续高负载，无空闲期；或 Safari 不支持 | 后台任务永远不执行 | `console.log` 跟踪；超时时间过长 | ① 设置合理 timeout（如 1000ms） ② polyfill：用 `MessageChannel` + `rAF` 剩余时间模拟 ③ 降级为 `setTimeout` |
| 5 | **Timer 嵌套过深** | `setTimeout`/`setInterval` 嵌套层级过深 | 浏览器强制最小延迟 4ms（嵌套 ≥ 5 层），实际延迟远大于预期 | Performance 面板查看 timer 间隔 | ① 减少嵌套层级 ② 用 rAF 替代定时动画 ③ 使用 `requestAnimationFrame` 与 `requestIdleCallback` 交替调度 |
| 6 | **事件风暴** | 高频 input/scroll/resize 事件连续触发 | 主线程被事件回调占满，渲染延迟 | Long Tasks 标记 + 事件归因 | ① 使用 `requestAnimationFrame` 合并 ② 用 `AbortController` 取消不需要的监听 ③ passive event listener |

## 调试工具

| 工具 | 用法 |
|------|------|
| **Chrome DevTools → Performance** | 录制页面活动，查看 Main 轨道中的 task 块，红色三角标记 > 50ms 的 Long Task，展开查看调用栈 |
| **Chrome DevTools → Performance Monitor** | 实时查看 CPU usage、FPS、DOM 节点数、JS heap size |
| **`PerformanceObserver` (longtask)** | 代码级监控 Long Task，上报 RUM 数据 |
| **`performance.mark()` / `performance.measure()`** | 自定义标记关键时间点，配合 Performance 面板分析 |
| **Lighthouse** | 自动审计 TBT（Total Blocking Time）、INP 相关建议 |
| **Web Vitals 库** | `web-vitals` npm 包，一行代码上报 INP/LCP/CLS 等 Core Web Vitals |
| **Chrome Task Manager** | `Shift+Esc`，查看各 tab/扩展的 CPU 和内存占用 |
| **`console.time()` / `console.timeEnd()`** | 快速测量同步代码段耗时 |

## 典型权衡

| 维度 | 方案A | 方案B | 选择建议 |
|------|-------|-------|----------|
| **长任务拆分方式** | `setTimeout(fn, 0)` 递归拆分 | `requestIdleCallback` 空闲期执行 | 需要尽快执行 → setTimeout；可延迟 → rIC（对渲染更友好） |
| **动画驱动** | `setInterval` + 手动计算 | `requestAnimationFrame` | **总是选 rAF**：自动同步刷新率、后台暂停、时间戳精确 |
| **计算密集任务** | 主线程 + 分片 | Web Worker | CPU 密集且无 DOM 访问 → Worker；需 DOM → 分片调度 |
| **微任务 vs 宏任务** | `queueMicrotask` 保证尽快执行 | `setTimeout` 延迟到下一轮 | 需要保证在渲染前完成 → 微任务；允许渲染插入 → 宏任务 |
| **事件监听节流** | rAF 节流（每帧只执行一次） | `throttle(fn, 16ms)` | 视觉相关 → rAF；非视觉 → throttle 即可 |
| **rIC 超时策略** | 不设 timeout，等真正空闲 | 设 timeout 保证执行 | 关键后台任务（如指标上报）→ 设 timeout；纯优化类任务 → 不设 |
| **Safari 兼容** | polyfill rIC（MessageChannel 模拟） | 降级为 setTimeout | 生产环境必须 polyfill 或降级，Safari 不支持 rIC |

## 最小验证实验

### 实验 1：宏任务/微任务执行顺序验证

```html
<!DOCTYPE html>
<html><body>
<script>
console.log('1. sync');

setTimeout(() => console.log('4. macrotask (setTimeout)'), 0);

Promise.resolve().then(() => {
  console.log('2. microtask (Promise)');
  Promise.resolve().then(() => console.log('3. microtask (nested Promise)'));
});

queueMicrotask(() => console.log('2.5. microtask (queueMicrotask)'));

console.log('1.5. sync');
</script>
</body></html>
```

**预期输出：** `1 → 1.5 → 2 → 2.5 → 3 → 4`

### 实验 2：rAF 与渲染同步

```js
let lastTime = performance.now();
function animate(timestamp) {
  const delta = timestamp - lastTime;
  console.log(`帧间隔: ${delta.toFixed(2)}ms`);
  lastTime = timestamp;
  requestAnimationFrame(animate);
}
requestAnimationFrame(animate);
// 预期：每帧间隔 ≈ 16.67ms（60Hz 屏幕）
```

### 实验 3：rIC 空闲调度

```js
function doWork(deadline) {
  while (deadline.timeRemaining() > 0 && tasks.length > 0) {
    processTask(tasks.shift());
  }
  if (tasks.length > 0) {
    requestIdleCallback(doWork, { timeout: 1000 });
  }
}
requestIdleCallback(doWork, { timeout: 1000 });
// 每帧利用剩余空闲时间处理任务，不影响渲染
```

### 实验 4：Long Task 检测

```js
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.warn(`Long task detected: ${entry.duration.toFixed(1)}ms`);
  }
});
observer.observe({ entryTypes: ['longtask'] });

// 触发一个长任务
function longTask() {
  const end = performance.now() + 100;
  while (performance.now() < end) {} // 阻塞 100ms
}
setTimeout(longTask, 1000);
```

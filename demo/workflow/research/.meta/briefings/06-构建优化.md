# P6-构建产物 — 组装 Briefing

## 命题信息
命题：构建产物：Code Splitting/Tree Shaking工程化配置
限定词：构建工具(Webpack/Vite)
通用占比：≥80%（通用层跨构建+网络+浏览器3层）

## 涉及能力摘要

### A7-资源加载策略 [用于: overview, edge-cases, trade-offs]
机制：通过浏览器原生 Resource Hints（preload/prefetch/preconnect/dns-prefetch）和现代加载属性（loading='lazy'/fetchpriority）实现资源优先级调度。preload 用于当前导航关键资源的高优先级预加载（需指定 as 属性），prefetch 用于未来导航资源的低优先级预获取，preconnect 提前完成 DNS+TCP+TLS 握手，lazy loading 延迟非首屏资源加载，fetchpriority 在同类资源内微调优先级（high/low/auto）。这些机制共同作用于浏览器的内部优先级模型，直接影响 LCP、FCP 等 Core Web Vitals。
瓶颈：
  - B1 preload 未被使用导致带宽浪费和 Console 警告（资源边界，P1）
  - B2 preload 与实际请求的 CORS 模式不匹配导致双重下载（输入变异，P0）
  - B3 过多 prefetch 与首屏关键资源争抢带宽导致 LCP 退化（时序竞争，P1）
  - B4 preconnect 超过 6 个域名浪费 DNS/TCP 资源（资源边界，P2）
  - B5 首屏 LCP 图片错误使用 loading='lazy' 导致 LCP 延迟（输入变异，P0）
  - B6 所有资源都设为 fetchpriority='high' 导致优先级提示失效（输入变异，P1）
  - B7 preload 写在 CSS @import 内或动态注入太晚失去预加载意义（时序竞争，P1）
权衡：
  - preload 当前关键资源 vs prefetch 未来页面资源：首屏优先用 preload，交互后用 prefetch，preload 控制在 5 个以内
  - 原生 lazy loading vs Intersection Observer JS 懒加载：优先用原生零成本方案，需要自定义阈值时用 IO
  - preconnect vs dns-prefetch：关键第三方域名用 preconnect，其余用 dns-prefetch，每页 preconnect 不超过 6 个

### A9-模块系统与构建优化 [用于: overview, edge-cases, trade-offs, experiment]
机制：ES Module的静态import/export结构使构建工具能在编译期分析完整依赖图，这是Tree Shaking的前提。动态import()返回Promise实现按需加载，构建工具将其识别为split point自动生成独立chunk。Tree Shaking通过usedExports标记和sideEffects剪枝两层机制消除死代码——sideEffects声明允许跳过整个模块子树。Code Splitting有三种策略：Entry Points手动分割、SplitChunksPlugin自动提取公共依赖、Dynamic Import按需加载。Vite利用浏览器原生ESM+esbuild预构建实现极快启动，生产模式基于Rollup打包。
瓶颈：
  - B1 Tree Shaking失效：CommonJS模块混入、模块顶层副作用、require()动态导入导致未消除死代码（输入变异，P0）
  - B2 过度分割：minSize过小导致chunk数量爆炸，HTTP请求激增，首屏加载反而变慢（资源边界，P0）
  - B3 重复依赖：多chunk引用同依赖不同版本，同一库被多次打包（资源边界，P1）
  - B4 动态import加载抖动：网络不稳定导致chunk加载失败和白屏（输入变异，P1）
  - B5 CJS/ESM互操作开销：混合模块系统产生大量shim代码和运行时包装器（资源边界，P1）
  - B6 构建缓存失效频繁：模块边界不合理导致增量构建缓存命中率低（时序竞争，P2）
权衡：
  - 分割粒度：粗粒度（少量大chunk）vs 细粒度（大量小chunk），推荐路由级中等粒度
  - 首屏vs按需：全量打包initial chunk vs 路由/功能动态import，首屏关键路径保持initial
  - Tree Shaking vs 兼容性：纯ESM依赖（最佳Tree Shaking）vs CJS依赖（兼容性好），优先选ESM版本
  - sideEffects声明：激进false vs 逐文件声明有副作用文件，项目代码推荐false，第三方需验证
  - runtime chunk：独立runtimeChunk vs 内联到entry，多入口必须独立，单入口可内联
  - 缓存策略：contenthash长期缓存 vs 版本号短期缓存，生产环境必须用contenthash

## 内容比例约束
开篇 10-15%：从构建产物体积膨胀痛点切入（bundle 过大、首屏加载慢、Tree Shaking 无效）
主体 70-80%：通用工程原理（ES Module 静态分析、Tree Shaking 机制、Code Splitting 策略、资源加载优先级）
收尾 10-15%：回到构建工具给落地方案（Webpack splitChunks / Vite Rollup 配置建议）

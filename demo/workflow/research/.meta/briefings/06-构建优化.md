## 涉及能力摘要

### A10-A10
机制：Code Splitting 将代码拆分为多个 chunk 按需加载，减少首屏 JS 体积。三种方式：Entry Points 手动拆分入口、SplitChunksPlugin 自动提取公共依赖、Dynamic Import 运行时懒加载。Webpack 将 import() 作为异步 chunk 边界，配合 cacheGroups 配置实现 vendor/runtime/业务代码分离。核心目标是减少首屏传输量并利用浏览器并行加载能力。
瓶颈：
  - 过度拆分导致 HTTP 请求过多，增加网络开销
  - chunk 间共享模块版本不一致导致重复打包
  - 动态导入需要处理加载状态（React.lazy + Suspense）
  - 首屏 vs 懒加载边界判断需要业务分析
  - 预加载/预取时机控制不当导致带宽浪费
权衡：
  - ****：细粒度拆分：按组件/路由级别拆分，减少单次传输量 vs 粗粒度拆分：减少请求数，降低延迟，建议：
  - ****：SplitChunksPlugin 自动化：配置简单，粒度较粗 vs 手动 Entry + import()：灵活可控，维护成本高，建议：

### A11-A11
机制：Tree Shaking 基于 ESM 静态结构，在构建阶段标记并移除未被引用的导出代码。两个层次：usedExports 标记未使用导出供 Terser 删除，sideEffects 允许跳过整个无副作用文件。CommonJS 无法被 Tree Shaking，Babel 默认转 ESM 为 CJS 会破坏此能力。需配合生产模式和正确的 sideEffects 声明才能生效。
瓶颈：
  - 副作用检测困难，Terser 无法可靠判断动态语言中的函数调用是否有副作用
  - CommonJS 库（如 lodash）无法 Tree Shaking，必须使用 ESM 版本
  - Babel 默认将 ESM 转为 CJS，需配置 modules: false 保留静态结构
  - CSS Tree Shaking 需额外工具（PurgeCSS），Webpack 原生不支持
  - re-export 场景可能导致误删
权衡：
  - ****：sideEffects：跳过整个未引用文件，效率最高 vs usedExports：标记单个导出，粒度更细，建议：
  - ****：启用：开发时即可发现未使用代码 vs 不启用：加快构建速度，生产模式自动启用，建议：
实验代码：// src/math.js

export function cube(x) { return x * x * x; }

// src/index.js
import { cube } from './math.js';
console.log(cube(5));

// webpack.config.js

### A28-A28
机制：ESM 使用 import/export 声明式语法，在编译期静态分析依赖图，是 Tree Shaking 的前提。CommonJS 通过 require() 运行时动态加载，导出值拷贝，无法被静态分析裁剪。import() 是唯一兼具 ESM 语法和动态加载的方式，Webpack 将其作为 Code Splitting 边界。Node.js 通过 .mjs/.cjs 后缀和 package.json type 字段管理双模块系统。
瓶颈：
  - CJS 模块无法 Tree Shaking，整个包必须全部打包
  - 混合模块系统（CJS + ESM）导致 bundler 需要复杂的互操作逻辑
  - 循环依赖在两种模块系统中行为不同，可能引发 TDZ 或部分初始化问题
  - 动态 require(variable) 阻碍静态分析，bundler 无法生成完整依赖图
  - 某些 npm 包的 __esModule 标记缺失导致 named import 失败
权衡：
  - ****：ESM：获得 Tree Shaking、静态分析、浏览器原生支持 vs CJS：最大兼容性，所有 Node.js 版本和工具链支持，建议：
  - ****：仅 ESM：最小包体积，强制用户升级 vs CJS + ESM 双输出：兼容性最好，但维护成本高，建议：

### A29-A29
机制：持久化缓存通过 contenthash 实现：文件内容不变则文件名不变，浏览器可长期缓存。关键措施包括 runtimeChunk 分离（避免 runtime 变化污染业务 chunk hash）、SplitChunksPlugin 提取 vendor chunk（第三方库独立缓存）、Webpack 5 的 deterministic moduleIds（解决模块 ID 不确定性）。理想效果是修改业务代码时只有 main chunk 的 hash 变化。
瓶颈：
  - 不分离 runtime 时，runtime 变化导致所有 chunk hash 污染
  - 模块 ID 基于解析顺序递增，增删模块导致其他模块 ID 变化
  - npm update 改变 node_modules 内容，vendor chunk 全部失效
  - CSS 提取插件的 contenthash 需要与 JS 独立计算
  - source map 文件的缓存策略需要特殊处理
权衡：
  - ****：单一 vendor chunk：缓存命中率高，但单次更新代价大 vs 按库拆分多个 vendor chunk：更新粒度细，但请求数增加，建议：
  - ****：短 hash（8 字符）：文件名短，碰撞概率略高 vs 长 hash（20 字符，Webpack 默认）：碰撞概率极低，建议：

### W1-W1
机制：Webpack 构建由 Loader 和 Plugin 驱动。Loader 是模块转换函数，执行顺序从右到左，处理单个文件的格式转换（如 Babel 转译、CSS 解析）。Plugin 通过 Tapable 事件系统挂载到 Compiler/Compilation 的各个 Hook，处理全局构建逻辑（如 HTML 生成、代码压缩）。构建流程：初始化 → 编译模块（Loader 链）→ 优化 → 输出（Plugin Hooks）。
瓶颈：
  - Loader 执行顺序（从右到左）容易配置错误
  - Plugin 的 Hook 时机选择不当会导致访问未初始化的资源
  - Loader 间数据传递依赖字符串，复杂转换链难以调试
  - Pitching Loader 可跳过后续 loader，增加调试难度
  - 某些 Plugin（CompressionPlugin）在 emit 阶段大量计算拖慢构建
权衡：
  - ****：完整 Loader 链（Babel + PostCSS + ESLint）：功能完善 vs 精简 Loader 链（esbuild-loader 替代 Babel）：构建提速 10-100x，建议：
  - ****：tap（同步）：性能好，适合简单逻辑 vs tapAsync/tapPromise（异步）：适合 I/O 密集操作，建议：

### VI1-VI1
机制：Vite 开发模式基于浏览器原生 ESM，通过 esbuild 进行依赖预构建：将 CJS/UMD 转为 ESM，将多模块依赖（如 lodash-es 600+ 文件）合并为单文件减少 HTTP 请求。HMR 通过文件监听 + WebSocket + ESM 模块缓存失效实现，性能与项目大小无关。生产构建使用 Rolldown（之前是 Rollup）。预构建产物缓存在 node_modules/.vite。
瓶颈：
  - 预构建缓存失效判断复杂，有时需要 --force 手动刷新
  - 某些 CJS 依赖的动态 require() 无法被 esbuild 完全转换
  - 开发（esbuild）与生产（Rolldown）构建行为差异可能导致环境不一致
  - esbuild 不支持 legacy 装饰器和某些 Babel 插件
  - 大型 monorepo 的 linked dependencies 预构建配置复杂
权衡：
  - ****：esbuild：极快（Go 原生），但不支持所有 Babel 插件 vs Babel：插件生态丰富，但速度慢 10-100x，建议：
  - ****：原生 ESM + 按需编译：冷启动极快，但请求数多 vs 全量预打包：请求数少，但冷启动慢，建议：

### W2-W2
机制：Module Federation 是 Webpack 5 的架构模式，允许独立构建的应用在运行时共享模块。每个应用是一个 container，通过 remoteEntry.js 暴露模块清单。宿主应用通过 import() 异步加载远程模块，共享模块（如 React）通过版本协商避免重复打包。支持动态远程容器（运行时决定加载哪个应用）、嵌套容器和循环依赖。是微前端场景的核心解决方案。
瓶颈：
  - 共享模块版本协商失败导致运行时错误或重复打包
  - 远程容器（remoteEntry.js）加载失败时缺少降级策略
  - 远程模块的 TypeScript 类型支持需要额外配置
  - 共享模块的 CSS 可能泄漏，需要 Shadow DOM 或 CSS Modules
  - 每个应用独立 Webpack 配置，shared 配置需保持一致
权衡：
  - ****：singleton：只有一个版本，节省内存，但版本不兼容会报错 vs 多版本：安全但内存占用更大，建议：
  - ****：静态远程：配置文件声明，简单但不灵活 vs 动态远程：运行时决定，灵活但需要额外加载逻辑，建议：

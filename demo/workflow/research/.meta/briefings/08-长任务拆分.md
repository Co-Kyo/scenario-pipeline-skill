# P8-长任务拆分 — 组装 Briefing

## 命题信息
命题：长任务拆分：requestIdleCallback/Web Worker主线程卸载
限定词：无（纯通用层）
通用占比：≥70%（通用层跨浏览器+多线程+工具3层）

## 涉及能力摘要

### A4-事件循环与任务调度 [用于: overview, edge-cases, trade-offs]
机制：事件循环单线程异步模型，宏任务/微任务队列执行优先级，requestAnimationFrame 渲染同步调度，requestIdleCallback 空闲期调度，Long Tasks API（50ms 阈值）长任务检测。
瓶颈：
  - B1 主线程长任务阻塞（时序竞争，P0）：大量同步 JS 执行、复杂计算未分片 → 页面卡顿、点击无响应、INP 劣化
  - B2 微任务队列饥饿（时序竞争，P0）：微任务队列过长（递归 Promise/MutationObserver）→ 宏任务被饿死，页面无响应
  - B3 rAF 回调过重（时序竞争，P1）：rAF 回调中执行复杂计算 → 掉帧，动画不流畅
  - B4 rIC 不触发（输入变异，P2）：Safari 不支持 rIC、高负载下浏览器不安排空闲期 → 低优先级任务永远不执行
权衡：
  - 任务拆分：requestIdleCallback（低优先级，Safari 不支持）vs scheduler.yield()（新 API，切分到下一个宏任务）
  - 动画驱动：setTimeout（不精确）vs requestAnimationFrame（与渲染同步），动画必须用 rAF

### A8-DevTools性能分析 [用于: overview, edge-cases, experiment]
机制：Performance 面板录制分析（火焰图/Long Task/帧率）、Memory 面板堆快照（Heap Snapshot/Allocation timeline）、Lighthouse 自动化审计（FCP/LCP/TBT/CLS）、PerformanceObserver API 异步观察性能条目、Long Tasks API 主线程阻塞检测（>50ms+attribution 归属）。
瓶颈：
  - B1 JavaScript 主线程阻塞（时序竞争，P0）：大量同步 JS 执行 → 页面卡顿、LCP/INP 差
  - B2 内存泄漏（状态跃迁，P0）：事件监听器未移除、闭包持有引用 → Heap 持续增长
  - B3 布局抖动 Layout Thrashing（时序竞争，P0）：JS 中交替读写 layout 属性 → 帧率骤降
权衡：
  - 性能监控：PerformanceObserver API（实时精确）vs web-vitals 库（封装好 2KB），生产用 web-vitals
  - 内存分析：Heap Snapshot（完整精准但文件大）vs Allocation Sampling（采样开销低）
  - Long Task 监控：PerformanceObserver+longtask（精确有 attribution）vs rAF+时间差检测（兼容性好）

### A12-Web Worker多线程 ⚠️ 该能力摘要缺失
机制（来自 capability-graph.json）：Web Worker 创建/通信 postMessage、SharedArrayBuffer、主线程卸载。
描述：Web Worker 是浏览器提供的多线程能力，允许在独立线程中执行 JS 计算而不阻塞主线程。通过 postMessage 进行主线程与 Worker 线程间的数据通信（序列化/反序列化），SharedArrayBuffer 支持共享内存（需 COOP/COEP 响应头配合）。Worker 无法访问 DOM，适合计算密集型任务卸载。

## 内容比例约束
开篇 10-15%：从长任务导致 INP 劣化切入，说明主线程拆分与 Worker 卸载的必要性
主体 70-80%：通用工程原理（事件循环调度模型、Worker 线程隔离、DevTools 诊断方法论）
收尾 10-15%：落地建议（渐进式拆分策略、Worker 选型、监控闭环）

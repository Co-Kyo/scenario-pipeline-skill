# P3-内存泄漏 — 组装 Briefing

## 命题信息
命题：内存泄漏：长时间运行页面的内存泄漏排查与治理——H5/SPA场景
限定词：H5/SPA
通用占比：≥80%（通用层跨JS引擎GC+DOM生命周期+DevTools诊断3层）

## 涉及能力摘要

### A2-DOM节点生命周期 [用于: overview, edge-cases, trade-offs]
机制：DOM 节点从创建到回收的完整生命周期：创建（createElement/createTextNode）→ 挂载（appendChild/insertBefore）→ 更新（属性/子树变更）→ 卸载（removeChild/remove）→ 垃圾回收。核心风险是 Detached DOM 内存泄漏——节点从文档树移除但 JS 仍持有引用导致无法 GC。
瓶颈：
  - B1 Detached DOM 内存泄漏（状态跃迁，P0）：DOM 节点被移除后 JS 仍持有引用（闭包、全局缓存、事件监听器未解绑）→ 页面内存持续增长，长时间运行后卡顿
  - B2 事件监听器泄漏（状态跃迁，P0）：addEventListener 后未 removeEventListener，SPA 路由切换时尤为突出 → 事件回调重复执行，内存中保留大量闭包
  - B3 批量 DOM 操作触发重排（时序竞争，P1）：循环中逐个 appendChild，每次触发 Layout → 界面闪烁，FPS 下降
权衡：
  - 批量插入：DocumentFragment（保留已有节点和事件监听器）vs innerHTML（更快但销毁已有子树），有交互节点用 Fragment
  - 事件绑定：直接 addEventListener 到每个子节点 vs 事件委托绑定到父节点，动态列表用事件委托
  - 更新策略：直接操作 DOM（原生 API）vs 虚拟 DOM Diff（React/Vue），复杂 UI 状态用框架声明式更新

### A3-V8 GC机制 [用于: overview, edge-cases, trade-offs]
机制：V8 引擎的垃圾回收机制，基于分代 GC 架构将堆分为新生代和老生代，分别采用 Scavenge（Semi-Space evacuation）和 Mark-Sweep/Mark-Compact 算法。Orinoco 项目将 GC 从 stop-the-world 演进为 Parallel + Concurrent + Incremental 混合策略，大幅降低主线程暂停时间。支持 WeakRef/WeakMap/WeakSet 管理弱引用，以及 FinalizationRegistry 清理回调。
瓶颈：
  - B1 Scavenge 停顿（时序竞争，P1）：大量短生命周期对象存活导致新生代 GC 卡顿 → 主线程暂停 >10ms
  - B2 Major GC 长暂停（时序竞争，P1）：老生代碎片化严重时暂停超 100ms → 页面明显卡顿
  - B3 闭包持有引用导致隐式泄漏（状态跃迁，P0）：闭包捕获了不再需要的变量 → GC 无法回收整个作用域链
  - B4 堆碎片化（资源边界，P2）：频繁分配/释放不同大小对象导致内存浪费 → 有效可用内存减少
权衡：
  - Map vs WeakMap 缓存：Map 持有强引用永不 GC vs WeakMap 键为对象弱引用自动释放，缓存临时对象用 WeakMap
  - 大对象手动解除引用 vs 依赖自动 GC：手动 null 赋值释放更快 vs GC 周期性回收，大对象（>1MB）建议手动释放
  - 增大堆上限 vs 优化代码减少内存：--max-old-space-size 治标 vs 代码层面减少分配治本

### A8-DevTools性能分析 [用于: overview, edge-cases, experiment]
机制：基于浏览器内置工具链的前端性能诊断体系：Performance 面板录制分析（火焰图/Long Task/帧率）、Memory 面板堆快照（Heap Snapshot/Allocation timeline/Allocation sampling）、Lighthouse 自动化审计（FCP/LCP/TBT/CLS/SI）、PerformanceObserver API 异步观察性能条目（longtask/LCP/CLS/paint/resource）、Long Tasks API 主线程阻塞检测（>50ms 任务 + attribution 归属）。
瓶颈：
  - B1 内存泄漏检测（状态跃迁，P0）：事件监听器未移除、闭包持有引用、DOM 节点未清理 → Heap 持续增长，标签页崩溃
  - B2 JavaScript 主线程阻塞（时序竞争，P0）：大量同步 JS 执行 → 页面卡顿、LCP/INP 差
  - B3 布局抖动 Layout Thrashing（时序竞争，P0）：JS 中交替读写 layout 属性 → 帧率骤降
权衡：
  - 内存分析：Heap Snapshot（完整精准但文件大）vs Allocation Sampling（采样开销低），开发深度分析用 Snapshot，线上持续监控用 Sampling
  - 性能监控：PerformanceObserver API（实时精确）vs web-vitals 库（封装好 2KB），生产 RUM 用 web-vitals
  - Long Task 监控：PerformanceObserver+longtask（精确有 attribution）vs rAF+时间差检测（兼容性好）

## 内容比例约束
开篇 10-15%：从 H5/SPA 内存泄漏痛点切入（SPA 长时间运行内存暴涨、移动端标签页崩溃、用户流失）
主体 70-80%：通用工程原理（V8 GC 分代机制、DOM 生命周期与 Detached DOM、DevTools Heap Snapshot/Allocation timeline 诊断方法、WeakRef/FinalizationRegistry 治理手段）
收尾 10-15%：回到 H5/SPA 落地方案（React/Vue 组件卸载清理模式、SPA 路由切换内存回收策略、移动端内存监控与告警）

# 内存泄漏：长时间运行页面的性能退化排查 — 组装 Briefing

## 命题信息
命题：内存泄漏：长时间运行页面的性能退化排查
通用占比：85%
限定词：通用

## 涉及能力摘要

### A12-V8垃圾回收机制
机制：V8 采用分代 GC 策略：Young Generation 使用 Semi-Space Scavenge（快速复制存活对象），Old Generation 使用 Mark-Sweep-Compact（标记-清除-压缩）。Orinoco 项目引入并发标记、增量标记、并发清除和 Idle-time GC，大幅减少主线程停顿。Write Barrier 维护跨代引用关系。
瓶颈：
  - 新生代 Scavenge 停顿（存活对象过多时）
  - 老生代 Full GC 可能超过 100ms
  - 内存碎片化触发压缩的额外开销
  - Write Barrier 运行时成本
  - 对象意外晋升到老生代

### A13-堆快照分析
机制：Chrome DevTools Memory 面板的 Heap Snapshot 功能拍摄 V8 堆内存快照，支持 Summary/Comparison/Containment/Dominators 四种视图。Comparison 视图通过对比两个快照定位内存增量泄漏。Detached DOM 节点和闭包持有大对象是最常见的泄漏模式。
瓶颈：
  - 快照体积巨大（数百 MB），加载分析慢
  - GC 时机干扰导致 Comparison 误读
  - 闭包引用链深且隐晦，难以定位泄漏源
  - 仅覆盖当前 V8 Isolate，无法分析 Worker/iframe
  - Retained Size 基于支配者假设，可能不完全准确

### A14-事件监听清理
机制：addEventListener 注册的监听器会阻止回调闭包和引用对象被 GC 回收。匿名函数无法通过 removeEventListener 移除，是主要泄漏源。AbortController/AbortSignal 可批量清理监听器。once: true 触发后自动移除。SPA 组件卸载时必须清理全局对象上的监听器。
瓶颈：
  - 匿名函数监听器无法移除，是内存泄漏主要来源
  - 闭包持有大对象阻止 GC
  - SPA 路由切换时全局监听器未清理
  - 第三方库内部监听器不暴露清理接口
  - 大量子元素直接绑定监听器造成内存膨胀

### A15-定时器与闭包管理
机制：setTimeout/setInterval 回调闭包引用的对象不会被 GC 回收，直到定时器被清除。匿名闭包持有大对象是常见泄漏模式。推荐用 setTimeout 递归替代 setInterval 避免堆叠。SPA 组件卸载时必须 clearTimeout/clearInterval。闭包最小化和置空引用是断开引用链的关键策略。
瓶颈：
  - SPA 组件卸载遗漏清除定时器
  - setInterval 堆叠导致内存和 CPU 压力
  - 闭包持有已移除 DOM 节点引用
  - 第三方库内部定时器不暴露清理接口
  - 递归 setTimeout 可能产生多条并行链路

### A16-WeakMap/WeakRef弱引用
机制：WeakMap 以对象为弱引用键，键被 GC 后条目自动清除，适合附加私有数据和缓存。WeakRef (ES2021) 创建对对象的弱引用，通过 deref() 访问（可能返回 undefined）。FinalizationRegistry 在对象被 GC 后执行清理回调，但时机不确定，不应用于关键逻辑。三者共同解决"引用阻止回收"问题。
瓶颈：
  - WeakMap 不能用原始值作键
  - 不可枚举导致调试困难
  - GC 时机不确定，WeakRef.deref() 行为不可预测
  - FinalizationRegistry 回调可能不执行
  - 跨 Worker 边界不可用

### A17-DevTools Performance/Memory面板
机制：Chrome DevTools的Performance面板通过火焰图展示主线程每帧工作（JS执行、Layout、Paint、Composite），可定位Long Task和掉帧。Memory面板提供Heap Snapshot（堆快照对比检测泄漏）、Allocation Timeline（分配热点）和Allocation Sampling三种模式。Lighthouse自动化审计输出0-100评分，核心指标包括FCP、LCP、TBT、CLS。
瓶颈：
  - **火焰图Long Task定位困难**：页面存在多个Long Task，调用栈复杂 → 火焰图中红色长条，难以定位根因
  - **Heap Snapshot对比噪声**：对比快照时框架/库内部对象干扰 → 难以区分正常分配和泄漏
  - **Lighthouse评分与真实体验不一致**：实验室环境与用户实际网络/设备差异大 → Lighthouse 90+但用户反馈卡顿
  - **Performance录制的性能开销**：录制时间过长或页面过于复杂 → 录制本身导致页面变慢，数据量过大
权衡：
  - **火焰图录制深度**：浅录制(Disable JS samples)：小数据量，无调用栈 vs 深录制(完整调用栈)：大数据量，完整信息，建议：快速定位帧级别问题用浅录制；定位具体函数用深录制
  - **Heap Snapshot vs Allocation Timeline**：Heap Snapshot：当前时刻快照，适合泄漏对比 vs Allocation Timeline：时间序列，适合分配热点，建议：泄漏排查用Snapshot对比；优化分配用Timeline
  - **Lighthouse vs 手动Performance录制**：Lighthouse：自动化，标准加载场景 vs 手动Performance：任意交互场景，可模拟真实操作，建议：CI/CD用Lighthouse做回归；性能调优用Performance深入分析

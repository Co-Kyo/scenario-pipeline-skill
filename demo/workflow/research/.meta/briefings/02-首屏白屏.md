## 涉及能力摘要

### A1-浏览器渲染管线
机制：浏览器将HTML/CSS字节流经由DOM和CSSOM构建渲染树，再依次执行Layout（计算几何信息）、Paint（光栅化像素）、Composite（GPU合成图层）输出到屏幕。CSS是渲染阻塞资源，JS（无async/defer）是解析阻塞资源。现代浏览器通过分层合成将transform/opacity动画卸载到GPU执行，避免主线程Layout和Paint。
瓶颈：
  - **CSS渲染阻塞**：外部CSS未加载完成 → 白屏、FCP延迟
  - **JavaScript阻塞解析**：script标签无async/defer → DOM构建延迟、首屏渲染推迟
  - **布局抖动Layout Thrashing**：JS交替读写布局属性 → 帧率骤降、动画卡顿
  - **过多合成层**：大量will-change/transform/opacity → GPU内存暴涨
  - **强制同步布局**：修改样式后立即读取布局属性 → 额外布局计算开销

### A6-HTTP缓存策略
机制：HTTP缓存分为强缓存（Cache-Control max-age/no-cache/no-store）和协商缓存（ETag/If-None-Match、Last-Modified/If-Modified-Since）两层。强缓存命中时直接使用本地副本不发请求；协商缓存通过条件请求验证资源是否更新，未变化返回304。关键指令包括private/public控制缓存范围、immutable标记不变资源、stale-while-revalidate实现异步验证。启发式缓存是无Cache-Control时的兜底机制。
瓶颈：
  - HTML入口文件缓存策略难以平衡更新及时性与性能
  - no-cache语义常被误解为不缓存
  - CDN的s-maxage与浏览器max-age独立管理导致不同步
  - Vary头导致缓存碎片化
  - 个性化内容错误标记public导致数据泄露
权衡：
  - 缓存时长 vs 更新及时性
  - ETag精度 vs 服务器CPU开销
  - stale-while-revalidate用户体验 vs 数据一致性

### A7-CDN与资源分发
机制：CDN通过全球边缘节点缓存源站内容，使用户就近获取资源减少延迟。工作流程：DNS智能解析→边缘节点响应（hit直接返回，miss回源）。HTTP/2时代应回归单域名（多域名增加DNS开销），替代HTTP/1.1的domain sharding。CDN作为共享缓存遵循HTTP缓存语义，可通过s-maxage、CDN-Cache-Control独立控制。Push CDN主动推送适合热点内容，Pull CDN按需拉取更常见。
瓶颈：
  - 多级缓存更新延迟导致新旧版本不一致
  - 动态内容无法缓存需区分路由策略
  - 缓存同时过期导致回源风暴
  - HTTPS证书在多边缘节点的管理复杂度
  - 全球节点缓存刷新不同步
权衡：
  - 缓存时长 vs 一致性（静态资源长缓存+hash vs HTML短缓存）
  - 全球覆盖节点数 vs CDN费用
  - 安全防护(WAF/DDoS) vs 额外故障点

### A8-资源预加载（preload/prefetch）
机制：preload以高优先级提前获取当前页面关键资源(CSS/JS/字体/图片)，不执行只缓存，后续引用时直接复用。prefetch以最低优先级在空闲时获取未来页面资源，跨导航持久化缓存。modulepreload专为ES模块设计，提前获取模块及静态依赖链，消除瀑布式加载。三者通过<link>标签或HTTP Link头声明，是CRP优化的核心工具。
瓶颈：
  - **preload资源未被使用**：声明preload但页面中无对应使用标签 → 浪费带宽、Console警告
  - **preload CORS模式不匹配**：preload跨域字体未加crossorigin → 同一资源请求两次
  - **prefetch与当前页面竞争带宽**：页面加载初期声明大量prefetch → 当前页面资源加载变慢
  - **modulepreload依赖链未覆盖**：只preload入口模块未覆盖依赖 → 依赖仍瀑布式加载
  - **过多预加载请求**：页面声明大量preload/prefetch → 请求队列拥塞

### A9-Critical Rendering Path
机制：CRP是浏览器将HTML字节转换为屏幕像素的完整路径：Bytes→Characters→Tokens→DOM Tree，同时CSS→CSSOM Tree，两者合并为Render Tree（只含可见节点），再经Layout计算几何信息、Paint光栅化像素、Composite合成输出。CSS是渲染阻塞资源（必须全部解析完），JS是解析阻塞资源（无async/defer时暂停HTML解析）。关键优化目标：减少关键资源数、缩短关键路径长度、优化加载顺序。
瓶颈：
  - **CSS阻塞渲染**：外部CSS文件大或加载慢 → 白屏时间长、FCP延迟
  - **JavaScript阻塞DOM构建**：script无async/defer位于head中 → DOM构建暂停
  - **DOM树过大**：页面包含数千个DOM节点 → Layout和Paint耗时线性增长
  - **Web字体阻塞文本渲染**：字体加载期间FOIT → 空白文字区域
  - **关键资源链过长**：关键资源存在依赖链 → 瀑布式加载总时间累加

### A10-A10
机制：Code Splitting 将代码拆分为多个 chunk 按需加载，减少首屏 JS 体积。三种方式：Entry Points 手动拆分入口、SplitChunksPlugin 自动提取公共依赖、Dynamic Import 运行时懒加载。Webpack 将 import() 作为异步 chunk 边界，配合 cacheGroups 配置实现 vendor/runtime/业务代码分离。核心目标是减少首屏传输量并利用浏览器并行加载能力。

  - 过度拆分导致 HTTP 请求过多，增加网络开销

### A11-A11
机制：Tree Shaking 基于 ESM 静态结构，在构建阶段标记并移除未被引用的导出代码。两个层次：usedExports 标记未使用导出供 Terser 删除，sideEffects 允许跳过整个无副作用文件。CommonJS 无法被 Tree Shaking，Babel 默认转 ESM 为 CJS 会破坏此能力。需配合生产模式和正确的 sideEffects 声明才能生效。
瓶颈：
  - 副作用检测困难，Terser 无法可靠判断动态语言中的函数调用是否有副作用
  - CommonJS 库（如 lodash）无法 Tree Shaking，必须使用 ESM 版本
  - Babel 默认将 ESM 转为 CJS，需配置 modules: false 保留静态结构
  - CSS Tree Shaking 需额外工具（PurgeCSS），Webpack 原生不支持
  - re-export 场景可能导致误删
权衡：
  - ****：sideEffects：跳过整个未引用文件，效率最高 vs usedExports：标记单个导出，粒度更细，建议：
  - ****：启用：开发时即可发现未使用代码 vs 不启用：加快构建速度，生产模式自动启用，建议：

### N1-Next/Nuxt SSR实践
机制：Next.js 和 Nuxt 提供 SSR/SSG/ISR 三种渲染模式。SSR 每次请求动态生成，适合实时数据；SSG 构建时生成，CDN 分发；ISR 结合两者，构建时生成 + 定期重验证。Next.js App Router 通过服务端组件和 fetch 缓存实现组件级数据获取和 ISR。Nuxt 通过 useAsyncData/useFetch 实现 SSR/CSR 通用数据获取。模式选择取决于数据变化频率和性能需求。
瓶颈：
  - SSR 服务端高并发下 Node.js 单线程瓶颈
  - 串行数据获取导致 TTFB waterfall
  - ISR 缓存一致性（stale-while-revalidate）
  - SSG 大量页面构建时间线性增长
  - 无论哪种模式客户端 hydration JS 执行量不变

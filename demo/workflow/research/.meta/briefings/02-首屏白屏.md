# P2-首屏白屏 — 组装 Briefing

## 命题信息
命题：首屏白屏：SSR/SSG/CSR混合渲染的首屏加载优化
限定词：前端框架(React/Vue) + Node.js SSR
通用占比：≥70%（通用层跨网络→协议→浏览器→构建→框架5层）

## 涉及能力摘要

### A1-浏览器渲染管线 [用于: overview, edge-cases, trade-offs]
机制：浏览器渲染管线（CRP）是将 HTML/CSS/JS 转换为屏幕像素的完整流程：DOM 构建 → CSSOM 构建 → 样式计算（Render Tree）→ 布局（Layout）→ 分层（Layer Tree）→ 绘制（Paint）→ 光栅化（Raster）→ 合成（Composite）。Layout 和 Paint 阶段开销最大，仅触发 Composite 的属性（transform/opacity）性能最优。管线中任何阶段都可能因 JS 介入被重新触发。
瓶颈：
  - B1 CSS 渲染阻塞（Style, P0）：<link rel='stylesheet'> 阻塞首屏渲染，关键 CSS 体积大或网络慢 → FCP 延迟，白屏时间长
  - B2 JS 阻塞 DOM 解析（Parsing, P0）：<script> 标签无 async/defer → DOM 构建延迟，TTFB 后长时间无内容渲染
  - B3 Layout Thrashing（Layout, P0）：JS 循环中交替读取布局属性和修改样式 → 帧率骤降，长任务 >50ms
  - B4 强制同步布局（Layout, P1）：读取 getBoundingClientRect()/offsetWidth 后立即修改样式 → 帧率下降
  - B5 大图解码阻塞（Raster, P1）：高分辨率图片在 Raster 阶段解码耗时过长 → 主线程阻塞，页面卡死
权衡：
  - CSS 加载：外部 <link> 全量加载 vs 内联 Critical CSS + 异步加载剩余，首屏敏感场景用后者
  - JS 加载：同步 <script> vs script defer/async，默认使用 defer，独立脚本用 async
  - 布局查询：同步读取 offsetWidth vs ResizeObserver 异步监听，优先异步

### A5-HTTP缓存协议 [用于: overview, edge-cases, trade-offs]
机制：HTTP缓存依据 RFC 9111 分为私有缓存（浏览器）和共享缓存（CDN/代理）。强缓存通过 Cache-Control max-age/Expires 判断新鲜期，命中时直接使用本地副本无需网络请求。协商缓存在强缓存过期后通过 ETag/If-None-Match 或 Last-Modified/If-Modified-Since 条件请求向源站验证，未变化返回 304 Not Modified。关键指令：no-cache（允许缓存但必须验证）、no-store（禁止缓存）、immutable（新鲜期内跳过验证）、stale-while-revalidate（过期后异步验证）。Vary 头控制缓存键维度，s-maxage 分离 CDN 与浏览器缓存策略。
瓶颈：
  - B1 缓存穿透（resource boundary, P0）：未缓存资源频繁回源导致源站过载 → 延迟升高
  - B2 缓存雪崩（timing, P0）：大量缓存同时过期造成瞬时回源洪峰 → 源站超时或宕机
  - B3 缓存一致性（state transition, P1）：cache-busting 缺失导致用户看到过期内容 → 内容陈旧
  - B4 Vary 过度膨胀（input variation, P1）：Vary:User-Agent 导致缓存命中率骤降 → 首屏加载变慢
  - B5 启发式缓存失控（input variation, P2）：缺少显式 Cache-Control 导致意外长期缓存 → 内容无法更新
权衡：
  - 静态资源：长缓存+cache-busting（高命中率）vs 短缓存（即时更新）— 选前者
  - HTML 入口：长缓存（可能看不到更新）vs no-cache（每次验证）— 选后者
  - ETag（精度高但计算开销）vs Last-Modified（简单但秒级精度）— 优先 ETag
  - stale-while-revalidate（异步验证用户体验好）vs 同步验证（保证数据新鲜）— 高频读选前者

### A7-资源加载策略 [用于: overview, edge-cases, trade-offs]
机制：通过浏览器原生 Resource Hints（preload/prefetch/preconnect/dns-prefetch）和现代加载属性（loading='lazy'/fetchpriority）实现资源优先级调度。preload 用于当前导航关键资源的高优先级预加载（需指定 as 属性），prefetch 用于未来导航资源的低优先级预获取，preconnect 提前完成 DNS+TCP+TLS 握手，lazy loading 延迟非首屏资源加载，fetchpriority 在同类资源内微调优先级。
瓶颈：
  - B1 preload 未被使用（input variation, P1）：关键资源未 preload 导致带宽浪费和 Console 警告
  - B2 preload CORS 模式不匹配（input variation, P1）：preload 与实际请求的 CORS 模式不匹配导致双重下载
  - B3 过多 prefetch 争抢带宽（resource boundary, P1）：过多 prefetch 与首屏关键资源争抢带宽导致 LCP 退化
  - B4 首屏 LCP 图片用 lazy（input variation, P0）：首屏 LCP 图片错误使用 loading='lazy' 导致 LCP 延迟
  - B5 preconnect 过多（resource boundary, P2）：preconnect 超过 6 个域名浪费 DNS/TCP 资源
权衡：
  - preload 当前关键资源 vs prefetch 未来页面资源：首屏优先用 preload，交互后用 prefetch，preload 控制在 5 个以内
  - 原生 lazy loading vs Intersection Observer JS 懒加载：优先用原生零成本方案
  - preconnect vs dns-prefetch：关键第三方域名用 preconnect，其余用 dns-prefetch

### A9-模块系统与构建优化 [用于: overview, edge-cases, trade-offs]
机制：ES Module 的静态 import/export 结构使构建工具能在编译期分析完整依赖图，这是 Tree Shaking 的前提。动态 import() 返回 Promise 实现按需加载，构建工具将其识别为 split point 自动生成独立 chunk。Code Splitting 有三种策略：Entry Points 手动分割、SplitChunksPlugin 自动提取公共依赖、Dynamic Import 按需加载。Vite 利用浏览器原生 ESM+esbuild 预构建实现极快启动，生产模式基于 Rollup 打包。
瓶颈：
  - B1 过度分割（resource boundary, P0）：minSize 过小导致 chunk 数量爆炸，HTTP 请求激增，首屏加载反而变慢
  - B2 Tree Shaking 失效（input variation, P1）：CommonJS 模块混入、模块顶层副作用、require() 动态导入导致未消除死代码
  - B3 动态 import 加载抖动（input variation, P1）：网络不稳定导致 chunk 加载失败和白屏
  - B4 重复依赖（resource boundary, P2）：多 chunk 引用同依赖不同版本，同一库被多次打包
权衡：
  - 分割粒度：粗粒度（少量大 chunk）vs 细粒度（大量小 chunk），推荐路由级中等粒度
  - 首屏 vs 按需：全量打包 initial chunk vs 路由/功能动态 import，首屏关键路径保持 initial
  - Tree Shaking vs 兼容性：纯 ESM 依赖（最佳 Tree Shaking）vs CJS 依赖（兼容性好），优先选 ESM 版本
  - 缓存策略：contenthash 长期缓存 vs 版本号短期缓存，生产环境必须用 contenthash

### A10-CDN与边缘计算 [用于: overview, edge-cases, trade-offs]
机制：CDN 通过全球分布的边缘节点缓存静态资源，基于 DNS 智能解析将用户请求导向最近节点，减少延迟和源站负载。核心机制：多级缓存架构（浏览器→边缘→汇聚层→源站）、基于 URL/Header 的 Cache Key 生成、TTL/版本化 URL/Purge 等缓存失效策略、Origin Shield 保护源站免受回源风暴。边缘计算将计算能力下沉到 CDN 节点。
瓶颈：
  - B1 缓存命中率低（resource boundary, P0）：动态内容过多、Cache Key 过细、TTL 设置不合理 → 回源率 >20%，延迟升高
  - B2 回源风暴（timing, P0）：热点内容同时过期、大促秒杀场景 → 源站超时或宕机
  - B3 缓存一致性延迟（state transition, P1）：内容更新后缓存未及时失效 → 用户看到旧版本内容
  - B4 CDN 故障/单点依赖（resource boundary, P1）：CDN 服务商宕机或区域网络故障 → 网站不可访问
权衡：
  - 缓存粒度：粗粒度（仅 URL）命中率高 vs 细粒度（URL+Header/Cookie）个性化，静态资源用粗粒度
  - 缓存 TTL：长 TTL（7天+）命中率高 vs 短 TTL（分钟级）实时性好，稳定资源用长 TTL+版本化 URL
  - 部署架构：单 CDN 管理简单 vs Multi-CDN 高可用，中小项目单 CDN+自托管回退
  - 边缘计算：纯缓存 CDN 性能最优 vs Edge Computing 灵活强大，简单加速用纯缓存

### A13-SSR/Hydration机制 [用于: overview, edge-cases, trade-offs]
机制：SSR 在服务端将组件树渲染为 HTML 字符串发送给浏览器，客户端通过 Hydration 将静态 HTML 激活为可交互应用。核心流程：服务端 renderToString/renderToReadableStream → HTML 传输 → 客户端 hydrateRoot/app.mount 绑定事件与状态。React 18 引入 Streaming SSR（Suspense 分段流式传输+Selective Hydration），Vue 3.5 引入 Lazy Hydration（onInteraction/onVisible/whenIdle）。SSG 为 SSR 的构建时变体，输出静态 HTML。关键约束：客户端首次渲染必须与服务端 HTML 完全一致，否则触发 hydration mismatch。
瓶颈：
  - B1 Hydration Mismatch（state transition, P0）：客户端与服务端 HTML 不一致导致事件错位 → 交互异常
  - B2 JS Bundle 过大（resource boundary, P0）：Hydration 需加载完整客户端 JS → FCP 快但 TTI 慢，交互延迟
  - B3 服务端 CPU 瓶颈（resource boundary, P1）：高并发下单线程 Node.js 渲染阻塞 → 响应延迟
  - B4 数据获取瓶颈（timing, P1）：SSR 预取数据受数据库/API 延迟制约 → TTFB 高
  - B5 跨请求状态污染（state transition, P1）：单例 store 在请求间共享状态 → 数据串扰
  - B6 浏览器 API 不可用（input variation, P1）：服务端引用 window/document 报错 → 构建失败
权衡：
  - SSR vs SSG：动态内容需 SSR，静态内容用 SSG 更高效
  - 流式 vs 同步渲染：延迟高选 Streaming，SEO 严格选同步+allReady
  - 全量 vs Selective Hydration：首屏交互要求高选全量，长页面选 Selective
  - Node.js vs 边缘运行时：需数据库选 Node.js，延迟敏感选边缘
  - 纯 SSR vs Islands 架构：交互密集选 Islands，整体性强选纯 SSR

## 内容比例约束
开篇 10-15%：从 SSR/首屏白屏痛点切入（FCP 白屏、TTI 交互延迟、Hydration 问题）
主体 70-80%：通用工程原理（渲染管线 CRP、HTTP 缓存协议、资源加载优先级、构建优化、CDN 分发）
收尾 10-15%：回框架落地方案（Next.js/Nuxt SSR 配置、Streaming SSR、Selective Hydration 实践）

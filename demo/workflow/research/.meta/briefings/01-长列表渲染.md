## 涉及能力摘要

### A1-浏览器渲染管线
机制：浏览器将HTML/CSS字节流经由DOM和CSSOM构建渲染树，再依次执行Layout（计算几何信息）、Paint（光栅化像素）、Composite（GPU合成图层）输出到屏幕。CSS是渲染阻塞资源，JS（无async/defer）是解析阻塞资源。现代浏览器通过分层合成将transform/opacity动画卸载到GPU执行，避免主线程Layout和Paint。
瓶颈：
  - **CSS渲染阻塞**：外部CSS未加载完成 → 白屏、FCP延迟
  - **JavaScript阻塞解析**：script标签无async/defer → DOM构建延迟、首屏渲染推迟
  - **布局抖动Layout Thrashing**：JS交替读写布局属性 → 帧率骤降、动画卡顿
  - **过多合成层**：大量will-change/transform/opacity → GPU内存暴涨
  - **强制同步布局**：修改样式后立即读取布局属性 → 额外布局计算开销

### A2-DOM节点生命周期
机制：DOM节点经历创建(createElement/cloneNode/解析器)→插入(appendChild/Fragment)→更新(textContent/style/attribute)→销毁(remove/removeChild)四阶段。插入文档树后触发Layout和Paint，批量操作可用DocumentFragment合并。节点移除后若JS仍持有引用则无法GC，导致Detached DOM内存泄漏。自定义元素提供connectedCallback/disconnectedCallback等生命周期钩子。
瓶颈：
  - **频繁DOM操作导致多次Layout**：循环中逐个插入/修改节点 → 创建列表时卡顿
  - **事件监听器泄漏**：组件销毁时未清理事件监听器 → SPA切页后内存持续增长
  - **innerHTML导致子节点全部重建**：使用innerHTML更新只改一个字 → 丢失子节点状态、大范围Reflow
  - **Detached DOM节点内存泄漏**：节点从文档移除但JS仍引用 → 内存占用持续增长

### A3-重绘与回流
机制：回流(Reflow/Layout)重新计算元素几何属性，重绘(Repaint)重新绘制像素。修改几何属性触发回流+重绘，修改视觉属性只触发重绘，修改transform/opacity只触发合成(Composite)。浏览器采用异步回流优化，但读取offsetWidth/getComputedStyle等布局属性会强制同步回流。布局抖动(Layout Thrashing)指JS交替读写布局属性导致多次强制回流，是常见性能杀手。
瓶颈：
  - **布局抖动Layout Thrashing**：JS循环中交替读取布局属性和修改样式 → 帧率骤降到个位数
  - **动画使用几何属性**：用left/top/width/height做动画 → 每帧触发Reflow+Repaint
  - **频繁插入DOM节点**：循环中逐个appendChild → 每次插入触发Reflow
  - **CSS选择器复杂度过高**：深层嵌套选择器 → Style Recalculation耗时增加
  - **getComputedStyle强制同步回流**：修改样式后立即调用getComputedStyle/getBoundingClientRect → 同步回流开销

### A4-虚拟化算法（可视区域计算）
机制：虚拟列表通过仅渲染可视区域内的DOM节点优化大数据量列表性能。核心包括：可视区域计算（根据scrollTop和itemHeight定位startIndex/endIndex）、缓冲区overscan（预渲染额外节点防止白屏）、DOM复用池（回收滚出可视区的节点复用）、容器高度占位（维持正确滚动条）。动态高度场景需维护heightMap，使用预估高度+渐进修正。
瓶颈：
  - **动态高度计算精度与性能**：列表项高度不固定，需实时测量 → 滚动时跳动闪烁，滚动条位置不准确
  - **快速滚动白屏**：用户快速拖拽滚动条，overscan不足 → 可见区域出现短暂空白
  - **DOM复用池内存泄漏**：复用池无限增长或节点重置不彻底 → 内存持续增长，复用节点残留旧数据
  - **滚动事件节流与渲染同步**：高频scroll事件触发重计算和DOM更新 → 滚动卡顿，帧率下降
  - **列表项内嵌套复杂组件**：列表项包含图片、视频、复杂组件 → 单个列表项渲染仍耗时过长
权衡：
  - **高度策略**：固定高度：O(1)定位，完美滚动精度 vs 动态高度：O(log n)二分查找，可能跳动，建议：优先选择固定高度；动态高度仅在业务必须时使用
  - **overscan大小**：小overscan(1-2)：DOM少，内存低，白屏风险高 vs 大overscan(5-10)：DOM多，内存高，白屏风险低，建议：根据列表项渲染复杂度和滚动速度动态调整
  - **DOM复用策略**：DOM复用：低GC压力，高代码复杂度 vs 每次创建销毁：低复杂度，高GC压力，建议：列表项渲染成本高时使用复用；简单列表可不使用

### A5-IntersectionObserver
机制：IntersectionObserver是W3C标准的异步可见性观察接口，通过注册回调在目标元素与根元素/视口交叉状态变化时触发通知。配置项包括root（根元素）、rootMargin（边距扩展）、threshold（交叉比例阈值）、trackVisibility（真实可见性追踪）。相比scroll事件，观察和计算在浏览器后台异步执行，不阻塞主线程，一个observer可观察多个目标。
瓶颈：
  - **回调在主线程执行**：回调函数中执行耗时操作 → 即使观察异步，回调仍可能阻塞主线程
  - **大量目标元素观察开销**：同时observe数千个元素 → 内存占用增加，回调批次过大
  - **threshold过细导致回调风暴**：设置大量threshold，元素在边界快速移动 → 短时间内大量回调，帧率下降
  - **嵌套滚动容器root选择**：目标元素在多层嵌套滚动容器中 → 选择错误root导致检测不准确
权衡：
  - **rootMargin预加载量**：小rootMargin(50px)：低网络开销，可能有加载闪烁 vs 大rootMargin(500px)：更流畅，可能加载不需要的内容，建议：图片懒加载用200-300px；无限滚动用1-2屏高度
  - **精确追踪vs性能**：trackVisibility=false：低开销，仅几何交叉 vs trackVisibility=true：高开销，检测遮挡/透明度，建议：大多数场景用false；广告可见性追踪用true
  - **单Observer vs 多Observer**：单Observer多目标：低内存，共享配置 vs 多Observer各自目标：高内存，独立配置，建议：同类目标用单observer；不同类型需求用多observer

### A27-CSS contain与content-visibility
机制：CSS contain告知浏览器元素渲染独立（size/layout/style/paint四种隔离），内部变化不触发外部reflow/repaint。content-visibility: auto在此基础上自动判断屏幕外元素跳过渲染，接近viewport时恢复。需配合contain-intrinsic-size提供占位尺寸。对长页面首屏加载性能提升显著，可减少50%+的渲染时间。
瓶颈：
  - contain:size未显式设宽高时元素坍缩为0
  - content-visibility:hidden的内容对Ctrl+F不可搜索
  - contain-intrinsic-size不准确导致滚动条跳动
  - 嵌套containment过深增加浏览器管理开销
  - contain属性不可动画化，content-visibility需transition-behavior:allow-discrete
权衡：
  - 渲染性能 vs 布局准确性：containment跳过子元素计算可能影响依赖子元素尺寸的父元素布局
  - content-visibility:auto vs 手动虚拟滚动：auto更简单无侵入，但对超长列表不如虚拟列表精细
  - containment粒度：strict最激进需显式尺寸，content更实用但收益略低

### A30-requestAnimationFrame调度
机制：requestAnimationFrame(rAF)在浏览器下一次重绘前调用回调，频率匹配显示器刷新率(60Hz=16.67ms)。回调在Style/Layout/Paint之前执行，修改DOM不会产生额外重排。rAF自动暂停于后台标签页。回调接收DOMHighResTimeStamp参数，应基于时间差计算动画进度而非帧计数。实际可用JS预算约10-12ms。
瓶颈：
  - **rAF回调中执行耗时操作**：rAF回调执行大量计算或DOM操作超过帧预算 → 掉帧，动画卡顿
  - **rAF嵌套导致帧预算超支**：在rAF中同步修改DOM触发Layout后再用rAF → 单帧内多次Layout(Layout thrashing)
  - **忘记取消rAF导致资源浪费**：组件卸载时未调用cancelAnimationFrame → 已卸载组件的回调仍在执行
  - **高刷新率屏幕性能退化**：应用在60Hz流畅但在120Hz/144Hz掉帧 → 高刷屏反而更卡
权衡：
  - **rAF vs setInterval/setTimeout**：rAF：帧同步，后台自动暂停，高精度 vs setTimeout：独立于渲染，后台继续执行，低精度，建议：所有视觉更新用rAF；定时轮询用setTimeout
  - **帧内切片 vs requestIdleCallback延后**：帧内切片：持续产出结果，可能降低帧率 vs requestIdleCallback：利用空闲时间，不影响帧率，建议：进度条/动画用切片；日志上报/预取用idle
  - **时间驱动 vs 帧驱动动画**：时间驱动(基于elapsed)：跨刷新率一致 vs 帧驱动(基于帧计数)：120Hz速度翻倍，建议：始终使用时间驱动，用timestamp参数

### V1-Vue patch flag优化
机制：Vue 3 通过编译时分析模板，为 VNode 打上 PatchFlag 标记动态内容类型（TEXT/CLASS/STYLE/PROPS 等），配合 Block Tree 收集动态节点。更新时只遍历 dynamicChildren 做靶向更新，跳过所有静态节点。额外优化包括静态提升、预字符串化和函数缓存。将 diff 性能从与模板大小正相关提升为与动态内容数量相关。

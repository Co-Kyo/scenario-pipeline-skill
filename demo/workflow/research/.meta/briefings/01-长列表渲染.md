# P1 长列表渲染 — 组装 Briefing

## 命题信息
命题：长列表渲染：万级数据的虚拟列表实现——社交动态流/聊天记录场景
通用占比：≥ 80%
限定词：无

## 涉及能力摘要

### A1-浏览器渲染管线 [用于: overview, edge-cases, trade-offs]
机制：CRP：HTML→DOM Tree + CSS→CSSOM → Render Tree → Layout → Paint → Composite。transform/opacity 只触发 Composite 最高效，width/height 触发 Layout 最昂贵。
瓶颈：
- B1 强制同步布局（时序竞争，P0）：JS 交替读写布局属性 → 帧率骤降至 15fps
- B2 过深 DOM 嵌套（规模拐点，P1）：DOM 超 1500 节点 → Layout 线性增长
权衡：动画属性选 transform/opacity（Composite only）vs width/height（Layout+Paint），优先用 transform

### A2-事件循环机制 [用于: overview, edge-cases]
机制：调用栈清空后先清微任务（Promise.then），再 rAF，再渲染，最后宏任务。微任务可独占主线程。
瓶颈：
- B1 微任务风暴（时序竞争，P0）：Promise 链过长 → 主线程被独占页面卡死
- B2 宏任务饥饿（规模拐点，P1）：每个宏任务 >16ms → 帧率 <60fps
权衡：同步批量执行（简单但阻塞）vs 切片执行（rAF/rIC，复杂度高）

### A5-虚拟列表算法 [用于: overview, edge-cases, trade-offs]
机制：只渲染可视区域 DOM，scrollTop→起始索引映射+缓冲区。动态高度需前缀和+二分查找。DOM 池复用。
瓶颈：
- B1 动态高度滚动跳动（状态跃迁，P0）：项高度渲染后才确定 → 滚动位置偏移
- B2 快速滚动白屏（资源边界，P1）：缓冲区太小 → 短暂空白
权衡：固定高度（O(1)定位）vs 动态高度（灵活但需前缀和+二分）

### A9-IntersectionObserver 与懒加载 [用于: overview, edge-cases]
机制：IO 异步观察元素与视口交叉状态，rootMargin 控制提前加载距离。不阻塞主线程。
瓶颈：
- B1 懒加载图片闪烁（状态跃迁，P0）：无占位 → CLS 增加
- B2 观察器未清理（资源边界，P1）：未 disconnect → 内存泄漏
权衡：IO（异步性能好）vs scroll+getBoundingClientRect（同步兼容IE）

### A13-CSS contain 与合成层 [用于: overview, edge-cases, trade-offs]
机制：contain 隔离子树允许浏览器跳过不必要布局/绘制。content-visibility:auto 跳过视口外元素。will-change 提升合成层 GPU 加速。
瓶颈：
- B1 层爆炸（资源边界，P1）：大量 will-change → GPU 内存暴涨
- B2 contain 尺寸异常（输入变异，P1）：contain:size 未设尺寸 → 高度塌陷
权衡：contain:content（轻量）vs contain:strict（完全隔离需手动设尺寸）

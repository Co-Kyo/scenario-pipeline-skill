# P3 首屏优化 — 组装 Briefing

## 命题信息
命题：首屏优化：SSR/SSG 首屏加载慢——Vue3/React 项目如何提升到 1s 内
通用占比：60-80%
限定词：Vue3/React

## 涉及能力摘要

### A1-浏览器渲染管线 [用于: overview, edge-cases]
机制：CRP 全链路：DOM+CSSOM→Render Tree→Layout→Paint→Composite。CSS 是渲染阻塞的，必须全部解析完才能构建 Render Tree。
瓶颈：
- B1 强制同步布局（时序竞争，P0）：读写布局属性交替 → 帧率骤降
权衡：transform/opacity 只触发 Composite vs width/height 触发 Layout

### A6-HTTP 缓存策略 [用于: overview, edge-cases, trade-offs]
机制：强缓存（Cache-Control max-age，不发请求）和协商缓存（ETag/If-None-Match，304）。CDN 在浏览器和源站间加缓存层。
瓶颈：
- B1 缓存失效全量回源（状态跃迁，P0）：部署时 hash/Cache-Control 配错 → 用户看旧版
- B2 CDN 与浏览器缓存不一致（状态跃迁，P1）：CDN 更新但浏览器强缓存未过期
权衡：长期缓存+hash 文件名（最优性能）vs no-cache 协商缓存（简单但每次请求）

### A7-Code Splitting 与 Tree Shaking [用于: overview, trade-offs]
机制：动态 import() 生成独立 chunk，路由级按需加载。Tree Shaking 基于 ESM 静态分析删除未使用代码。
瓶颈：
- B1 Tree Shaking 失效（输入变异，P0）：CommonJS 或 sideEffects 未配置 → 未使用代码仍在
- B2 vendor chunk 膨胀（规模拐点，P1）：大库污染 vendor → vendor.js >500KB
权衡：路由级分割（简单）vs 组件级分割（精细但复杂）

### A8-Core Web Vitals 指标体系 [用于: overview, edge-cases]
机制：LCP（加载 ≤2.5s）、INP（交互 ≤200ms）、CLS（布局偏移 ≤0.1）。PerformanceObserver 采集。
瓶颈：
- B1 LCP 元素加载慢（资源边界，P0）：首屏最大元素渲染延迟 → LCP >4s
- B2 CLS 布局偏移（状态跃迁，P1）：图片/字体加载导致元素位移
权衡：LCP 优化用 preload（资源型）vs SSR 直出（文本型）

### A11-SSR/Hydration 原理 [用于: overview, edge-cases, trade-offs]
机制：服务端渲染 HTML→浏览器显示→加载 JS→Hydration 绑定事件→可交互。流式渲染减少 TTFB。
瓶颈：
- B1 Hydration 不匹配（输入变异，P0）：SSR/CSR 输出不一致 → 全量重渲染
- B2 Hydration 阻塞交互（时序竞争，P0）：首屏 JS 过大 → FCP 快但 TTI 慢
权衡：SSR（首屏快，服务端开销大）vs SSG（构建时生成，无服务端开销）

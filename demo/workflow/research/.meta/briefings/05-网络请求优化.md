# P5 网络请求优化 — 组装 Briefing

## 命题信息
命题：网络请求优化：接口串行导致首屏阻塞——瀑布流请求与并发控制
通用占比：≥ 80%
限定词：无

## 涉及能力摘要

### A1-浏览器渲染管线 [用于: overview]
机制：CRP 全链路。CSS 是渲染阻塞的，JS 是解析阻塞的。
瓶颈：
- B1 强制同步布局（时序竞争，P0）：JS 阻塞解析 → 首屏延迟

### A6-HTTP 缓存策略 [用于: overview, edge-cases, trade-offs]
机制：强缓存 + 协商缓存。CDN 缓存协作。
瓶颈：
- B1 缓存失效全量回源（状态跃迁，P0）
- B2 协商缓存回源风暴（规模拐点，P1）：大量请求同时回源验证 → 源站 QPS 飙高
权衡：长期缓存+hash vs no-cache

### A8-Core Web Vitals 指标体系 [用于: overview, edge-cases]
机制：LCP/INP/CLS 三大指标。
瓶颈：
- B1 LCP 元素加载慢（资源边界，P0）：资源被串行请求阻塞

### A9-IntersectionObserver 与懒加载 [用于: edge-cases]
机制：IO 异步观察元素可见性。
瓶颈：
- B1 懒加载图片闪烁（状态跃迁，P0）：无占位 → CLS 增加

### A15-请求并发与数据预取 [用于: overview, edge-cases, trade-offs]
机制：Promise.all 并发，BFF 聚合，数据预取。并发限制避免 6 连接限制。
瓶颈：
- B1 请求瀑布流（时序竞争，P0）：多接口串行 → 首屏=所有请求时间之和
- B2 请求竞态（时序竞争，P1）：快速切换页面 → 旧请求覆盖新数据
权衡：客户端并发 vs BFF 聚合；React Query/SWR vs 手动管理

## 内容比例约束
开篇 10-15%：从接口串行导致首屏 3-5s 的痛点切入
主体 70-80%：HTTP 缓存、请求并发、BFF 聚合、数据预取等通用原理
收尾 10-15%：回到 React Query/SWR + 接口聚合给落地方案

# P7-渲染性能 — 组装 Briefing

## 命题信息
命题：渲染性能：重排重绘触发机制与合成层优化
限定词：无（纯通用层）
通用占比：100%（浏览器层+CSS层+工具层跨4层）

## 涉及能力摘要

### A1-浏览器渲染管线 [用于: overview, edge-cases, trade-offs]
机制：浏览器渲染管线（CRP）是将 HTML/CSS/JS 转换为屏幕像素的完整流程：DOM 构建 → CSSOM 构建 → 样式计算（Render Tree）→ 布局（Layout）→ 分层（Layer Tree）→ 绘制（Paint）→ 光栅化（Raster）→ 合成（Composite）。Layout 和 Paint 阶段开销最大，仅触发 Composite 的属性（transform/opacity）性能最优。管线中任何阶段都可能因 JS 介入被重新触发，形成强制同步布局或布局抖动。
瓶颈：
  - B1 Layout Thrashing（时序竞争，P0）：JS 循环中交替读取布局属性和修改样式 → 帧率骤降，长任务 >50ms
  - B2 Paint 开销过大（输入变异，P1）：复杂 CSS 效果（box-shadow、filter、大图 background）→ 掉帧，Paint 时间长
  - B3 图层爆炸（资源边界，P1）：过度使用 will-change 或浏览器隐式创建过多图层 → 内存暴涨，低端设备卡顿
  - B4 强制同步布局（时序竞争，P1）：读取 getBoundingClientRect()/offsetWidth 后立即修改样式 → 帧率下降
权衡：
  - 动画实现：top/left（触发 Layout+Paint+Composite）vs transform: translate()（仅触发 Composite），优先 transform
  - 图层管理：浏览器自动分层 vs will-change 手动分层，仅对高优先级动画元素手动分层
  - 布局查询：同步读取 offsetWidth vs ResizeObserver 异步监听，优先异步

### A2-DOM节点生命周期 [用于: overview, edge-cases, trade-offs]
机制：DOM 节点从创建到回收的完整生命周期：创建（createElement/createTextNode）→ 挂载（appendChild/insertBefore）→ 更新（属性/子树变更）→ 卸载（removeChild/remove）→ 垃圾回收。核心风险是 Detached DOM 内存泄漏——节点从文档树移除但 JS 仍持有引用导致无法 GC。
瓶颈：
  - B1 Detached DOM 内存泄漏（状态跃迁，P0）：DOM 节点被移除后 JS 仍持有引用 → 页面内存持续增长，长时间运行后卡顿
  - B2 事件监听器泄漏（状态跃迁，P0）：addEventListener 后未 removeEventListener，SPA 路由切换时尤为突出 → 事件回调重复执行
  - B3 批量 DOM 操作触发重排（时序竞争，P1）：循环中逐个 appendChild，每次触发 Layout → 界面闪烁，FPS 下降
权衡：
  - 批量插入：DocumentFragment（保留已有节点和事件监听器）vs innerHTML（更快但销毁已有子树），有交互节点用 Fragment
  - 事件绑定：直接 addEventListener 到每个子节点 vs 事件委托绑定到父节点，动态列表用事件委托
  - 更新策略：直接操作 DOM（原生 API）vs 虚拟 DOM Diff（React/Vue），复杂 UI 状态用框架声明式更新

### A4-事件循环与任务调度 [用于: overview, edge-cases, trade-offs]
机制：事件循环单线程异步模型，宏任务/微任务队列执行优先级，requestAnimationFrame 渲染同步调度，requestIdleCallback 空闲期调度，Long Tasks API（50ms 阈值）长任务检测。
瓶颈：
  - B1 主线程长任务阻塞（时序竞争，P0）：大量同步 JS 执行、复杂计算未分片 → 页面卡顿、点击无响应、INP 劣化
  - B2 rAF 回调过重（时序竞争，P1）：rAF 回调中执行复杂计算 → 掉帧，动画不流畅
  - rIC 在 Safari 不支持/高负载下不触发（输入变异，P2）
权衡：
  - 任务拆分：requestIdleCallback（低优先级，Safari 不支持）vs scheduler.yield()（新 API，切分到下一个宏任务）
  - 动画驱动：setTimeout（不精确）vs requestAnimationFrame（与渲染同步），动画必须用 rAF

### A6-CSS布局与合成层 [用于: overview, edge-cases, trade-offs, experiment]
机制：CSS contain 属性通过 size/layout/style/paint 四种隔离模式，将元素子树与外部文档解耦，使浏览器可跳过不必要的 Layout 和 Paint 重算。content-visibility:auto 在 contain 基础上自动跳过屏幕外元素渲染，长页面首屏可缩短 50%-90%。will-change 向浏览器声明即将变化的属性，触发合成层提升，使动画仅走 Composite 阶段（GPU 合成，60fps）。
瓶颈：
  - B1 合成层爆炸（资源边界，P0）：大量元素被提升为合成层（滥用 will-change 或 translateZ(0)）→ 内存暴涨、GPU 显存不足
  - B2 Paint Storm（时序竞争，P1）：大面积 repaint（background-color、box-shadow 变化）→ 滚动或动画时掉帧
  - B3 content-visibility 滚动跳动（输入变异，P1）：未设置 contain-intrinsic-size → 滚动条突然跳动
  - B4 will-change 内存泄漏（状态跃迁，P1）：样式表中静态设置 will-change 且不移除 → 合成层常驻内存
  - B5 contain 导致布局错乱（输入变异，P2）：对 position:fixed 或 counter 元素添加 contain → fixed 定位异常、counter 被重置
权衡：
  - 动画属性：top/left（触发 Layout+Paint）vs transform:translate（仅 Composite），性能差距 10 倍以上
  - 合成层策略：静态 will-change 全局应用 vs 动态 JS 管理（动画前后切换），避免常驻内存
  - contain 粒度：根元素 contain:strict vs 组件级 contain:content，组件级隔离副作用小

### A8-DevTools性能分析 [用于: overview, edge-cases, experiment]
机制：Performance 面板录制分析（火焰图/Long Task/帧率）、Memory 面板堆快照（Heap Snapshot/Allocation timeline）、Lighthouse 自动化审计（FCP/LCP/TBT/CLS）、PerformanceObserver API 异步观察性能条目、Long Tasks API 主线程阻塞检测（>50ms+attribution 归属）。
瓶颈：
  - B1 JavaScript 主线程阻塞（时序竞争，P0）：大量同步 JS 执行 → 页面卡顿、LCP/INP 差
  - B2 内存泄漏（状态跃迁，P0）：事件监听器未移除、闭包持有引用 → Heap 持续增长
  - B3 布局抖动 Layout Thrashing（时序竞争，P0）：JS 中交替读写 layout 属性 → 帧率骤降
权衡：
  - 性能监控：PerformanceObserver API（实时精确）vs web-vitals 库（封装好 2KB），生产用 web-vitals
  - 内存分析：Heap Snapshot（完整精准但文件大）vs Allocation Sampling（采样开销低）
  - Long Task 监控：PerformanceObserver+longtask（精确有 attribution）vs rAF+时间差检测（兼容性好）

## 内容比例约束
开篇 10-15%：从真实场景切入——动画卡顿、滚动掉帧、页面长时间运行变慢
主体 70-80%：通用工程原理（渲染管线全链路、重排重绘触发机制、合成层原理、事件循环调度、DevTools 诊断方法）
收尾 10-15%：落地方案总结（checklist 形式：哪些属性只触发 Composite、何时用 contain、如何动态管理 will-change）

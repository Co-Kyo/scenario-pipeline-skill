## 涉及能力摘要

### A1-浏览器渲染管线
机制：浏览器将HTML/CSS字节流经由DOM和CSSOM构建渲染树，再依次执行Layout（计算几何信息）、Paint（光栅化像素）、Composite（GPU合成图层）输出到屏幕。CSS是渲染阻塞资源，JS（无async/defer）是解析阻塞资源。现代浏览器通过分层合成将transform/opacity动画卸载到GPU执行，避免主线程Layout和Paint。
瓶颈：
  - **CSS渲染阻塞**：外部CSS未加载完成 → 白屏、FCP延迟
  - **JavaScript阻塞解析**：script标签无async/defer → DOM构建延迟、首屏渲染推迟
  - **布局抖动Layout Thrashing**：JS交替读写布局属性 → 帧率骤降、动画卡顿
  - **过多合成层**：大量will-change/transform/opacity → GPU内存暴涨
  - **强制同步布局**：修改样式后立即读取布局属性 → 额外布局计算开销

### A3-重绘与回流
机制：回流(Reflow/Layout)重新计算元素几何属性，重绘(Repaint)重新绘制像素。修改几何属性触发回流+重绘，修改视觉属性只触发重绘，修改transform/opacity只触发合成(Composite)。浏览器采用异步回流优化，但读取offsetWidth/getComputedStyle等布局属性会强制同步回流。布局抖动(Layout Thrashing)指JS交替读写布局属性导致多次强制回流，是常见性能杀手。
瓶颈：
  - **布局抖动Layout Thrashing**：JS循环中交替读取布局属性和修改样式 → 帧率骤降到个位数
  - **动画使用几何属性**：用left/top/width/height做动画 → 每帧触发Reflow+Repaint
  - **频繁插入DOM节点**：循环中逐个appendChild → 每次插入触发Reflow
  - **CSS选择器复杂度过高**：深层嵌套选择器 → Style Recalculation耗时增加
  - **getComputedStyle强制同步回流**：修改样式后立即调用getComputedStyle/getBoundingClientRect → 同步回流开销

### A17-DevTools Performance/Memory面板
机制：Chrome DevTools的Performance面板通过火焰图展示主线程每帧工作（JS执行、Layout、Paint、Composite），可定位Long Task和掉帧。Memory面板提供Heap Snapshot（堆快照对比检测泄漏）、Allocation Timeline（分配热点）和Allocation Sampling三种模式。Lighthouse自动化审计输出0-100评分，核心指标包括FCP、LCP、TBT、CLS。
瓶颈：
  - **火焰图Long Task定位困难**：页面存在多个Long Task，调用栈复杂 → 火焰图中红色长条，难以定位根因
  - **Heap Snapshot对比噪声**：对比快照时框架/库内部对象干扰 → 难以区分正常分配和泄漏
  - **Lighthouse评分与真实体验不一致**：实验室环境与用户实际网络/设备差异大 → Lighthouse 90+但用户反馈卡顿
  - **Performance录制的性能开销**：录制时间过长或页面过于复杂 → 录制本身导致页面变慢，数据量过大
权衡：
  - **火焰图录制深度**：浅录制(Disable JS samples)：小数据量，无调用栈 vs 深录制(完整调用栈)：大数据量，完整信息，建议：快速定位帧级别问题用浅录制；定位具体函数用深录制
  - **Heap Snapshot vs Allocation Timeline**：Heap Snapshot：当前时刻快照，适合泄漏对比 vs Allocation Timeline：时间序列，适合分配热点，建议：泄漏排查用Snapshot对比；优化分配用Timeline
  - **Lighthouse vs 手动Performance录制**：Lighthouse：自动化，标准加载场景 vs 手动Performance：任意交互场景，可模拟真实操作，建议：CI/CD用Lighthouse做回归；性能调优用Performance深入分析

### A30-requestAnimationFrame调度
机制：requestAnimationFrame(rAF)在浏览器下一次重绘前调用回调，频率匹配显示器刷新率(60Hz=16.67ms)。回调在Style/Layout/Paint之前执行，修改DOM不会产生额外重排。rAF自动暂停于后台标签页。回调接收DOMHighResTimeStamp参数，应基于时间差计算动画进度而非帧计数。实际可用JS预算约10-12ms。
瓶颈：
  - **rAF回调中执行耗时操作**：rAF回调执行大量计算或DOM操作超过帧预算 → 掉帧，动画卡顿
  - **rAF嵌套导致帧预算超支**：在rAF中同步修改DOM触发Layout后再用rAF → 单帧内多次Layout(Layout thrashing)
  - **忘记取消rAF导致资源浪费**：组件卸载时未调用cancelAnimationFrame → 已卸载组件的回调仍在执行
  - **高刷新率屏幕性能退化**：应用在60Hz流畅但在120Hz/144Hz掉帧 → 高刷屏反而更卡
权衡：
  - **rAF vs setInterval/setTimeout**：rAF：帧同步，后台自动暂停，高精度 vs setTimeout：独立于渲染，后台继续执行，低精度，建议：所有视觉更新用rAF；定时轮询用setTimeout
  - **帧内切片 vs requestIdleCallback延后**：帧内切片：持续产出结果，可能降低帧率 vs requestIdleCallback：利用空闲时间，不影响帧率，建议：进度条/动画用切片；日志上报/预取用idle
  - **时间驱动 vs 帧驱动动画**：时间驱动(基于elapsed)：跨刷新率一致 vs 帧驱动(基于帧计数)：120Hz速度翻倍，建议：始终使用时间驱动，用timestamp参数

### A31-事件循环与宏微任务
机制：事件循环每轮迭代按顺序执行：(1)取一个macrotask执行；(2)清空microtask queue全部微任务；(3)渲染阶段(input→rAF→Observer→Style→Layout→Paint→Composite)。Macrotask来源包括setTimeout/setInterval/事件回调，每轮只执行一个。Microtask来源包括Promise.then/queueMicrotask/MutationObserver，必须全部清空才进入渲染。递归microtask可阻塞渲染。50ms以上连续占用构成Long Task。
瓶颈：
  - **微任务递归导致渲染饥饿**：Promise.then回调中不断产生新Promise → 页面冻结，无法响应输入，无法渲染
  - **setTimeout最小延迟不准**：setTimeout(fn,0)实际延迟4ms(嵌套超5层) → 定时器回调间隔不可靠
  - **事件处理回调阻塞主线程**：click/scroll/input回调执行耗时操作 → 用户操作后页面无响应
  - **宏任务队列积压**：大量事件同时触发(快速滚动) → 队列任务处理延迟越来越大
  - **async/await隐式微任务**：大量连续await产生大量微任务 → 看似同步的代码实际被拆成大量微任务
权衡：
  - **微任务 vs 宏任务调度**：Microtask：高优先级，可阻塞渲染，适合状态一致性 vs Macrotask：低优先级，不阻塞渲染，适合延迟执行，建议：需要立即一致的用microtask；允许延迟的用macrotask
  - **requestAnimationFrame vs requestIdleCallback**：rAF：渲染前高优先级，占用帧预算 vs rIC：渲染后空闲时，不影响帧率，建议：视觉相关用rAF；后台任务用rIC
  - **同步 vs 异步事件处理**：同步处理：无延迟，立即占用主线程 vs 异步处理(rAF/timeout)：有延迟，分散到后续帧，建议：简单操作同步处理；复杂计算异步切片

### A32-Web Worker多线程
机制：Web Worker 在后台线程运行 JavaScript，与主线程并行执行不阻塞 UI。数据通过 postMessage 传递（默认结构化克隆），Transferable Objects 支持零拷贝转移所有权，SharedArrayBuffer 实现真正的共享内存（需跨域隔离）。Worker 拥有独立的全局对象、事件循环和内存空间，无法访问 DOM。
瓶颈：
  - 结构化克隆大数据开销大
  - Worker 启动成本高（脚本加载解析）
  - Worker 需要 Pool 复用避免频繁创建/销毁
  - SharedArrayBuffer 竞态条件需手动 Atomics 管理
  - 跨域限制（同源策略）

### A33-Performance API与Long Task
机制：Performance API族提供浏览器性能测量基础设施。PerformanceObserver异步观察性能条目（longtask/mark/measure/resource/paint等）。Long Tasks API检测主线程连续占用≥50ms的任务，通过attribution提供归因信息。User Timing允许performance.mark()/measure()插入自定义标记和测量，支持DevTools Extensibility API在Performance面板自定义轨道显示。所有条目进入统一Performance Timeline。
瓶颈：
  - **Long Task检测误报与漏报**：浏览器合并小任务或跨帧边界不准确 → 报告的duration与实际卡顿感不一致
  - **PerformanceObserver内存开销**：观察多种entryType且不及时清理条目 → 内存占用持续增长
  - **User Timing条目累积**：频繁调用performance.mark()/measure() → Performance Timeline中条目过多，查询变慢
  - **Long Task归因信息不足**：任务发生在框架内部或跨iframe → attribution无法定位到具体业务代码
  - **跨浏览器兼容性**：不同浏览器对Performance API支持程度不同 → 某些entryType或属性在Safari/Firefox不可用
权衡：
  - **Long Task阈值选择**：50ms(规范阈值)：误报率高，漏报率低 vs 100ms(宽松阈值)：误报率中，漏报率中，建议：监控用50ms；告警用100ms；优化目标<50ms
  - **PerformanceObserver vs 轮询getEntries**：Observer：异步推送，低开销，时序保证 vs 轮询getEntries：依赖间隔，高开销，可能遗漏，建议：始终使用PerformanceObserver
  - **buffered: true vs false**：buffered:true：获取历史数据，高完整性 vs buffered:false：仅获取创建后数据，建议：页面加载相关用true；运行时监控用false

## 阶段一：理解请求阻塞首屏的机制

### 你将理解什么
为什么接口串行会导致首屏 3-5s？浏览器的请求调度机制是什么？

### Step 1：理解浏览器请求瀑布流，用实验直观对比
**做**：阅读 `05-网络请求优化/overview.md` 的 A1 节点（CRP）和 A15 节点（请求瀑布流）。理解串行请求如何叠加延迟。
**然后动手**：打开 `experiment/src/index.html`，默认「❌ 串行请求」模式。观察瀑布图：5 个请求依次串行，总耗时 = 5 × 延迟。然后切换到「⚡ 并发请求」→「🔀 BFF 聚合」→「🚀 预取 + 缓存」，观察总耗时逐步缩短。调整 API 延迟滑块到 800ms，对比差异更明显。
**你会看到什么**：串行: 4000ms → 并发: 800ms → BFF: 830ms → 预取: 首屏只需 800ms。
**这说明了什么**：串行是首屏慢的头号杀手，并发是免费的性能优化。BFF 减少请求数，预取提前加载非关键数据。
**接下来去哪**：阅读 `edge-cases.md` 的请求瀑布流坑点。
**做到才算过**：能画出串行 vs 并发的请求时序图。能在实验中对比四种模式的总耗时差异。

### Step 2：理解 HTTP 缓存对请求的优化
**做**：阅读 `overview.md` 的 A6 节点。理解缓存命中时浏览器直接跳过网络请求。
**你会看到什么**：强缓存命中 = 0ms（不发请求），协商缓存命中 = 少量 RTT（304）。
**这说明了什么**：最好的网络优化是"不发请求"。
**接下来去哪**：进入阶段二，学习并发控制。
**做到才算过**：能解释强缓存 vs 协商缓存的流程，以及它们对首屏时间的影响。

- [ ] 能画出串行 vs 并发的请求时序图
- [ ] 能解释浏览器同域 6 连接限制
- [ ] 能说出"不发请求"是最优的网络优化
- 做不到？→ 回到 `overview.md` A1/A6/A15 节点重读

## 阶段二：掌握并发控制与数据预取

### 你将理解什么
如何把串行请求改成并发？BFF 聚合是什么？数据预取怎么实现？

### Step 3：用 Promise.all 实现并发
**做**：阅读 `05-网络请求优化/overview.md` 的 A15 节点。把项目中 3 个串行 fetch 改成 `Promise.all`。
**然后动手**：在实验中将 API 延迟调到 500ms，对比「❌ 串行请求」（总耗时 2500ms）和「⚡ 并发请求」（总耗时 500ms）。观察瀑布图从串行变成并行。
**你会看到什么**：总时间 = max(各请求时间) 而不是 sum(各请求时间)。5 个请求从 2500ms 降到 500ms。
**这说明了什么**：Promise.all 是并发的最简实现，但要注意错误处理（一个失败全部失败）。
**接下来去哪**：阅读 `edge-cases.md` 的请求竞态坑点。
**做到才算过**：能将 3 个串行请求改写为 `Promise.all` 并发。能在实验中对比串行 vs 并发的总耗时。

### Step 4：理解 BFF 聚合与数据预取
**做**：阅读 `trade-offs.md` 的"客户端并发 vs BFF 聚合"。理解 BFF 层如何把多个后端接口合并为一个前端请求。
**你会看到什么**：BFF 聚合减少了前端请求数，但增加了服务端复杂度；数据预取在用户交互前就开始请求。
**这说明了什么**：并发是"同时发多个请求"，聚合是"减少请求数"，预取是"提前发请求"——三个维度可以叠加。
**接下来去哪**：进入阶段三，学习缓存协同。
**做到才算过**：能说出客户端并发、BFF 聚合、数据预取的区别和适用场景。

### 阶段二过关标准
- [ ] 能用 Promise.all 实现并发请求
- [ ] 能说出客户端并发 vs BFF 聚合的区别
- [ ] 能解释数据预取的实现方式
- 做不到？→ 回到 `overview.md` A15 节点 + `trade-offs.md` 重读

## 阶段三：理解缓存协同与 CWV 度量

### 你将理解什么
缓存和并发如何协同？LCP 受哪些请求影响？CLS 如何因懒加载而增加？

### Step 5：理解缓存回源风暴
**做**：阅读 `edge-cases.md` 的缓存回源风暴坑点。理解大量请求同时过期 → 源站 QPS 飙高的场景。
**你会看到什么**：加随机 jitter 到 Cache-Control max-age 可以分散回源时间。
**这说明了什么**：缓存策略要考虑规模化场景，单机优化和集群优化是不同的问题。
**接下来去哪**：阅读 `overview.md` 的 A8/A9 节点。
**做到才算过**：能说出缓存回源风暴的根因和 2 种缓解方案。

### Step 6：用 CWV 度量请求优化效果
**做**：阅读 `overview.md` 的 A8 节点。用 Lighthouse 测量优化前后的 LCP。
**你会看到什么**：LCP 元素如果是图片或大文本块，其加载时间直接影响 LCP 指标。
**这说明了什么**：请求优化不是"感觉快了"，而是要用 LCP/INP/CLS 量化。
**接下来去哪**：进入阶段四，综合落地。
**做到才算过**：能用 Lighthouse 测量 LCP 并解释其与请求优化的关系。

### 阶段三过关标准

- [ ] 能解释 LCP 与请求优化的关系
- [ ] 能用 Lighthouse 度量优化效果

## 阶段四：综合运用——React Query/SWR 落地

### 你将理解什么
如何用 React Query/SWR 把并发+缓存+预取组合成一个完整的请求管理方案？

### Step 7：配置 React Query/SWR
**做**：阅读 `trade-offs.md` 的"React Query/SWR vs 手动管理"。在项目中引入 React Query 或 SWR。
**你会看到什么**：声明式请求管理自动处理缓存、去重、重试、后台刷新。
**这说明了什么**：React Query/SWR 本质上是"请求并发+缓存+预取"的封装，比手动管理高效 10 倍。
**接下来去哪**：阅读 `edge-cases.md` 的防御方案。
**做到才算过**：能在项目中用 React Query/SWR 实现一个列表页的请求管理。

### Step 8：走完优化全流程
**做**：回顾 `overview.md` 的决策清单（场景→方案速查表），对照自己的项目逐项检查。
**你会看到什么**：不同场景（中后台、C 端内容、轻量落地页）的最优方案不同。
**这说明了什么**：没有万能方案，关键是理解约束、匹配方案。
**接下来去哪**：完成！

# 权衡分析：网络请求优化中的策略选择

## 从一个常见困惑说起

"首屏为什么这么慢？"——打开 DevTools Network 面板，你会看到一排请求像多米诺骨牌一样依次排队：先等用户信息返回，再用用户 ID 去拉推荐列表，最后才拿到首屏需要的核心数据。这些请求之间存在隐含的依赖关系，导致整个瀑布流（waterfall）拉长了数百毫秒甚至数秒。

面对这个问题，团队通常会陷入两难：是应该在客户端想办法并发这些请求，还是应该在服务端把它们聚合起来一次返回？缓存策略又该怎么定——长期缓存配合文件 hash 还是每次走 no-cache？请求库用 React Query/SWR 这类声明式方案，还是自己手写 fetch 逻辑？

这些不是"哪个更好"的问题，而是"在什么条件下选哪个"的问题。以下逐一拆解。

---

## 权衡一：客户端并发 vs BFF 聚合

### 背景

当一个页面需要调用 5-8 个接口，且其中部分存在依赖关系时，串行请求会导致首屏阻塞。解决路径分两条：前端做并发编排，或者引入 BFF（Backend For Frontend）层在服务端聚合。

### 客户端并发

**原理**：利用 `Promise.all` / `Promise.allSettled` 将无依赖的请求并行发出，依赖链上的请求用 `Promise.all` 嵌套或状态机编排。

**优势**：
- **灵活性高**：前端可以按需组合，不需要后端配合
- **迭代快**：接口变更时前端自行调整，不涉及后端部署
- **可观测性好**：每个请求的耗时、失败原因在浏览器 Network 面板清晰可见
- **渐进渲染**：各模块数据独立到达，可实现骨架屏 → 局部渲染的渐进体验

**劣势**：
- **浏览器并发限制**：同域名下 HTTP/1.1 默认 6 个 TCP 连接，并发过多请求会排队
- **重复请求**：多个组件可能各自请求同一份数据（如用户信息），造成浪费
- **错误处理复杂**：部分成功、部分失败时的降级逻辑需要精心设计
- **首字节时间（TTFB）叠加**：每个请求都要走一次完整网络链路，总延迟 = max(各请求耗时) 但带宽开销 = sum(各请求大小)

### BFF 聚合

**原理**：在前端和微服务之间加一层薄的聚合服务，将多个下游接口合并为一个前端专用接口。

**优势**：
- **减少网络往返**：前端只需一次请求，总延迟显著降低
- **减少连接数**：绕过浏览器并发限制，对 HTTP/1.1 环境尤其友好
- **数据裁剪**：BFF 可以只返回前端需要的字段，减少传输体积
- **统一错误处理**：聚合接口内部的失败可以被 BFF 层统一兜底

**劣势**：
- **增加架构复杂度**：多一个服务要部署、监控、维护
- **缓存粒度粗**：聚合接口的缓存失效策略比单接口更难设计——其中任何一个子接口数据变化，整个聚合缓存都可能失效
- **灵活性受限**：前端新增一个数据需求，需要 BFF 配合改接口，涉及双端联调
- **瀑布流只是转移了**：BFF 内部调用下游服务时仍然可能是串行的，如果 BFF 实现不当，问题只是从客户端移到了服务端

### 决策关键

| 维度 | 选客户端并发 | 选 BFF 聚合 |
|------|-------------|-------------|
| 接口依赖关系 | 多数无依赖或依赖链短 | 存在深层依赖链（3层+） |
| 团队结构 | 前端自治、后端接口稳定 | 有专职 BFF 团队或后端可快速响应 |
| 网络环境 | HTTP/2 普及（无并发限制） | 仍大量 HTTP/1.1 或弱网场景 |
| 数据复用度 | 各页面数据需求差异大 | 多页面共用同一组数据组合 |
| 迭代节奏 | 前端高频迭代、后端变更少 | 前后端同步发版 |

## 权衡二：HTTP 缓存——长期缓存 + Hash vs No-Cache

### 背景

缓存策略直接影响请求量和加载速度。两种极端方案各有适用场景。

### 长期缓存 + 内容 Hash（Cache-Control: max-age=31536000 + 文件名 hash）

**原理**：静态资源（JS/CSS/图片）文件名中嵌入内容哈希（如 `app.3a7f2b.js`），配合极长的 max-age。内容不变时浏览器直接用缓存；内容变化时文件名变化，自动触发重新请求。

**优势**：
- **极致的缓存命中率**：不变的资源永远不发请求
- **部署安全**：新版本自动生成新文件名，不存在缓存脏读问题
- **CDN 友好**：CDN 节点可以长期持有资源，回源频率极低

**劣势**：
- **首次访问无收益**：新用户首次访问仍需下载全部资源
- **HTML 不能用此策略**：HTML 入口文件必须每次检查更新（否则永远加载旧版 JS/CSS 引用）
- **小文件收益低**：对 API 响应等动态数据，hash 策略不适用

### No-Cache（Cache-Control: no-cache）

**原理**：每次请求都必须向服务器验证资源是否过期（条件请求，304 Not Modified）。服务器决定是否返回新内容。

**优势**：
- **数据实时性高**：适合频繁变化的业务数据
- **实现简单**：不需要 hash 生成流水线
- **调试友好**：开发者每次都能看到最新响应

**劣势**：
- **请求量大**：即使是 304，每次都要走一次网络验证（DNS + TCP + TLS + HTTP 往返）
- **弱网下体验差**：304 判断本身也需要网络连通性
- **服务器压力**：每次请求都要经过应用层处理（验证逻辑）

### 决策关键

| 维度 | 选长期缓存 + Hash | 选 No-Cache |
|------|------------------|-------------|
| 资源类型 | 静态资源（JS/CSS/字体/图片） | 动态 API 数据、用户个性化内容 |
| 更新频率 | 低频（发版时变化） | 高频（实时数据、用户操作相关） |
| 网络环境 | 带宽有限、CDN 覆盖好 | 内网或延迟极低的服务间调用 |
| 部署流程 | 有构建流水线（Webpack/Vite 自动 hash） | 快速原型、无构建工具的简单项目 |

**实践中的组合策略**：大多数生产环境同时使用两种策略——静态资源走长期缓存 + hash，API 数据走 `no-cache` 或短 `max-age` + `stale-while-revalidate`。

## 权衡三：React Query / SWR vs 手动管理请求状态

### 背景

当页面涉及多个请求时，每个请求的 loading / error / success 状态、缓存失效、重试逻辑、乐观更新等都是需要管理的横切关注点。是用成熟的请求状态库，还是自己封装？

### React Query / SWR 等声明式方案

**原理**：以 queryKey 为缓存键，自动管理请求生命周期、缓存、后台刷新、重试、去重。

**优势**：
- **请求去重**：同一 queryKey 的并发请求自动合并，避免重复请求（天然解决前文提到的"多组件请求同一数据"问题）
- **后台刷新**：`stale-while-revalidate` 模式下，先返回缓存数据，后台静默更新，用户体验流畅
- **自动重试**：网络抖动时自动重试，不需要手写重试逻辑
- **缓存管理**：预取（prefetch）、失效（invalidate）、乐观更新（optimistic update）等 API 开箱即用
- **DevTools**：React Query DevTools 可视化缓存状态，调试体验好

**劣势**：
- **学习成本**：queryKey 设计、staleTime/gcTime 配置、失效策略等概念需要团队理解
- **包体积**：React Query 约 13KB（gzip），对极致轻量项目是负担
- **抽象泄漏**：当需求超出库的设计假设时（如复杂的流式数据、WebSocket 混合场景），需要 hack 或降级
- **框架绑定**：React Query 绑定 React，SWR 也主要面向 React；跨框架项目需要额外适配

### 手动管理

**原理**：用 `fetch` / `axios` + 自定义 hooks 或状态管理库手动处理请求状态。

**优势**：
- **完全控制**：任何边界情况都可以精确处理，没有"库不支持"的限制
- **零依赖**：不引入额外包，项目更轻量
- **学习曲线平**：团队已经熟悉 fetch/axios，不需要学习新概念
- **灵活组合**：可以自由搭配任何状态管理方案（Redux、Zustand、Jotai 等）

**劣势**：
- **重复劳动**：每个项目都在重新实现缓存、重试、去重、loading 状态
- **容易出错**：竞态条件（组件卸载后更新状态）、内存泄漏、缓存一致性问题高频出现
- **缺乏统一规范**：不同开发者写出不同风格的请求管理代码，维护成本高
- **无后台刷新**：用户切换回页面时数据可能是过期的，除非手动实现

### 决策关键

| 维度 | 选 React Query/SWR | 选手动管理 |
|------|-------------------|-----------|
| 项目复杂度 | 多页面、多接口、频繁数据交互 | 简单页面、接口少、CRUD 为主 |
| 团队规模 | 中大型团队需要统一规范 | 小团队或个人项目 |
| 数据特征 | 有大量共享数据、需要缓存一致性 | 数据基本不共享、实时性要求极高 |
| 技术栈 | React 生态为主 | 多框架混用或非 React 项目 |
| 性能敏感度 | 需要精细的 stale-while-revalidate 控制 | 对缓存策略要求不高 |

## 收尾：推荐策略组合

没有"最优解"，只有"最适配"。以下是基于典型场景的推荐组合：

### 场景 A：中后台管理系统（中等复杂度）

- **请求编排**：客户端并发（`Promise.all`）——页面模块独立性强，接口依赖链短
- **静态资源**：长期缓存 + hash（Webpack/Vite 默认配置即可）
- **API 数据**：React Query —— 多页面共享用户信息、字典数据，需要缓存和去重
- **首屏优化**：关键接口用 React Query 的 `prefetchQuery` 在路由切换前预取

### 场景 B：C 端内容型产品（高并发、强首屏）

- **请求编排**：BFF 聚合首屏关键数据 + 非关键数据客户端并发延迟加载
- **静态资源**：长期缓存 + hash + CDN
- **API 数据**：`stale-while-revalidate` 策略，优先展示缓存数据，后台静默更新
- **首屏优化**：SSR/SSG + HTTP/2 Server Push 或 103 Early Hints

### 场景 C：轻量落地页 / 简单工具

- **请求编排**：顺序请求即可（接口少，依赖关系简单）
- **静态资源**：长期缓存 + hash
- **API 数据**：手写 fetch + 简单 loading 状态——不值得引入 React Query
- **首屏优化**：关键 CSS 内联、JS defer 加载

### 通用原则

1. **先测量再优化**：用 Lighthouse / WebPageTest 量化首屏瓶颈，不要凭直觉优化
2. **HTTP/2 改变并发策略**：HTTP/2 多路复用消除了浏览器并发限制，客户端并发方案的劣势大幅削弱
3. **缓存分层**：静态资源长期缓存、API 数据用 stale-while-revalidate、敏感数据 no-store
4. **渐进增强**：骨架屏 → 关键数据渲染 → 非关键数据延迟加载，比一次性等齐所有数据体验更好
5. **避免过度抽象**：如果只有 3 个接口，手写 fetch 就够了；React Query 的价值在 10+ 接口时才显现

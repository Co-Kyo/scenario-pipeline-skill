# MVE: 网络请求优化 — 瀑布流 vs 并发 vs 聚合

## 环境基线
- 浏览器：Chrome 90+ / Firefox 89+ / Safari 15+
- 无需 Node.js 或构建工具，纯 HTML+JS
- 操作系统：无约束

## 一键启动
```bash
open 05-网络请求优化/experiment/src/index.html
```

## 验证检查点

### 检查点 1：串行请求（瀑布流）
1. 保持默认「❌ 串行请求」模式
2. 观察瀑布图：5 个请求依次串行，总耗时 = 5 × 延迟
3. 调整 API 延迟滑块，观察总耗时线性增长
4. **验证能力**：A15-请求并发（瀑布流问题）

### 检查点 2：并发请求
1. 点击「⚡ 并发请求」
2. 瀑布图显示 5 个请求同时发起
3. 总耗时 = 单个请求延迟（而非 5 倍）
4. **验证能力**：A15-请求并发（Promise.all）

### 检查点 3：BFF 聚合
1. 点击「🔀 BFF 聚合」
2. 只有 1 个请求，后端内部并发
3. 总耗时 = 单次延迟 + 30ms（网络开销最小化）
4. **验证能力**：A15-请求并发（BFF 聚合模式）

### 检查点 4：预取 + 缓存
1. 点击「🚀 预取 + 缓存」
2. 首屏关键数据（用户+订单）先到，立即渲染
3. 非关键数据（推荐/消息/广告）后台预取
4. **验证能力**：A15-请求并发（分级加载）+ A8-Core Web Vitals

### 检查点 5：延迟敏感性
1. 将 API 延迟调到 800ms
2. 对比四种模式的总耗时差异
3. 串行: 4000ms vs 并发: 800ms vs BFF: 830ms
4. **验证能力**：A15-请求并发（延迟对不同模式的影响）

## 故障排除

- **瀑布图为模拟数据**：本实验用可视化方式展示请求时序关系
- **想看真实网络**：用 Chrome DevTools → Network 面板查看实际瀑布图

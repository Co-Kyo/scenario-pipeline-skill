# 权衡分析：构建产物优化

## 从一个常见困惑说起

"首屏 JS 超过 500KB，我要拆 chunk，但拆多细才对？"

这个问题背后藏着一系列连锁决策：拆得太粗，首屏体积降不下来；拆得太细，浏览器并行请求激增，HTTP/1.1 下反而更慢。更麻烦的是，chunk 拆分策略直接影响缓存命中率——你精心拆出的 vendor chunk，一次小版本升级就全量失效，缓存形同虚设。

这个困惑的本质是：**代码拆分粒度、缓存策略、加载性能三者互相耦合，没有一个维度可以独立优化。**

---

## 权衡一：缓存策略 —— 长期缓存 + Hash 文件名 vs 协商缓存 (no-cache)

### 核心矛盾

构建产物是"内容可变、但变化有规律"的资源。应用代码频繁变更，第三方依赖相对稳定。缓存策略必须在这个差异上做文章，否则要么用户拿到过期代码，要么每次请求都重新下载。

### 两条路径

| 维度 | 长期缓存 + 内容 Hash | 协商缓存 (no-cache) |
|------|----------------------|---------------------|
| **原理** | 文件名含内容哈希（如 `app.a1b2c3.js`），设置 `Cache-Control: max-age=31536000`，内容不变则文件名不变，浏览器直接用本地缓存 | 每次请求携带 `ETag`/`Last-Modified`，服务器比较后决定返回 200（新内容）或 304（未修改） |
| **缓存命中条件** | 文件名精确匹配即可 | 需要一次网络往返验证 |
| **内容更新感知** | 内容变了 → 哈希变了 → 文件名变了 → 自动拉新 | 依赖服务器正确返回 304/200 |
| **网络开销** | 命中时零网络请求 | 每次至少一次条件请求（304 也有延迟） |
| **构建复杂度** | 需要构建工具支持哈希生成，HTML 模板需动态引用 | 无需特殊构建配置 |
| **CDN 友好度** | 极高——CDN 可永久缓存，无需回源验证 | 较低——CDN 需要回源做条件请求 |

### 原则与判断

**优先选择长期缓存 + Hash 文件名。** 原因：

1. **现代构建工具原生支持。** Webpack 的 `[contenthash]`、Vite 的默认行为都已内置哈希文件名，额外成本几乎为零。
2. **减少一次网络往返。** 在移动端弱网环境下，304 的 RTT（往返时延）可能高达数百毫秒，直接命中缓存省掉了这个代价。
3. **CDN 缓存效率翻倍。** 长期缓存的资源可以被 CDN 边缘节点永久持有，命中率远高于需要回源验证的协商缓存。

**协商缓存的适用场景：** 频率极低更新的静态资源（如字体文件），或需要精确控制服务端返回内容的场景（如基于 Cookie 的个性化资源）。但这些场景在构建产物中占比很小。

### 关键陷阱

Hash 文件名的缓存优势依赖一个前提：**你必须正确拆分 chunk。** 如果一个 200KB 的 vendor chunk 里混入了频繁变更的业务代码，每次发布都会导致整个 vendor chunk 哈希变化，长期缓存彻底失效。这就是为什么缓存策略和代码拆分策略必须联动考虑。

## 权衡二：Code Splitting 粒度 —— 路由级 vs 组件级

### 核心矛盾

拆分的目标是"首屏只加载首屏需要的代码"。但"首屏需要什么"本身就是一个模糊概念——用户打开首页，到底需要哪些组件？哪些是首屏渲染必需的，哪些可以延迟加载？

### 两条路径

| 维度 | 路由级分割 | 组件级分割 |
|------|-----------|-----------|
| **粒度** | 按页面/路由为单位拆分（如 `/home`、`/dashboard` 各一个 chunk） | 按组件/模块为单位拆分（如 `<Modal>`、`<Chart>` 各一个 chunk） |
| **实现复杂度** | 低——路由库天然支持懒加载（`React.lazy`、`() => import(...)`） | 高——需要识别每个组件的加载时机，可能涉及 `Suspense` 边界设计 |
| **首屏体积控制** | 粗粒度，但对 SPA 效果显著（从全量 500KB 降到单路由 150KB） | 细粒度，可精确控制每个组件的加载时机 |
| **请求碎片化风险** | 低——通常只多 1-3 个请求 | 高——可能产生几十个小 chunk，在 HTTP/1.1 下成为瓶颈 |
| **代码复用** | 天然——路由间共享的代码自动进入公共 chunk | 需要额外配置，否则每个组件 chunk 可能包含重复代码 |
| **维护成本** | 低——新增页面只需加一行懒加载 | 高——需要持续审查哪些组件该懒加载 |

### 原则与判断

**默认从路由级分割开始，组件级分割作为补充手段。** 原因：

1. **投入产出比最高。** 路由级分割用最少的改动（通常就几行代码）就能把首屏体积砍掉 50-70%，这是最划算的优化。
2. **避免请求碎片化。** HTTP/1.1 对同一域名有 6 个并发连接限制，chunk 过多会导致队头阻塞。即使在 HTTP/2 下，过多小请求也会增加服务端和客户端的处理开销。
3. **组件级分割是"高阶武器"。** 当路由级分割已经把首屏压到合理范围后，再针对特定重型组件（如富文本编辑器、图表库、地图 SDK）做组件级懒加载。这些组件通常是"交互触发才需要"的，延迟加载不会影响首屏渲染。

### 关键陷阱

过度分割（over-splitting）比不分割更危险。每个 chunk 都有固定的元数据开销（模块清单、运行时胶水代码），chunk 越小，这个开销占比越大。经验法则：**单个 chunk 不要小于 20KB（压缩后）。**

## 权衡三：Vendor Chunk 策略 —— 单一 vendor vs 拆分 vendor

### 核心矛盾

第三方依赖（React、Lodash、Ant Design……）通常占总体积的 60-80%。这些依赖更新频率远低于业务代码。如何组织 vendor chunk，直接决定了缓存效率。

### 两条路径

| 维度 | 单一 vendor chunk | 拆分 vendor chunk |
|------|------------------|-------------------|
| **做法** | 所有 `node_modules` 打入一个 `vendor.[hash].js` | 按依赖更新频率拆分：`react-vendor.[hash].js`（极少更新）、`ui-vendor.[hash].js`（偶尔更新）、`util-vendor.[hash].js`（频繁更新） |
| **缓存命中率** | 低——任何一个依赖升级，整个 vendor chunk 哈希变化 | 高——只更新变化的部分，其他 chunk 缓存继续有效 |
| **HTTP 请求数** | 少——1 个 vendor 请求 | 多——可能 3-5 个 vendor 请求 |
| **配置复杂度** | 低——Webpack `splitChunks` 默认策略即可 | 高——需要精细配置 `cacheGroups`，或 Vite 的 `manualChunks` |
| **公共代码去重** | 自动——单一 chunk 天然无重复 | 需要确保跨 chunk 不重复打包同一依赖 |

### 原则与判断

**推荐"中等粒度"拆分 vendor。** 不要追求极致细粒度的 vendor 拆分，也不要把所有依赖塞一个包。具体策略：

1. **识别"稳定层"依赖。** React/Vue 核心、路由库、状态管理这类依赖，升级周期通常是季度甚至年度级别，单独成 chunk 后可以享受超长缓存。
2. **识别"活跃层"依赖。** 业务 UI 组件库、工具函数库，更新相对频繁，可以归为一组。
3. **让构建工具自动处理剩余。** Webpack 的 `splitChunks.maxSize` 和 Vite 的 `manualChunks` 可以设置上限，自动拆分过大的 chunk，无需手动枚举每个依赖。

### 关键陷阱

**不要忽略公共代码重复问题。** 拆分 vendor 时，如果 A chunk 和 B chunk 都依赖 `lodash`，而没有正确配置公共代码提取，`lodash` 会被打包两次。Webpack 的 `splitChunks` 默认会提取公共模块，但 `manualChunks`（Vite）需要你自己处理这个逻辑。

## Webpack vs Vite：推荐策略

两个工具的哲学差异决定了优化路径的不同：

### Webpack

- **拆分策略：** 用 `splitChunks.cacheGroups` 精细控制。推荐配置：
  - `vendors` 组：`node_modules` 中的依赖
  - `common` 组：被 2 个以上 chunk 引用的业务代码
  - `maxSize: 250000`（250KB）：自动拆分过大的 chunk
- **缓存策略：** `[contenthash]` 文件名 + `max-age=31536000`，配合 `HtmlWebpackPlugin` 自动注入带哈希的引用
- **Tree Shaking：** 确保 `mode: 'production'`，使用 ES Module 格式的依赖（检查 `package.json` 的 `sideEffects` 字段）

### Vite

- **拆分策略：** 生产构建基于 Rollup，用 `build.rollupOptions.output.manualChunks` 控制。推荐做法：
  - 将 `react`/`vue` 核心单独提取
  - 将大型依赖（`echarts`、`antd` 等）单独提取
  - 其余交给 Rollup 自动拆分（`output.manualChunks` 可设为函数，按路径判断）
- **缓存策略：** Vite 默认生成带哈希的文件名，天然支持长期缓存
- **优势：** 开发阶段的 ESM 原生加载不需要打包，生产构建的 Rollup 配置比 Webpack 更简洁
- **注意：** Vite 的 `manualChunks` 不像 Webpack 的 `splitChunks` 那样自动去重，拆分时需注意避免重复打包

### 最终推荐

| 场景 | 推荐工具 | 核心策略 |
|------|---------|---------|
| 已有 Webpack 项目 | 保持 Webpack | `splitChunks` + `[contenthash]` + 路由级懒加载 |
| 新项目 | Vite | `manualChunks` 提取稳定依赖 + 路由级懒加载 |
| 首屏 > 500KB 的紧急优化 | 两者均可 | 先做路由级分割（最快见效），再拆 vendor（缓存收益），最后针对重型组件做组件级懒加载 |

**优先级排序：路由级 Code Splitting > Vendor 拆分 + 长期缓存 > 组件级懒加载 > Tree Shaking 深度优化。**

前三步通常就能把首屏 JS 从 500KB 压到 150KB 以内。Tree Shaking 的边际收益递减，适合在其他手段用尽后再精细打磨。

## 阶段一：网络协议基础

### 你将理解什么
网络优化的底层是 HTTP 协议。理解缓存、多路复用、QUIC，才能做出正确的优化决策。

### Step 1：HTTP 缓存策略
**做**：读 `capabilities/A6-HTTP缓存策略.md`。
**你会看到什么**：强缓存（Cache-Control: max-age）不发请求；协商缓存（ETag/Last-Modified）发请求但可能 304。
**这说明了什么**：缓存策略决定了"第二次访问要不要重新下载"，是首屏优化的基础。
**接下来去哪**：读 `overview.md` 第一节。
**做到才算过**：能区分强缓存和协商缓存的触发条件。

### Step 2：HTTP/2 多路复用
**做**：读 `capabilities/A18-HTTP2多路复用.md`。
**你会看到什么**：HTTP/1.1 的队头阻塞（同一连接只能串行请求）；HTTP/2 单连接多流并行。
**这说明了什么**：HTTP/2 下不再需要域名分片（多域名并发），合并文件反而可能有害（无法并行）。
**接下来去哪**：读 `capabilities/A19-HTTP3与QUIC.md`。
**做到才算过**：能解释 HTTP/2 的多路复用为什么消除了队头阻塞。

### Step 3：HTTP/3 与 QUIC
**做**：读 `capabilities/A19-HTTP3与QUIC.md`。
**你会看到什么**：QUIC 基于 UDP，0-RTT 连接，解决了 TCP 层的队头阻塞。
**这说明了什么**：弱网环境下 QUIC 的优势最明显——丢包只影响单个流，不影响整个连接。
**接下来去哪**：读 `trade-offs.md` 中 HTTP/2 vs HTTP/3 的对比。
**做到才算过**：能解释 QUIC 相比 TCP 的三个核心优势。

### 阶段一过关标准
- [ ] 能设计一个静态资源的缓存策略
- [ ] 能解释 HTTP/1.1 → HTTP/2 → HTTP/3 的演进动机
- [ ] 做不到？→ 回看 `capabilities/A6` + `A18` + `A19`

## 阶段二：资源策略

### 你将理解什么
协议之外，优化的核心是"让资源更近（CDN）、更小（压缩）、更少（接口合并）"。

### Step 4：CDN 与边缘缓存
**做**：读 `capabilities/A7-CDN与资源分发.md`。
**你会看到什么**：CDN 将资源缓存到离用户最近的边缘节点，减少 RTT。
**这说明了什么**：CDN + HTTP 缓存 = 首次访问快 + 后续访问更快。
**接下来去哪**：读 `capabilities/A20-资源压缩与gzip-brotli.md`。
**做到才算过**：能解释 CDN 的缓存层级（边缘→中间→源站）。

### Step 5：资源压缩
**做**：读 `capabilities/A20-资源压缩与gzip/brotli.md`。
**你会看到什么**：gzip 压缩率约 70%，brotli 约 75-80%，但 brotli 压缩更慢。
**这说明了什么**：静态资源用 brotli 预压缩（构建时），动态内容用 gzip（实时压缩 CPU 开销低）。
**接下来去哪**：读 `capabilities/A21-接口合并与BFF.md`。
**做到才算过**：能在一个项目中配置 gzip/brotli 压缩。

### Step 6：接口合并与 BFF
**做**：读 `capabilities/A21-接口合并与BFF.md`。
**你会看到什么**：BFF 聚合层将多个后端接口合并为一个前端请求；GraphQL 让前端按需请求字段。
**这说明了什么**：减少请求数是最直接的网络优化——但 HTTP/2 下请求数的影响变小了。
**接下来去哪**：读 `trade-offs.md` 的选型建议。
**做到才算过**：能解释 BFF 和 GraphQL 各自的适用场景。

### 阶段二过关标准
- [ ] 能设计一个 CDN + 缓存 + 压缩的完整资源策略
- [ ] 能解释 BFF 的价值和局限
- [ ] 做不到？→ 回看 `capabilities/A7` + `A20` + `A21`

## 阶段三：容错与降级

### 你将理解什么
弱网下不是"优化"问题，而是"能不能用"的问题。需要重试、降级、离线缓存。

### Step 7：请求重试与降级
**做**：读 `capabilities/A22-请求重试与降级.md`。
**你会看到什么**：指数退避重试（1s→2s→4s→8s）、fallback 缓存、graceful degradation。
**这说明了什么**：重试不是"失败了再试一次"，是有策略地退避+限制次数+降级兜底。
**接下来去哪**：读 `capabilities/A23-ServiceWorker离线缓存.md`。
**做到才算过**：能实现一个指数退避重试函数。

### Step 8：Service Worker 离线缓存
**做**：读 `capabilities/A23-ServiceWorker离线缓存.md`。
**你会看到什么**：SW 拦截请求，命中缓存时直接返回，不依赖网络。Workbox 提供开箱即用的策略（CacheFirst、NetworkFirst 等）。
**这说明了什么**：SW 是"离线优先"架构的核心——但要注意缓存更新策略，否则用户看到旧版本。
**接下来去哪**：读 `edge-cases.md` 关于"SW 缓存更新"的部分。
**做到才算过**：能用 Workbox 配置一个静态资源的 CacheFirst 策略。

### 阶段三过关标准
- [ ] 能实现指数退避重试
- [ ] 能用 Service Worker 实现离线缓存
- [ ] 做不到？→ 回看 `capabilities/A22` + `A23`

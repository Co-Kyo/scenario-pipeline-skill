# 边缘案例与深水坑：网络优化 — 弱网环境下的资源加载策略

> 命题：网络优化：弱网环境下的资源加载策略
> 生成时间：2026-05-05
> 分类层级：领域 → 子域 → 坑点
> 约束：通用 ≥ 95%，每个坑点含：现象、触发条件、检测、缓解、能力ID

## 一、缓存策略与一致性

### 1.1 HTTP 缓存语义误解

#### 1.1.1 no-cache ≠ 不缓存

| 维度 | 内容 |
|------|------|
| **现象** | 开发者将 `Cache-Control: no-cache` 理解为"不缓存"，实际该指令表示"每次使用前必须向源服务器验证"。浏览器仍会缓存资源，但每次使用前发条件请求（`If-None-Match` / `If-Modified-Since`）。若验证失败，仍返回 304 而非重新下载。结果是：看似"禁用缓存"的配置，实际上缓存行为完全生效，只是多了一轮验证请求。在弱网环境下，条件请求本身也会阻塞资源可用时间。 |
| **触发条件** | 将 `no-cache` 与 `no-store` 混淆；或仅设 `no-cache` 而未配合 `ETag` / `Last-Modified`，导致验证机制缺失，浏览器退化为强制重新请求。 |
| **检测** | DevTools Network 面板观察请求状态码：若预期每次 200 但实际频繁 304，说明缓存仍在生效；检查响应头是否同时包含 `ETag` 或 `Last-Modified`。 |
| **缓解** | 严格区分 `no-cache`（需验证）与 `no-store`（不存储）；静态资源用内容哈希文件名 + 长期缓存（`max-age=31536000, immutable`）；动态资源用 `no-store` 或短 `max-age` + `must-revalidate`。 |
| **能力ID** | `CAP-CACHE-SEMANTICS` |

#### 1.1.2 CDN 的 s-maxage 与 max-age 不同步

| 维度 | 内容 |
|------|------|
| **现象** | `Cache-Control: max-age=60, s-maxage=3600` 配置下，CDN 节点缓存 1 小时，浏览器缓存 1 分钟。当源站更新内容后，CDN 仍返回旧版本长达 1 小时，而浏览器已过期重新请求，拿到的仍是 CDN 的过期缓存。用户看到的是"更新了但没完全更新"的割裂体验。更复杂的是，部分 CDN 会忽略 `s-maxage`，直接使用 `max-age`，导致行为不可预测。 |
| **触发条件** | 同时设置 `max-age` 和 `s-maxage` 且差值较大；CDN 回源策略与源站缓存头不一致；多级 CDN（CDN → 中间层 → 源站）各层对 `s-maxage` 的解析不同。 |
| **检测** | 对比不同层级（浏览器 → CDN 边缘 → CDN 中间层 → 源站）的响应头和响应体版本；使用 `curl -I` 直接请求 CDN 边缘节点验证缓存命中状态（`X-Cache: HIT` 等）。 |
| **缓解** | 统一 `max-age` 与 `s-maxage` 为相同值，或明确分层策略；使用版本化 URL（内容哈希）配合长期缓存；配置 CDN 的缓存清除（purge）能力作为紧急更新通道。 |
| **能力ID** | `CAP-CDN-SYNC` |

#### 1.1.3 Vary 头碎片化导致缓存命中率崩塌

| 维度 | 内容 |
|------|------|
| **现象** | `Vary: Accept-Encoding, Accept-Language, User-Agent` 配置下，CDN 需要为每种组合维护独立缓存副本。`User-Agent` 值有数千种变体，导致缓存条目爆炸式增长，命中率急剧下降。极端情况下，每个请求都回源，CDN 形同虚设。 |
| **触发条件** | `Vary` 头包含高基数字段（如 `User-Agent`、`Cookie`）；CDN 缓存容量有限而变体组合过多；中间代理（如 Nginx）自行添加额外 `Vary` 头。 |
| **检测** | 统计 CDN 缓存命中率，若低于 80% 且流量稳定，需排查 `Vary` 配置；检查 CDN 日志中的缓存 key 构成，确认变体字段范围。 |
| **缓解** | `Vary` 仅包含必要字段（通常只需 `Accept-Encoding`）；将 `User-Agent` 分类为有限组（如"现代浏览器"/"旧浏览器"）后通过中间层标准化请求头；避免在 `Vary` 中使用 `Cookie`，改用独立 URL 路径区分登录态内容。 |
| **能力ID** | `CAP-VARY-FRAGMENT` |

### 1.2 多级缓存与回源风暴

#### 1.2.1 多级缓存更新延迟放大

| 维度 | 内容 |
|------|------|
| **现象** | 架构为 浏览器缓存 → CDN 边缘 → CDN 中间层 → 应用缓存（Redis） → 数据库，每一级都有独立的 TTL。当数据更新时，需要等待所有层级的缓存自然过期，最坏情况下延迟为各级 TTL 之和。用户可能在更新后数分钟甚至数小时内仍看到旧数据。 |
| **触发条件** | 缓存层级多（≥3 级）且各级 TTL 独立设置；无主动失效（purge/invalidation）机制；不同团队分别管理不同缓存层，缺乏统一协调。 |
| **检测** | 在源站更新数据后，逐级检查各缓存层的返回版本，记录每级的更新时间差；监控各层缓存命中率和 TTL 分布。 |
| **缓解** | 建立统一的缓存失效协议，源站更新时级联清除各层缓存；采用版本化 URL 使旧缓存自然失效；减少缓存层级，合并功能相似的缓存层。 |
| **能力ID** | `CAP-CASCADE-TTL` |

#### 1.2.2 动态内容缓存失效

| 维度 | 内容 |
|------|------|
| **现象** | 个性化内容（如用户推荐、购物车、实时库存）被错误缓存，导致用户 A 看到用户 B 的数据，或看到过期的库存状态。由于动态内容通常设置 `Cache-Control: no-store`，完全放弃缓存后在弱网下页面加载极慢。 |
| **触发条件** | BFF 层未区分公共与私有数据，将含用户信息的响应整体标记为可缓存；CDN 配置忽略了 `Set-Cookie` 头的存在，缓存了带 cookie 的响应；缓存 key 未包含用户标识。 |
| **检测** | 多用户并发访问同一 URL，对比返回内容是否一致；检查缓存命中时响应中是否包含 `Set-Cookie` 头（不应命中）。 |
| **缓解** | 严格分离公共数据（可缓存）与私有数据（不可缓存）；对混合页面采用 ESI（Edge Side Includes）或客户端拼装策略；动态数据使用短 TTL + stale-while-revalidate 模式平衡新鲜度与性能。 |
| **能力ID** | `CAP-DYNAMIC-CACHE` |

#### 1.2.3 缓存同时过期引发回源风暴

| 维度 | 内容 |
|------|------|
| **现象** | 大量 CDN 节点在同一时刻缓存过期，瞬间向源站发送海量回源请求，源站被压垮，响应变慢或直接返回 503。弱网用户的请求排在队列末尾，等待时间极长。 |
| **触发条件** | 缓存资源使用固定 TTL 且内容在短时间内被大量节点同时缓存（如大促预热、热点新闻）；源站发布新版本后 CDN purge 触发全局失效；TTL 设置过短导致频繁集体过期。 |
| **检测** | 监控源站 QPS 是否出现周期性尖峰（与 TTL 周期吻合）；CDN 回源率突然从 <5% 飙升到 >80%；源站响应时间（P99）出现周期性劣化。 |
| **缓解** | TTL 添加随机抖动（如 `max-age=3600 + random(0, 300)`）；使用 stale-while-revalidate 允许返回过期缓存的同时后台刷新；部署源站限流和排队机制，保护后端不被打垮。 |
| **能力ID** | `CAP-THUNDERING-HERD` |

## 二、传输协议与连接优化

### 2.1 HTTP/2 的隐藏陷阱

#### 2.1.1 服务端推送效果不如预期

| 维度 | 内容 |
|------|------|
| **现象** | 启用 HTTP/2 Server Push 后，性能反而下降。浏览器已有缓存的资源被重复推送，浪费带宽；推送的资源优先级与浏览器实际需求不匹配，抢占了关键资源的带宽；在弱网下，推送的资源阻塞了主请求的传输。 |
| **触发条件** | 未检查浏览器缓存就无差别推送；推送资源列表固定，未根据页面实际依赖动态调整；推送资源体积过大或数量过多；网络带宽 < 1Mbps 时推送反而增加拥塞。 |
| **检测** | 对比开启/关闭 Server Push 前后的 LCP、FCP 指标；检查推送资源的缓存命中率（若 > 90% 说明大部分推送是浪费）；分析带宽利用率，确认推送是否挤占了关键路径资源。 |
| **缓解** | 使用 `103 Early Hints` 替代 Server Push（浏览器自行决定是否获取）；仅推送首屏关键资源且体积 < 14KB（避免拥塞窗口溢出）；实现缓存感知推送，通过 cookie 或缓存 API 检测浏览器是否已有该资源。 |
| **能力ID** | `CAP-H2-PUSH` |

#### 2.1.2 HTTP/2 队头阻塞在丢包时更严重

| 维度 | 内容 |
|------|------|
| **现象** | HTTP/2 虽然解决了应用层队头阻塞，但 TCP 层的队头阻塞仍然存在。当一个 TCP 包丢失时，该连接上的所有流都被阻塞，即使其他流的数据已完整到达。在弱网（丢包率 > 2%）环境下，HTTP/2 单连接的吞吐量可能低于 HTTP/1.1 的多连接方案。 |
| **触发条件** | 弱网环境（高丢包率、高延迟）；所有资源复用单一 TCP 连接；传输大文件时丢包触发拥塞窗口收缩，影响同连接上的小文件（如关键 CSS/JS）。 |
| **检测** | 对比弱网下 HTTP/1.1（6 连接）与 HTTP/2（单连接）的页面加载时间；通过 `chrome://net-internals/#http2` 观察流状态和阻塞情况；模拟丢包环境（tc netem）测试吞吐量衰减曲线。 |
| **缓解** | 对弱网场景可回退到 HTTP/1.1 多连接策略；部署 HTTP/3（QUIC）彻底解决传输层队头阻塞；关键资源使用独立域名/连接以避免被大文件阻塞。 |
| **能力ID** | `CAP-H2-HOL` |

### 2.2 QUIC / HTTP/3 的部署陷阱

#### 2.2.1 UDP 被企业防火墙拦截

| 维度 | 内容 |
|------|------|
| **现象** | QUIC 基于 UDP 传输，大量企业网络、学校网络、运营商中间设备仅放行 TCP 端口（80/443），UDP 流量被静默丢弃。客户端发起 QUIC 连接后超时回退到 TCP，增加了首次连接延迟（多了一个 RTT 的浪费）。部分防火墙甚至会重置 UDP 连接，导致异常行为。 |
| **触发条件** | 用户处于企业内网、学校网络、公共 WiFi 等受限网络环境；运营商中间设备（如透明代理）不支持 UDP；防火墙配置了严格的出站规则仅允许 TCP。 |
| **检测** | 监控 QUIC 连接建立失败率和回退到 TCP 的比例；分析不同网络环境下（企业/家庭/移动）的协议分布；使用 UDP 可达性测试工具检测端到端 UDP 连通性。 |
| **缓解** | 必须保留 TCP/TLS 回退路径，QUIC 作为优化而非替代；实现智能协议选择，根据历史连接成功率自动降级；在 Alt-Svc 头中声明 HTTP/3 支持，让浏览器自行探测。 |
| **能力ID** | `CAP-QUIC-FIREWALL` |

#### 2.2.2 0-RTT 重放攻击风险

| 维度 | 内容 |
|------|------|
| **现象** | QUIC 的 0-RTT 连接恢复允许客户端在首次握手时就发送应用数据，但这些数据可被攻击者捕获并重放。对于非幂等请求（如扣款、下单），重放会导致重复执行业务逻辑。 |
| **触发条件** | 服务端启用了 0-RTT 且未做防重放处理；客户端使用旧的会话票据（session ticket）发送 0-RTT 数据；请求为非幂等操作（POST 写入类接口）。 |
| **检测** | 安全审计中检查 0-RTT 是否启用；验证服务端是否对 0-RTT 请求做了幂等性保护；检查是否有 anti-replay 机制（如 strike register 或时间窗口）。 |
| **缓解** | 仅对 GET 等幂等请求启用 0-RTT；非幂等接口强制要求 1-RTT 完整握手；实现服务端 anti-replay 机制（基于 strike register 或单次令牌）；在响应头中标记 `Early-Data: 1` 供中间层识别。 |
| **能力ID** | `CAP-0RTT-REPLAY` |

#### 2.2.3 中间设备兼容性问题

| 维度 | 内容 |
|------|------|
| **现象** | 部分企业级负载均衡器、WAF、DPI（深度包检测）设备不理解 QUIC 协议，可能错误修改或丢弃 QUIC 数据包。表现为连接建立后频繁断开、数据损坏、或连接莫名降级到 TCP。问题难以定位，因为中间设备通常对应用层透明。 |
| **触发条件** | 网络路径中存在不支持 QUIC 的中间设备（老旧负载均衡器、企业 WAF）；中间设备尝试解析 UDP 载荷但格式不匹配；TLS 1.3 的加密扩展导致 DPI 设备无法检查内容而选择丢弃。 |
| **检测** | 使用 `quicreach` 等工具探测端到端 QUIC 可达性；对比直连和经过企业网络的 QUIC 连接成功率；抓包分析 QUIC Initial 包是否被中间设备修改或丢弃。 |
| **缓解** | 实现 QUIC 连接健康检查，连续失败时自动回退 TCP；使用 QUIC v1（RFC 9000）标准端口 443，避免非标端口被拦截；在运维文档中明确 QUIC 的网络兼容性要求。 |
| **能力ID** | `CAP-QUIC-COMPAT` |

## 三、压缩与编码优化

### 3.1 压缩策略的反直觉陷阱

#### 3.1.1 Brotli 压缩 CPU 开销过高

| 维度 | 内容 |
|------|------|
| **现象** | Brotli 最高压缩级别（11）相比 gzip 能多压缩 15-25%，但压缩耗时增加 10-50 倍。在动态压缩场景下，CPU 成为瓶颈，响应延迟反而增加。弱网用户虽然下载量减少，但等待服务端压缩的时间更长，总耗时可能更差。 |
| **触发条件** | 对动态内容实时使用 brotli-11 压缩；服务器 CPU 资源不足或并发请求量大；未区分静态预压缩和动态压缩的策略。 |
| **检测** | 监控压缩中间件的 P99 延迟；对比 brotli 与 gzip 的端到端响应时间（含压缩耗时）；观察服务器 CPU 使用率是否因压缩而接近饱和。 |
| **缓解** | 静态资源预压缩（构建时生成 brotli-11 压缩文件），动态内容使用 brotli-4 或 gzip；根据 CPU 负载自适应切换压缩级别；弱网场景下压缩收益更高，但需确保压缩耗时 < 网络传输节省时间。 |
| **能力ID** | `CAP-BROTLI-CPU` |

#### 3.1.2 小文件压缩后反而更大

| 维度 | 内容 |
|------|------|
| **现象** | 压缩算法对小文件（< 1KB）效果有限，压缩后的数据可能比原始数据更大。brotli 的元数据开销（字典初始化）比 gzip 更大，对极小文件的"负压缩"更明显。大量小文件（如图标、配置片段）被压缩后体积反而增加，浪费 CPU 且增大传输量。 |
| **触发条件** | 压缩中间件未设置最小体积阈值；大量小文件（< 256B）被无差别压缩；已压缩格式（JPEG、PNG、WASM）被二次压缩。 |
| **检测** | 统计压缩前后的体积比，筛选出压缩比 > 1 的资源；检查压缩中间件的配置是否有最小体积和内容类型过滤。 |
| **缓解** | 设置压缩最小体积阈值（通常 256B-1KB）；排除已压缩格式（image/*、font/woff2、application/wasm）；对小文件使用 gzip 而非 brotli（gzip 元数据开销更小）。 |
| **能力ID** | `CAP-COMPRESS-SMALL` |

#### 3.1.3 压缩级别选择的两难

| 维度 | 内容 |
|------|------|
| **现象** | gzip 级别 1 与级别 9 的压缩率差异约 10-15%，但耗时差异 5-10 倍。选择高压缩级别减少传输量但增加 CPU 和延迟；选择低压缩级别节省 CPU 但弱网用户下载更慢。没有"万能"级别，不同场景需要不同策略。 |
| **触发条件** | 全局统一压缩级别，未根据资源类型和网络条件差异化；未考虑 CDN 是否已开启压缩（双重压缩浪费）；压缩级别配置后从未根据实际数据调优。 |
| **检测** | 对不同压缩级别进行基准测试（压缩率 vs 耗时 vs CPU）；统计实际传输中各资源类型的压缩收益分布；检查 CDN 是否已压缩，避免源站重复压缩。 |
| **缓解** | 静态资源用最高压缩级别预压缩（离线构建）；动态内容用中等级别（gzip-4 / brotli-4）平衡速度与压缩率；根据资源类型定制策略（JSON/HTML 高压缩，图片跳过压缩）。 |
| **能力ID** | `CAP-COMPRESS-LEVEL` |

## 四、BFF 与聚合层

### 4.1 BFF 层的架构风险

#### 4.1.1 BFF 成为单点故障

| 维度 | 内容 |
|------|------|
| **现象** | BFF（Backend For Frontend）层聚合了多个下游服务的数据，一旦 BFF 本身故障或过载，所有前端页面都无法正常加载。弱网用户的请求到达 BFF 时已消耗大量时间，BFF 响应慢进一步恶化体验。BFF 从"优化层"变成了"瓶颈层"。 |
| **触发条件** | BFF 层无降级策略，下游任一服务超时即整体失败；BFF 未做限流，突发流量直接穿透到下游；单机部署或实例数不足，无法应对故障转移。 |
| **检测** | 监控 BFF 层的可用性和 P99 延迟；检查 BFF 对下游服务的依赖拓扑，识别关键路径；模拟单个下游服务故障，观察 BFF 的级联影响范围。 |
| **缓解** | 为每个下游调用设置独立超时和熔断器；实现部分降级，下游不可用时返回缓存数据或默认值；BFF 层水平扩展，避免单点；前端实现 BFF 旁路，关键数据可直连后端服务。 |
| **能力ID** | `CAP-BFF-SPOF` |

#### 4.1.2 GraphQL N+1 查询问题

| 维度 | 内容 |
|------|------|
| **现象** | GraphQL 查询中，解析嵌套关系时对每个父对象执行一次数据库查询。例如查询 100 个用户的订单，可能触发 1 次用户查询 + 100 次订单查询。弱网环境下，这 101 次查询的累计延迟让用户等待数秒甚至超时。 |
| **触发条件** | GraphQL Schema 中存在嵌套关系且 resolver 未做批量优化；查询深度和广度不受限，客户端可构造高嵌套查询；未使用 DataLoader 等批处理工具。 |
| **检测** | 监控单次 GraphQL 查询触发的数据库查询数量；使用 APM 工具追踪 resolver 执行次数；设置查询复杂度分析，标记超过阈值的查询。 |
| **缓解** | 使用 DataLoader 实现批量加载和缓存；设置查询深度限制和复杂度上限；对高频查询模式实现预编译和缓存；考虑 REST + BFF 聚合替代过度灵活的 GraphQL。 |
| **能力ID** | `CAP-GQL-N1` |

#### 4.1.3 聚合层增加额外延迟

| 维度 | 内容 |
|------|------|
| **现象** | BFF 聚合多个下游服务的数据，即使各服务单独响应都很快（如各 50ms），串行聚合后总延迟为各服务之和（如 5 个服务 = 250ms）。在弱网下用户已等待较长时间，额外的聚合延迟可能导致超时。 |
| **触发条件** | 下游调用串行执行而非并行；下游服务数量多（>5 个）；未设置合理的超时和快速失败策略；聚合逻辑在主线程执行，阻塞其他请求。 |
| **检测** | 绘制 BFF 请求的瀑布图，识别串行调用链；统计聚合延迟占端到端延迟的比例；对比并行与串行调用的响应时间差异。 |
| **缓解** | 使用 Promise.all 或类似机制并行调用下游服务；设置每个下游调用的独立超时（如 200ms），超时快速失败返回部分数据；实现请求合并，将多次 BFF 调用合并为一次批量查询。 |
| **能力ID** | `CAP-AGG-LATENCY` |

## 五、容错与降级策略

### 5.1 重试与退避机制

#### 5.1.1 重试风暴导致雪崩

| 维度 | 内容 |
|------|------|
| **现象** | 当服务端出现短暂故障时，大量客户端同时重试，重试流量可能是正常流量的数倍，进一步压垮服务端，形成"故障 → 重试 → 更大故障"的恶性循环。弱网用户的请求本身成功率就低，频繁重试加剧了这个问题。 |
| **触发条件** | 客户端对所有失败请求无差别重试；无指数退避或退避参数过于激进；多个客户端在同一时刻触发重试（无抖动）；重试次数无上限或上限过高。 |
| **检测** | 监控服务端的请求量趋势，故障期间是否出现 N 倍放大；统计客户端重试率和重试成功率；分析重试请求的时间分布是否集中在故障时刻。 |
| **缓解** | 实现指数退避 + 随机抖动（如 `base * 2^attempt + random(0, jitter)`）；区分可重试错误（5xx、超时）和不可重试错误（4xx）；设置最大重试次数（通常 2-3 次）和总超时上限；引入断路器，连续失败后停止重试一段时间。 |
| **能力ID** | `CAP-RETRY-STORM` |

#### 5.1.2 指数退避参数选择不当

| 维度 | 内容 |
|------|------|
| **现象** | 退避基数过小（如 100ms），在服务端需要较长时间恢复时，客户端仍以高频重试加剧负载；退避基数过大（如 30s），服务端已恢复但客户端还在等待，造成不必要的长时间不可用。参数需根据服务端恢复时间动态调整，但大多数实现使用固定参数。 |
| **触发条件** | 使用固定退避参数未根据服务类型差异化；未设置退避上限（cap），高指数导致等待时间爆炸；退避参数未考虑弱网环境的高延迟特性。 |
| **检测** | 绘制退避时间曲线，检查是否在合理范围；对比不同退避参数下的恢复时间和重试负载；监控退避期间的用户体验指标（如页面白屏时间）。 |
| **缓解** | 退避基数设为服务端 P99 响应时间的 1-2 倍；退避上限（cap）设为用户可接受的最大等待时间（通常 10-30s）；添加随机抖动（±25%）避免重试同步；根据错误类型调整策略（如 503 用长退避，429 用 `Retry-After` 头）。 |
| **能力ID** | `CAP-BACKOFF-PARAMS` |

### 5.2 降级策略

#### 5.2.1 降级策略影响用户体验

| 维度 | 内容 |
|------|------|
| **现象** | 降级策略过于粗暴（如直接显示错误页面、返回空白内容），用户在弱网下频繁触发降级后完全无法使用产品。或者降级策略过于保守（如从不降级），弱网用户长时间等待最终超时，体验更差。 |
| **触发条件** | 降级策略是全局开关，未区分功能优先级；降级阈值设置不合理（过低导致频繁降级，过高导致降级无效）；降级后返回的内容对用户无实际价值。 |
| **检测** | 统计降级触发频率和用户在降级状态下的行为（如跳出率）；A/B 测试不同降级策略对用户留存的影响；收集用户对降级体验的反馈。 |
| **缓解** | 分级降级：核心功能保留完整版，非核心功能返回简化版或缓存版本；降级 UI 提供明确的解释和预期恢复时间；实现渐进式降级，先减少数据量，再减少功能，最后才显示错误。 |
| **能力ID** | `CAP-GRACEFUL-DEGRADE` |

## 六、Service Worker 的隐性坑

### 6.1 SW 缓存更新不及时

| 维度 | 内容 |
|------|------|
| **现象** | Service Worker 的更新遵循"等待"策略：新的 SW 安装后不会立即激活，需等待所有使用旧 SW 的页面关闭。用户可能长时间（数小时甚至数天）使用旧版本，看到过期内容或存在已修复的 bug。弱网用户尤其依赖 SW 缓存，更新不及时的影响更大。 |
| **触发条件** | SW 文件未使用内容哈希，浏览器认为 SW 未变化而跳过更新；`install` 事件中的缓存预加载耗时过长；用户打开多个标签页，旧 SW 一直在"等待"状态。 |
| **检测** | 通过 `navigator.serviceWorker.controller` 检查当前活跃的 SW 版本；监听 `updatefound` 和 `statechange` 事件记录更新过程；对比 SW 缓存内容与服务端最新版本。 |
| **缓解** | 在 `install` 事件中调用 `self.skipWaiting()` 强制激活新 SW；配合 `clients.claim()` 让新 SW 立即接管所有页面；SW 文件名包含版本哈希，确保每次部署触发更新；实现"更新提示"UI，引导用户刷新页面获取最新版本。 |
| **能力ID** | `CAP-SW-STALE` |

### 6.2 SW 作用域限制

| 维度 | 内容 |
|------|------|
| **现象** | SW 的作用域默认为其所在路径及其子路径。若 SW 文件放在 `/app/sw.js`，则无法控制 `/api/` 或 `/static/` 的请求。开发者期望 SW 拦截所有请求，但实际上大部分请求绕过了 SW，缓存策略形同虚设。 |
| **触发条件** | SW 文件放置在子目录而非根目录；未通过 `Service-Worker-Allowed` 头扩展作用域；多应用共享域名时，各应用的 SW 作用域互相冲突。 |
| **检测** | 检查 `navigator.serviceWorker.controller` 是否为预期的 SW；使用 DevTools Application 面板查看 SW 的实际作用域；测试各路径的请求是否经过 SW 的 fetch 事件处理器。 |
| **缓解** | 将 SW 文件部署在域名根目录（`/sw.js`）；通过 `Service-Worker-Allowed: /` 响应头扩展作用域；注册时显式指定 `scope: '/'`；文档中明确标注 SW 的实际作用域范围。 |
| **能力ID** | `CAP-SW-SCOPE` |

### 6.3 SW 注册失败静默无告警

| 维度 | 内容 |
|------|------|
| **现象** | SW 注册失败时（如 HTTPS 证书问题、文件路径错误、MIME 类型不对），`navigator.serviceWorker.register()` 返回 rejected promise，但很多代码未处理该异常，导致 SW 静默失效。用户和运维人员毫不知情，缓存策略和离线能力全部失效。 |
| **触发条件** | 页面未通过 HTTPS 访问（SW 强制要求安全上下文）；SW 文件路径 404 或 MIME 类型非 `application/javascript`；浏览器不支持 SW 但未做特性检测；SW 脚本本身有语法错误。 |
| **检测** | 在 SW 注册代码中添加 `.catch()` 并上报错误；定期检查 SW 注册状态（`navigator.serviceWorker.getRegistrations()`）；监控前端错误上报中 SW 相关的异常。 |
| **缓解** | 注册 SW 时添加完整的错误处理和上报逻辑；实现 SW 健康检查端点，定期验证 SW 状态；在构建流程中验证 SW 文件的语法和路径；为不支持 SW 的环境提供降级方案（如 AppCache 或纯网络请求）。 |
| **能力ID** | `CAP-SW-REGISTER` |

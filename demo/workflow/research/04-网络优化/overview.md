# 网络优化：弱网环境下的资源加载策略

> 弱网不是异常，是常态。移动端、地铁、电梯、偏远地区——用户随时可能跌入低带宽、高延迟、高丢包的网络环境。优化策略的目标不是让弱网变快，而是让弱网下依然可用。

## 网络请求链路全景

```
DNS解析 → TCP/UDP连接 → HTTP请求 → 缓存决策 → 响应压缩 → 容错降级 → 离线兜底
  [A7]      [A18/A19]      [A21]      [A6/A7]      [A20]       [A22]       [A23]
```

---

## 1. DNS 解析：缩短第一跳

DNS 是请求链路的第一环，弱网下一次 DNS 查询可能耗费数百毫秒甚至超时。

- **DNS 预解析**：通过 `<link rel="dns-prefetch">` 提前解析关键域名，减少用户触发请求时的等待
- **DNS 智能解析**：CDN 根据用户地理位置、网络状况返回最优边缘节点 IP，避免跨区域解析导致的高延迟 [A7-CDN]
- **减少域名数量**：每多一个域名就多一次 DNS 查询，合并资源域名降低解析开销

> **弱网要点**：DNS 查询在弱网下极易超时。减少域名依赖 + 预解析是最低成本的优化手段。

---

## 2. 连接建立：减少握手成本

TCP 三次握手 + TLS 握手在弱网下可能耗费 1-3 秒，这是用户感知到的"白屏时间"的主要来源。

### TCP 连接优化 [A18-HTTP/2]

- **连接复用**：HTTP/2 在单个 TCP 连接上实现多流多路复用，避免为每个资源建立新连接
- **头部压缩（HPACK）**：通过静态表 + 霍夫曼编码压缩重复头部，弱网下节省显著带宽
- **服务端推送**：服务端主动推送关键资源（CSS/JS），省去客户端一轮请求-响应的往返时间

### UDP 连接优化 [A19-HTTP/3 QUIC]

- **0-RTT 连接建立**：QUIC 基于 UDP，首次连接 1-RTT，后续连接可 0-RTT，大幅缩短弱网下的连接耗时
- **队头阻塞消除**：TCP 层丢包会阻塞所有流，QUIC 在传输层实现独立流控，单个流的丢包不影响其他流
- **连接迁移**：网络切换（Wi-Fi ↔ 4G）时 QUIC 连接不中断，避免重新握手

> **弱网要点**：弱网下优先启用 HTTP/3（QUIC）。队头阻塞消除对高丢包场景的体验提升最为显著。

---

## 3. HTTP 请求：减少请求次数与体积

每一次请求都是弱网下的成本。减少请求数量和单次请求的复杂度是核心策略。

### 接口聚合 [A21-BFF]

- **BFF 聚合层**：在服务端将多个下游接口聚合为一个请求返回，客户端一次请求获取完整数据
- **GraphQL 聚合**：客户端精确声明所需字段，服务端按需返回，避免过度传输
- **接口编排**：将有依赖关系的串行请求在服务端编排为并行执行，减少总耗时

> **弱网要点**：弱网下每减少一次 RTT 都是巨大收益。BFF 聚合将 N 次请求压缩为 1 次，是最直接的弱网优化手段。

---

## 4. 缓存策略：避免重复请求

缓存是弱网优化中投入产出比最高的策略——命中缓存意味着零网络开销。

### HTTP 缓存 [A6-HTTP缓存]

- **强缓存**：`Cache-Control: max-age=31536000` 配合内容哈希文件名，静态资源直接从本地缓存读取，完全不发请求
- **协商缓存**：`ETag` / `Last-Modified` 配合 `If-None-Match` / `If-Modified-Since`，资源未变化时返回 304，避免重复传输
- **stale-while-revalidate**：先返回过期缓存保证可用性，后台静默更新，用户无感知

### CDN 缓存 [A7-CDN]

- **边缘节点缓存**：资源缓存在离用户最近的 CDN 节点，物理距离缩短 = 延迟降低
- **s-maxage 独立控制**：通过 `s-maxage` 单独控制 CDN 缓存时长，与浏览器缓存策略解耦
- **缓存分层**：浏览器缓存 → CDN 边缘 → CDN 中间层 → 源站，层层拦截减少回源

> **弱网要点**：强缓存是弱网的"免死金牌"。对于静态资源，宁可缓存时间长、文件名带哈希，也不要弱网下重新请求。

---

## 5. 传输压缩：缩小响应体积

在带宽受限的弱网环境下，减少传输体积等同于提升加载速度。

### 压缩策略 [A20-压缩]

| 压缩算法 | 压缩率 | CPU 开销 | 适用场景 |
|---------|-------|---------|---------|
| gzip | 中等 | 低 | 通用兼容，作为兜底 |
| brotli | 高（比 gzip 提升 15-25%） | 较高 | 预压缩静态资源，配合 CDN |

- **静态资源预压缩**：构建时生成 `.br` / `.gz` 文件，CDN 直接返回，避免实时压缩的 CPU 开销
- **动态内容实时压缩**：API 响应使用 gzip 实时压缩（CPU/带宽 trade-off 偏向带宽节省）
- **图片格式优化**：WebP/AVIF 替代 JPEG/PNG，同质量下体积减少 30-50%

> **弱网要点**：弱网下带宽是稀缺资源。Brotli 预压缩 + 图片格式优化的组合，能在不增加 CPU 负担的前提下显著减少传输体积。

---

## 6. 容错与降级：失败不是终点

弱网的本质特征是**不稳定**——请求可能超时、中断、返回错误。健壮的容错机制决定用户体验的下限。

### 重试与降级 [A22-重试降级]

- **指数退避重试**：请求失败后按 1s → 2s → 4s → 8s 的间隔重试，避免在弱网下疯狂重试加剧网络拥塞
- **Fallback 缓存**：网络请求失败时，降级返回上次成功的缓存数据，保证界面有内容而非白屏
- **优雅降级（Graceful Degradation）**：
  - 高清图 → 低质量占位图 → 纯色块
  - 实时数据 → 本地缓存数据 → 默认值
  - 富交互 → 简化交互 → 只读模式

> **弱网要点**：弱网优化的终极目标不是"快"，而是"不崩"。降级策略让用户在任何网络状态下都能看到有意义的内容。

---

## 7. 离线能力：超越网络限制

当网络彻底不可用时，离线能力让用户依然能使用核心功能。

### Service Worker [A23-Service Worker]

- **SW 生命周期**：注册 → 安装（预缓存关键资源）→ 激活 → 拦截请求，形成独立于网络的缓存层
- **Cache API**：程序化控制缓存的读写，实现比 HTTP 缓存更精细的策略
- **Workbox 策略**：
  - **Cache First**：优先读缓存，适合静态资源（JS/CSS/字体）
  - **Network First**：优先走网络，失败回退缓存，适合 API 数据
  - **Stale While Revalidate**：先返回缓存，后台更新，适合频繁变动但不实时关键的资源
  - **Network Only / Cache Only**：特定场景的极端策略

```
请求 → Service Worker 拦截
         ├─ Cache First  → 缓存命中？→ 返回缓存 : → 网络请求 → 缓存响应 → 返回
         ├─ Network First → 网络请求 → 成功？→ 缓存响应 → 返回 : → 返回缓存
         └─ SWR           → 返回缓存（同时后台更新缓存）
```

> **弱网要点**：Service Worker 是弱网优化的"最后一道防线"。配合 Workbox 的预缓存 + 运行时缓存策略，能让应用在完全离线状态下依然可用。

## 策略优先级（投入产出比排序）

| 优先级 | 策略 | 能力 | 收益 |
|-------|------|------|------|
| P0 | 强缓存 + 内容哈希 | A6 | 二次访问零网络开销 |
| P0 | 接口聚合 | A21 | 请求数量级减少 |
| P1 | CDN + 边缘缓存 | A7 | 物理距离缩短 |
| P1 | 传输压缩 | A20 | 带宽占用降低 |
| P1 | 重试降级 | A22 | 体验下限兜底 |
| P2 | HTTP/3 QUIC | A19 | 连接性能质变 |
| P2 | Service Worker 离线 | A23 | 完全离线可用 |
| P3 | HTTP/2 推送 | A18 | 减少往返 |

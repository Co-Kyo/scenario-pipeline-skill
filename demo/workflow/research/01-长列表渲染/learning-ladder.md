# 长列表渲染 — 学习阶梯

> 跟着阶梯走，每步有具体任务，做到才算过关。
> 不需要从头到尾读完 overview/experiment/trade-offs，阶梯会告诉你什么时候去看什么。

## 你需要什么基础
- 了解 HTML/CSS/JS 基本概念
- 知道 DOM 是什么、浏览器会把页面渲染出来
- 用过 React 或 Vue 的列表渲染（`v-for` / `.map()`）

## 阶梯总览
- **阶段一：理解渲染瓶颈**（对应能力 A1 渲染管线 + A2 事件循环）
- **阶段二：掌握虚拟列表核心算法**（对应能力 A5 虚拟列表）
- **阶段三：补齐可见性检测与 CSS 隔离**（对应能力 A9 IntersectionObserver + A13 CSS contain）
- **阶段四：综合运用——场景落地**（全部能力联动）

## 阶段一：理解渲染瓶颈

### 你将理解什么
为什么万级数据直接渲染会卡？瓶颈在浏览器渲染管线的哪个环节？

### Step 1：画出渲染管线，用实验感受瓶颈
**做**：打开 `01-长列表渲染/overview.md`，找到"链路全景图"部分，用纸笔画出 CRP 流程。
**然后动手**：打开 `experiment/src/index.html`，保持「全量渲染」模式，数据量设为 10000。观察「DOM 节点」和「渲染耗时」。
**你会看到什么**：DOM 节点 = 10,000，渲染耗时 >100ms（远超 16ms 帧预算）。切换到「虚拟列表」后，DOM 降到 ~30-50，渲染耗时 <5ms。
**这说明了什么**：减少 DOM 节点数 = 减少 Layout 工作量 = 提升帧率。这就是虚拟列表的核心价值。
**接下来去哪**：带着"如何减少 DOM 节点"这个问题，进入 Step 2。
**做到才算过**：能不看图手画 CRP 流程，并在实验中观察到全量渲染 vs 虚拟列表的 DOM 数量和帧率差异。

### Step 2：理解事件循环与帧预算
**做**：打开 `01-长列表渲染/overview.md`，找到"事件循环"节点。理解 16ms 帧预算、微任务 vs 宏任务 vs rAF。
**然后动手**：在实验中点击「▶ 自动滚动测试」，全量渲染模式下观察 FPS 图表——FPS 会跌到 <30，出现红色低帧标记。切到虚拟列表后重试，FPS 保持 >55。
**你会看到什么**：全量渲染滚动时帧率骤降，因为每个宏任务 >16ms。虚拟列表只更新少量 DOM，帧率稳定。
**这说明了什么**：不只是 DOM 数量的问题，JS 执行时间也必须控制在帧预算内。
**接下来去哪**：现在你知道了瓶颈，去看 `edge-cases.md` 的 E2（强制同步布局）和 E3（微任务风暴）。
**做到才算过**：能解释为什么 `setTimeout(cb, 0)` 不会立即执行，以及 rAF 的执行时机。

### 阶段一过关标准
- [ ] 能手画 CRP 流程图并标注每步的性能代价
- [ ] 能说出 16ms 帧预算的含义和超时后果
- [ ] 能区分微任务、宏任务、rAF 的执行顺序
- 做不到？→ 回到 `overview.md` 重新阅读 A1 和 A2 节点

## 阶段二：掌握虚拟列表核心算法

### 你将理解什么
虚拟列表如何只渲染可视区域的 DOM？动态高度怎么处理？DOM 池复用是什么？

### Step 3：理解窗口化裁剪原理
**做**：阅读 `01-长列表渲染/overview.md` 的"虚拟列表算法"节点。理解 scrollTop→起始索引映射 + 缓冲区的概念。
**然后动手**：在实验中切换到「虚拟列表」模式，将数据量拉到 100,000。观察「DOM 节点」始终保持 ~30-50，「渲染耗时」~2-5ms，不随数据量增长。
**你会看到什么**：固定高度时 O(1) 定位，渲染开销与数据量无关——10 万条和 1 千条的渲染耗时几乎一样。
**这说明了什么**：虚拟列表的核心是"只渲染看得见的"，用计算换 DOM。
**接下来去哪**：去看 `trade-offs.md` 的"固定高度 vs 动态高度"对比。
**做到才算过**：能解释给定 scrollTop、itemHeight、bufferSize 时，如何计算 startIndex 和 endIndex。能在实验中验证 100K 数据下虚拟列表的渲染耗时不变。

### Step 4：理解动态高度的坑
**做**：阅读 `edge-cases.md` 的 E1（动态高度滚动跳动）和 E5（快速滚动白屏）。
**你会看到什么**：项高度渲染后才确定 → 滚动位置偏移；缓冲区太小 → 短暂空白。
**这说明了什么**：动态高度是虚拟列表最难的部分，需要预估+修正的两阶段策略。
**接下来去哪**：进入阶段三，学习 IntersectionObserver 作为可见性检测的替代方案。
**做到才算过**：能说出"预估高度→渲染→测量真实高度→修正偏移"的完整流程。

### 阶段二过关标准
- [ ] 能画出虚拟列表的 DOM 结构示意图（可视区 + 缓冲区）
- [ ] 能解释固定高度 O(1) 定位 vs 动态高度前缀和+二分的区别
- [ ] 能说出动态高度滚动跳动的根因和修复策略
- 做不到？→ 回到 `overview.md` A5 节点 + `trade-offs.md` 重读

## 阶段三：补齐可见性检测与 CSS 隔离

### 你将理解什么
IntersectionObserver 如何做懒加载？CSS contain 如何隔离子树减少布局计算？

### Step 5：用 IntersectionObserver 实现懒加载
**做**：阅读 `01-长列表渲染/overview.md` 的 A9 节点。尝试写一个简单的图片懒加载：用 IO 监听图片进入视口，进入后设置 `src`。
**你会看到什么**：IO 是异步的，不阻塞主线程；rootMargin 可以控制提前加载距离。
**这说明了什么**：IO 是虚拟列表和懒加载的通用基础设施。
**接下来去哪**：阅读 `edge-cases.md` 的 E4（懒加载闪烁）和 E8（Observer 泄漏）。
**做到才算过**：能用 IntersectionObserver 写一个图片懒加载 demo，解释 rootMargin 的作用。

### Step 6：理解 CSS contain 与合成层
**做**：阅读 `01-长列表渲染/overview.md` 的 A13 节点 + `trade-offs.md` 的 contain 对比。
**你会看到什么**：`contain: layout paint` 允许浏览器跳过子树的布局和绘制；`will-change` 提升合成层但消耗 GPU 内存。
**这说明了什么**：CSS 手段是免费的性能优化，但过度使用会适得其反（层爆炸）。
**接下来去哪**：进入阶段四，把所有能力串起来落地。
**做到才算过**：能解释 `contain: content` vs `contain: strict` 的区别，以及 `will-change` 的利弊。

### 阶段三过关标准
- [ ] 能用 IntersectionObserver 实现懒加载
- [ ] 能解释 rootMargin 的用途
- [ ] 能说出 contain 属性的三种取值及适用场景
- 做不到？→ 回到 `overview.md` A9/A13 节点 + `edge-cases.md` 重读

## 阶段四：综合运用——场景落地

### 你将理解什么
社交动态流和聊天记录两种场景，虚拟列表的实现有什么差异？

### Step 7：对比两条技术路线
**做**：阅读 `trade-offs.md` 全文。理解固定高度路线（社交动态流推荐）vs 动态高度+前缀和路线（聊天记录推荐）的差异。
**你会看到什么**：没有万能方案，选路线取决于数据特征（高度是否可预知、滚动方向、数据量级）。
**这说明了什么**：技术选型的核心是"理解约束，匹配方案"。
**接下来去哪**：阅读 `references.md` 中的 react-window / vue-virtual-scroller 源码。
**做到才算过**：能说出社交动态流为什么适合固定高度、聊天记录为什么需要动态高度。

### Step 8：整合坑点防御
**做**：回顾 `edge-cases.md` 的 10 个坑点，对照 `overview.md` 的链路，标注每个坑点在链路中的位置。
**你会看到什么**：坑点分布在链路的每个节点，不是某一个环节能解决的。
**这说明了什么**：防御必须是全链路的——渲染优化 + 事件循环控制 + 懒加载 + CSS 隔离缺一不可。
**接下来去哪**：完成！你已经掌握了长列表渲染的完整知识体系。
**做到才算过**：能画出"坑点→链路节点→防御手段"的映射图。

- [ ] 能说出 10 个坑点中至少 5 个的防御手段
- [ ] 能完整描述从数据到像素的优化全链路
- 做不到？→ 回到 `trade-offs.md` + `edge-cases.md` 重读

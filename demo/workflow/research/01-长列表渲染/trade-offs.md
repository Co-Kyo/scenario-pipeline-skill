# 长列表渲染 — 方案对比

## 方案总览

| 维度 | 纯虚拟列表 | CSS 原生方案 | 分页/懒加载 | 混合方案 |
|------|-----------|-------------|------------|---------|
| DOM 节点数 | 恒定（overscan 窗口内） | 全量，但不可见项跳过渲染 | 单页少量 | 恒定 + CSS 辅助优化 |
| 滚动流畅度 | ★★★★★ | ★★★☆☆ | ★★★★★（页内） | ★★★★★ |
| 实现复杂度 | ★★★☆☆ | ★★☆☆☆ | ★★☆☆☆ | ★★★★☆ |
| 兼容性 | 通用框架均有库 | 需 Chromium 85+ | 通用 | 通用 |
| 搜索/定位 | 需自行实现 scrollToIndex | 天然支持 | 需跨页定位 | 需自行实现 |

## 1. 纯虚拟列表

### 技术栈选择

- **React**：`react-window`（轻量）、`react-virtuoso`（功能丰富，支持动态高度）
- **Vue**：`vue-virtual-scroller`（RecycleScroller）、`@tanstack/vue-virtual`

### 核心实现思路

仅渲染可视区域 ± overscan 范围内的 DOM 节点。通过滚动偏移量计算当前应显示的数据切片，容器用 `padding` 或绝对定位模拟完整滚动高度。

关键算法权衡：

- **高度策略**：固定高度 → O(1) 直接定位，性能最优；动态高度 → O(log n) 二分查找测量，需要先缓存每项高度或预估。**建议优先采用固定高度**，仅在业务强制要求时切换动态模式。
- **overscan 大小**：overscan=1~2 时 DOM 少、内存低，但快速滚动可能出现白屏闪烁；overscan=5~10 时白屏风险低，但 DOM 数量增加。经验值：**桌面端 5，移动端 3**。
- **DOM 复用**：复用已离开视口的 DOM 节点（类似 RecyclerView）可降低 GC 压力，但增加索引映射复杂度；每次销毁重建实现简单，但万级滚动时 GC 可能造成卡顿。`react-window` 默认不复用，`vue-virtual-scroller` 的 RecycleScroller 支持对象池复用。

### 适用场景

- 数据量 ≥ 1000 条，且用户需要连续滚动浏览（非跳跃式）
- 列表项结构相对统一（固定或有限种高度）
- 对首屏渲染性能有严格要求

### 牺牲点（Trade-off）

- **锚点定位**：`scrollToIndex` 需要自行维护滚动位置，动态高度下定位可能跳动
- **DOM 不可见**：列表项内的 `<input>`、`<video>` 等元素在离开视口后被销毁，表单状态丢失（React 中 key 的选择直接影响 state 保持——使用天然 ID 而非 index）
- **无障碍**：屏幕阅读器需要完整 DOM 树才能正确遍历，虚拟列表需额外处理 `aria-setsize` / `aria-posinset`
- **样式依赖**：部分 CSS 选择器（如 `:nth-child`）在虚拟化后语义失效

## 2. CSS 原生方案（contain + content-visibility）

### 技术栈选择

- **框架无关**，纯 CSS 属性：`content-visibility: auto`、`contain-intrinsic-size`、`contain: layout style paint`
- 可配合 `IntersectionObserver` 做精细化控制

### 核心实现思路

```css
.list-item {
  content-visibility: auto;
  contain-intrinsic-size: 0 48px; /* 预估高度，避免布局跳动 */
}
```

浏览器自动跳过视口外元素的渲染（样式计算、布局、绘制），但 DOM 节点依然存在于内存中。`contain` 属性将每个列表项隔离为独立的渲染层，避免单个项的变化触发整棵子树重排。

### 适用场景

- 列表项高度不统一且无法预估（动态内容、用户生成内容）
- 需要保留完整 DOM 结构（无障碍、SEO、锚点跳转）
- 数据量 1000~10000，浏览器兼容性要求不高的内部系统
- 快速原型，不想引入虚拟滚动库

### 牺牲点（Trade-off）

- **兼容性**：`content-visibility` 需要 Chromium 85+、Firefox 125+，Safari 尚未完整支持（截至 2025 年底）
- **性能天花板**：DOM 节点全量存在，10 万级数据时即使跳过渲染，内存占用和 DOM 查询开销仍然显著
- **`contain-intrinsic-size` 估算误差**：预估高度与实际不符时会出现滚动条跳动或空白间隙
- **无法解决 JS 执行开销**：列表项绑定的事件监听器、响应式数据绑定依然全量存在

## 3. 分页 / 懒加载

### 技术栈选择

- **传统分页**：UI 组件库自带分页器（Ant Design、Element Plus 等）
- **无限滚动**：`IntersectionObserver` 监听哨兵元素触底加载，或 `react-infinite-scroll-component` / `vue-infinite-scroll`
- **滚动锚点恢复**：`scrollRestoration: 'manual'` + `sessionStorage` 缓存位置

### 核心实现思路

每次只加载一页数据（通常 20~50 条），通过分页器或触底检测请求下一页。`IntersectionObserver` 的 `rootMargin` 可配置预加载距离：

- `rootMargin: '50px'`：低开销，但网络慢时用户可能看到加载态闪烁
- `rootMargin: '500px'`：提前加载更流畅，但可能浪费带宽（用户未必会继续滚动）

关键权衡：单 Observer 监听哨兵元素（简单高效）vs 多 Observer 独立追踪每个列表项（可做精细化懒加载，如图片延迟加载）。

### 适用场景

- 后端 API 已有分页接口（offset/cursor-based）
- 用户行为以"查找特定项"为主，而非连续浏览
- 移动端带宽敏感场景
- 管理后台、搜索结果页等传统 UI

### 牺牲点（Trade-off）

- **连续性断裂**：快速滚动时出现加载骨架屏，体验不如虚拟列表流畅
- **状态管理复杂**：无限滚动需要维护 loading/hasMore/error 状态，还需处理重复请求、竞态条件
- **搜索困难**：Ctrl+F 只能搜索当前已加载的数据，无法全量搜索
- **位置恢复**：浏览器前进/后退或刷新后，滚动位置恢复需要额外实现且不够可靠
- **数据一致性**：列表数据可能在分页期间发生变化（新增/删除），导致偏移量漂移

## 4. 混合方案（虚拟列表 + CSS containment）

### 技术栈选择

- 虚拟列表库（react-window / vue-virtual-scroller）作为滚动容器
- 列表项内部使用 `contain: layout style paint` + `will-change: transform` 优化渲染层
- 可选：`IntersectionObserver` 处理列表项内的图片/视频懒加载
- `requestAnimationFrame` 控制滚动事件节流（优于 `setInterval`，帧同步且后台自动暂停）

### 核心实现思路

虚拟列表负责"只渲染可见项"的宏观策略，CSS containment 负责微观层面的渲染隔离：

```
┌─ 虚拟列表容器（管理可视窗口） ─────────────┐
│  contain: strict                             │
│  ┌─ 列表项 ──────────────────────────────┐  │
│  │  contain: layout style paint           │  │
│  │  content-visibility: auto（可选）      │  │
│  │  子元素变化不触发兄弟重排               │  │
│  └────────────────────────────────────────┘  │
│  ┌─ 列表项 ──────────────────────────────┐  │
│  │  ...                                   │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

滚动事件处理的权衡（A30）：

- `requestAnimationFrame`：与浏览器帧同步，页面不可见时自动暂停，推荐
- `requestIdleCallback`：优先级最低，可能延迟执行，不适合滚动处理
- 时间驱动 vs 帧驱动：时间驱动（基于 `performance.now()` 计算插值）在不同刷新率屏幕上表现一致，适合精确动画

### 适用场景

- 数据量 10 万+，需要极致滚动性能
- 列表项内部结构复杂（嵌套组件、图片、交互控件）
- 需要支持动态高度 + 快速滚动
- 对帧率有严格要求（60fps）

### 牺牲点（Trade-off）

- **实现成本最高**：需要同时理解虚拟列表原理和 CSS containment 机制，调试工具链更复杂
- **Vue 场景的额外考量**：模板编译优化（静态提升、patchFlag）在虚拟列表中效果有限，因为列表项本身就是动态的；render function 虽然灵活但失去了编译期优化
- **过度优化风险**：`contain` 属性可能在某些边界条件下（如列表项需要 `overflow: visible` 超出自身边界）产生视觉截断
- **rAF 节流的副作用**：在低端设备上 rAF 回调本身的调度开销也需要考虑，极端情况下可能需要降级为被动滚动（`passive: true` + 时间阈值判断）

## 推荐方案

### 首选：方案 4（混合方案）

**理由**：

1. 虚拟列表是万级数据场景下唯一能保证 DOM 恒定的策略，CSS containment 是成本最低的渲染层隔离手段，两者互补而非替代
2. 在 React 和 Vue 生态中均有成熟库支持，不需要从零实现
3. 为后续扩展（动态高度、分组列表、瀑布流）保留了架构空间

### 降级路径

| 数据规模 | 推荐方案 |
|---------|---------|
| < 1000 条 | 方案 2（CSS 原生）—— 零依赖，快速上线 |
| 1000 ~ 10000 条 | 方案 1（纯虚拟列表）—— 够用，避免过度设计 |
| 10000+ 条 / 复杂列表项 | 方案 4（混合方案）—— 长期维护的正确投资 |
| 后端已有分页 / 搜索结果页 | 方案 3（分页/懒加载）—— 与后端协议对齐，减少前端复杂度 |

### 实施建议

1. **先测量再优化**：用 Chrome DevTools Performance 面板确认瓶颈在渲染层而非数据层
2. **固定高度优先**：除非业务强制，不要上来就做动态高度——这是虚拟列表最常见的复杂度陷阱
3. **Key 选择**（React）：使用数据中的天然唯一 ID，而非 `index`，否则列表项重排时状态会错乱
4. **IntersectionObserver 作为补充**：虚拟列表解决"渲染哪些"，IO 解决"何时触发副作用"（图片加载、曝光埋点），两者可共存

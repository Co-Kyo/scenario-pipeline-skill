# MVE: 虚拟列表 vs 全量渲染

## 环境基线
- 浏览器：Chrome 90+ / Firefox 89+ / Safari 15+
- 无需 Node.js 或构建工具，纯 HTML+JS
- 操作系统：无约束

## 一键启动
```bash
# 方式一：直接打开
open 01-长列表渲染/experiment/src/index.html

# 方式二：本地服务器（可选，用于 Performance 面板录制）
python3 -m http.server 8080
# 浏览器打开 http://localhost:8080/01-长列表渲染/experiment/src/index.html
```

## 验证检查点

### 检查点 1：全量渲染的 DOM 数量与渲染耗时
1. 保持默认「全量渲染」模式，数据量设为 10000
2. 观察左上角「DOM 节点」统计：应显示 **10,000**
3. 观察「渲染耗时」：应 **>100ms**（远超 16ms 帧预算）
4. **验证能力**：A1-浏览器渲染管线（DOM 节点过多导致 Layout 线性增长）

### 检查点 2：虚拟列表的 DOM 数量与渲染耗时
1. 点击「虚拟列表」按钮
2. 观察「DOM 节点」统计：应显示 **~30-50**（可视区域 + 缓冲区）
3. 观察「渲染耗时」：应 **<5ms**
4. **验证能力**：A5-虚拟列表算法（只渲染可视区域 DOM）

### 检查点 3：滚动帧率对比
1. 点击「▶ 自动滚动测试」，观察 FPS 图表
2. 全量渲染模式下：FPS 应 **<30**，出现红色低帧标记
3. 切换到虚拟列表，重置后再次滚动：FPS 应保持 **>55**
4. **验证能力**：A1-渲染管线 + A2-事件循环（帧预算 16ms）

### 检查点 4：数据量敏感性
1. 将数据量滑块拉到 100,000
2. 全量渲染：页面可能冻结数秒（Long Task >500ms）
3. 虚拟列表：渲染耗时几乎不变（~2-5ms），DOM 节点数不变
4. **验证能力**：A5-虚拟列表算法（O(1) 定位，渲染开销与数据量无关）

### 检查点 5：CSS contain 效果验证（需 DevTools）
1. 打开 Chrome DevTools → Rendering → 勾选「Paint Flashing」
2. 全量渲染模式滚动：整个列表区域持续闪烁（全量重绘）
3. 虚拟列表模式滚动：仅新出现的项闪烁（局部更新）
4. **验证能力**：A13-CSS contain（隔离渲染子树）

## 故障排除

- **页面卡死**：全量渲染 100K 数据可能冻结浏览器，耐心等待或直接刷新页面
- **FPS 统计不准**：确保浏览器标签页在前台，后台标签页会被节流
- **内存统计不显示**：`performance.memory` 仅 Chrome 支持，Firefox/Safari 显示为 `-`

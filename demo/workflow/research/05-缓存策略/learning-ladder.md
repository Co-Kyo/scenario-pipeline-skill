## 阶段一：HTTP 缓存协议

### 你将理解什么
浏览器如何判断"这个资源要不要重新请求"——强缓存和协商缓存的判定逻辑，以及 Cache-Control 各指令的工程含义。

### Step 1：强缓存验证
**做**：用 Chrome DevTools Network 面板访问任意设置了 `Cache-Control: max-age=3600` 的资源，观察首次和二次请求的 Status 列。
**你会看到什么**：首次 `200`，二次（刷新前）`200 (from disk cache)` 或 `200 (from memory cache)`。
**这说明了什么**：浏览器在 max-age 有效期内直接使用本地副本，不发网络请求——这就是强缓存。
**接下来去哪**：打开 `05-缓存策略/overview.md` 的"Layer 1: 浏览器缓存"章节，理解强缓存 vs 协商缓存的完整判定流程。
**做到才算过**：能画出"强缓存命中 → 协商缓存 → 网络请求"的三段判定流程图。

### Step 2：协商缓存验证
**做**：修改服务端响应，添加 `Cache-Control: no-cache` + `ETag: "v1"`，首次请求后再次访问，观察请求头和响应状态码。
**你会看到什么**：请求头自动带上 `If-None-Match: "v1"`，服务端返回 `304 Not Modified`。
**这说明了什么**：协商缓存不跳过网络请求，但通过条件请求避免重复传输——304 响应体为空。
**接下来去哪**：阅读 `05-缓存策略/edge-cases.md` 中"304 风暴"和"启发式缓存失控"两个坑点。
**做到才算过**：能解释 ETag vs Last-Modified 的精度差异，以及 no-cache vs no-store 的语义区别。

### 阶段一过关标准
- [ ] 能独立配置强缓存（max-age/s-maxage）和协商缓存（ETag/Last-Modified）
- [ ] 能用 Network 面板验证缓存命中状态
- [ ] 理解 stale-while-revalidate 的异步验证机制
做不到？→ 回到 `05-缓存策略/overview.md` 重读 Layer 1 章节

## 阶段二：CDN 边缘缓存

### 你将理解什么
HTTP 缓存不是只在浏览器——CDN 在全球边缘节点复制了同样的缓存逻辑，形成多级防线。

### Step 3：CDN 缓存命中验证
**做**：使用 `curl -sI "https://cdn.example.com/asset.js" | grep -iE "x-cache|age|cache-control"` 观察 CDN 响应头。
**你会看到什么**：`X-Cache: HIT` + `Age: 1234`（表示已在 CDN 缓存了 1234 秒）。
**这说明了什么**：CDN 用 `s-maxage` 独立控制边缘缓存 TTL，与浏览器的 `max-age` 互不影响。
**接下来去哪**：阅读 `05-缓存策略/overview.md` 的"Layer 2: CDN 边缘缓存"章节，理解 Edge PoP → Mid-tier → Origin Shield 三级架构。
**做到才算过**：能解释 s-maxage vs max-age 的分离控制策略，以及 Vary 头对缓存键的影响。

### Step 4：缓存失效策略设计
**做**：阅读 `05-缓存策略/trade-offs.md` 中"缓存失效策略"对比组。
**你会看到什么**：版本化 URL（contenthash）+ 长 TTL > 主动 Purge > 短 TTL 的优先级排序。
**这说明了什么**：缓存失效没有银弹，版本化 URL 是最安全的策略——URL 变了就是新资源，旧缓存自然过期。
**接下来去哪**：阅读 `05-缓存策略/edge-cases.md` 中"缓存穿透"和"回源风暴"两个坑点。
**做到才算过**：能为一个前端项目设计"静态资源长缓存 + HTML 短缓存 + 版本化 URL"的完整策略。

### 阶段二过关标准
- [ ] 理解 CDN 多级缓存架构和 Cache Key 生成规则
- [ ] 能设计 s-maxage/max-age 分离的缓存策略
- [ ] 知道 Origin Shield 的作用和适用场景
做不到？→ 回到 `05-缓存策略/trade-offs.md` 重读 CDN 相关对比组

## 阶段三：Service Worker 编程式缓存

### 你将理解什么
前两层是"被动缓存"——浏览器和 CDN 按 HTTP 头自动工作。Service Worker 让你"主动控制"缓存：决定缓存什么、何时更新、离线时返回什么。

### Step 5：Cache-First 策略实现
**做**：阅读 `05-缓存策略/experiment/` 中的 `sw.js`，理解 `caches.match()` + `fetch()` 的拦截逻辑。
**你会看到什么**：fetch 事件先查 Cache API，命中则直接返回，未命中才发网络请求并缓存响应。
**这说明了什么**：Cache-First 策略适合静态资源——首次慢，后续极快。但需要版本管理，否则用户永远看到旧版本。
**接下来去哪**：阅读 `05-缓存策略/edge-cases.md` 中"缓存版本污染"和"首次加载延迟"两个坑点。
**做到才算过**：能手写一个 Service Worker 实现 Cache-First + activate 事件清理旧缓存。

### Step 6：混合策略设计
**做**：阅读 `05-缓存策略/trade-offs.md` 中"Cache-First vs Network-First"和"三层协作模式"对比组。
**你会看到什么**：静态资源用 Cache-First，API 用 Network-First，高频页面用 Stale-While-Revalidate。
**这说明了什么**：没有万能策略，按资源类型选择不同策略才是正解。
**接下来去哪**：阅读 `05-缓存策略/overview.md` 的"三层协作实战"章节。
**做到才算过**：能为一个 SPA 项目设计完整的三层缓存方案（HTML/API/静态资源各用什么策略）。

### 阶段三过关标准
- [ ] 能手写 Service Worker 拦截 fetch 请求
- [ ] 理解 Cache API 与 HTTP 缓存的区别（Cache API 不遵循 HTTP 头）
- [ ] 能设计"浏览器缓存 + CDN + SW"三层协作方案
做不到？→ 回到 `05-缓存策略/experiment/` 运行实验，逐行调试 sw.js

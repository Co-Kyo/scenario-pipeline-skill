📖 skillpack monorepo 完全教学

这是一份面向初学者的教程。你不需要先了解任何关于 skillpack 的知识。 我们会从项目长什么样开始，一点点深入到每一步是怎么工作的。 每章都有"小白须知"帮你抓住重点。

1

欢迎 — 这个项目是做什么的？

小白须知

这个项目不是"产品"，而是一套开发工具。它的目标只有一个：把 TypeScript 代码变成 Markdown 文档。

skillpack 是一个"打包工具"——但它打包的不是 JavaScript，而是 LLM Skill。

所谓"Skill"，就是一组步骤（step），每个步骤告诉 LLM 要做什么事。比如一个"前端技术调研"Skill 包含：

  • Step 00：主题解析
  • Step 01：多维度探索（并行 3 个 Agent）
  • Step 02：分类整理
  • Step 03：深度研究
  • Step 04：报告生成

你用 TypeScript 定义这些步骤的"蓝图"（steps、依赖关系、每个步骤内部的调度图），然后 skillpack 帮你：校验、排序、生成最终的 Markdown 文件。

整个 monorepo 就是这套工具的源码 + 一个示例 Skill。

记住
  • skillpack = 把 TS 定义变成 MD 文档的打包器
  • monorepo = 工具源码（packages/）+ 示例（demos/）
  • 你读这份教程的目的：理解这个工具怎么用、怎么改

项目总览 — 一眼看懂目录结构

小白须知

整个项目只有 3 个顶层目录。记住：框架代码在 packages/，示例在 demos/，产物在 dist/。改代码永远在 packages/ 和 demos/ 里改。

打开项目根目录，你看到的是这样的结构：

monorepo-dev/
# ─ 根配置 ─
.gitignore
package.json              # npm workspaces 声明 + 统一脚本
tsconfig.json             # tsc -b 项目引用编排
README.md

# ─ 框架（4 个包）─
packages/
  skillpack-types/        #   类型 + 辅助函数（叶子节点）
  skillpack-common/       #   校验 + 拓扑排序 + 图遍历
  skillpack-build/        #   构建器 + CLI 入口
  skillpack-validate/     #   独立校验 CLI

# ─ 示例 ─
demos/
  skillpack.config.ts     #   配置（声明 skill 路径 + 输出目录）
  skill.ts                #   Skill 定义（5 个步骤）

# ─ 产物（自动生成，不提交）─
dist/                     # SKILL.md + processes/

# ─ 文档 ─
docs/                     # 架构图 + 你在看的这个页面

下面这张图用颜色区分了三层：框架层（蓝色系）、示例层（黄色/绿色）、输出层（红色）。箭头表示"谁用谁"。

📦 packages/4 个框架包，编译后生成 .js 给 demo 用。它们之间有严格的依赖顺序。
🎯 demos/示例项目，一套配置（config）+ 一套定义（skill）。可以看成"用户怎么用 skillpack"。
📄 dist/构建产物。每次 npm run demo 都会重新生成。gitignored，从不手动修改。
🔗 关系config 从 build 包 import defineConfig，CLI 读 config → 加载 skill → 调用 buildPipeline → 输出到 dist
记住
  • packages/ = 工具箱，demos/ = 使用示范，dist/ = 成果物
  • 所有源代码都在 packages/*/src/ 里，每一个包只导出一个入口（index.ts）

包依赖 — 四个库谁依赖谁？

小白须知

4 个包的关系是单向的：types → common → {build, validate}。不能反向依赖。这是整个项目的"宪法"。

每个包都有自己的 package.json 和 tsconfig.json。它们通过两种方式连接：

① tsconfig project references（构建时）：tsc -b 按引用顺序编译，保证 types 先编译完，common 才能编译，以此类推。

② npm workspace dependencies（运行时）：每个包的 package.json 声明 "skillpack-types": "*"，npm workspaces 自动把包链接到 node_modules。

📐 skillpack-types纯类型 + 纯函数。没有 node_modules 依赖，不依赖任何其他包。
⚙️ skillpack-common依赖 types。实现校验逻辑、Kahn 拓扑排序、图遍历。
🏗 skillpack-build依赖 types + common。打包器 + Markdown 渲染器 + CLI 入口。
🔍 skillpack-validate依赖 types + common。独立的校验 CLI。

构建顺序：

tsc -b 执行顺序（tsconfig references 决定）
Phase 1: skillpack-types        // 无依赖，先编译
Phase 2: skillpack-common       // 等 types 编译完
Phase 3: skillpack-build        // 等 common 编译完
         skillpack-validate     // 和 build 并行（都只依赖 common）
（注意：demos/ 不在 tsconfig references 中 — 它用 tsx 直接跑源码）
记住
  • types 是地基，common 是承重墙，build 和 validate 是一楼的两个房间
  • 改 types → common 和 build 都需要 rebuild；改 demos/ → 零构建，直接跑

定义 Skill — 从 createSkill 开始

小白须知

如果你用过 Vue 3，createSkill 就像 createApp：你传一个对象进去，得到一个完整的 Skill 定义出来。整个文件只 export 一个东西。

示例项目里的 skill.ts 是 Skill 定义的"样板房"。它演示了所有功能：

demos/skill.ts
import { createSkill, agent, batch, mapWork, edge }
  from 'skillpack-types';

// ... 5 个 step 定义 ...

export const skill = createSkill({
  name: 'tech-research',
  title: '前端技术调研',
  description: '针对一个前端技术主题...',
  steps: [
    stepParse,       // 主题解析（单 Agent）
    stepExplore,     // 多维度探索（Batch + 收敛 + 门禁）
    stepOrganize,    // 分类整理（两个 Agent 串行）
    stepDeepDive,    // 深度研究（两个 Agent DAG）
    stepReport,      // 报告生成（Map）
  ],
});

下面这张图展示了 createSkill 的完整结构——从外层的 wrapper，到中间每个步骤的字段，再到内部调度图：

📦 createSkill wrapper接收 { name, title, description, steps[] }，返回同一个对象。纯类型辅助，零运行时开销。
📋 steps 数组StepDefinition 对象的数组。顺序不重要——真正的顺序由 dependsOn 字段 + Kahn 排序决定。
🗺 每个 step 里的 schedule一个调度图（nodes + edges + entry），描述这个步骤内部 Agent 们怎么配合。
🔔 生命周期钩子barrier（检查点）/ reuse（复用）/ degrade（降级）——不是每个 step 都需要。
记住
  • 一个 skill.ts = 一个 export（createSkill 的结果）
  • steps 数组里的顺序不重要，重要的是每个 step 的 dependsOn 字段

步骤结构 — StepDefinition 的 11 个字段

小白须知

每个 StepDefinition 就像一张"设计图纸"——它完整描述了 LLM 在这一步要读什么文件、做什么事、产出什么结果、以及异常怎么办。

以 demo skill 里的第三步"深度研究"为例：

demos/skill.ts — stepDeepDive
const stepDeepDive: StepDefinition = {
  id:          'deep-dive',
  title:       '深度研究',
  description: '核心概念研究 → 实践模式研究',
  dependsOn:   ['organize'],
  schedule:    { /* 见下章 */ },
  reads:  [{ path: '{workDir}/.meta/organized.json', required: true }],
  writes: [{ path: '{workDir}/.meta/dive-concepts.json' }],
  reuse: [{ checkFile: '{workDir}/.meta/dive-concepts.json', ... }],
  degrade: { maxRetries: 2, onDegrade: 'continue' },
  barrier: { checkItems: ['核心概念发现数', ...], ... },
};

字段速查表：

字段类型必需?说明
idstring必步骤唯一标识，kebab-case
titlestring必人类可读的标题
descriptionstring必一句话描述做什么
dependsOnstring[]必前置步骤的 id 列表
scheduleScheduleGraph必步骤内部的调度图（下一章）
readsFileRef[]选步骤需要读哪些文件
writesFileRef[]必步骤产出哪些文件
barrierBarrierDef选检查点 — 执行到这一步会暂停让用户确认
reuseReuseRule[]选增量复用 — 如果文件已存在就跳过
degradeDegradeProtocol选降级 — 失败后重试或降级处理
pluginsstring[]选条件性加载的插件
记住
  • id + dependsOn + schedule + writes 是必须的，其余按需使用
  • dependsOn 决定了步骤的先后顺序（构建时用 Kahn 算法排序）

调度图 — 步骤内部的"工作流引擎"

小白须知

如果说 dependsOn 是"步骤之间怎么串"，那 schedule 就是"一个步骤内部怎么干"。它用图（graph）来描述工作流：有节点（Agent），有边（谁先谁后）。

调度图的核心只有 3 个概念：

节点（nodes）：三种类型——agent（单兵）、batch（组团）、map（流水线）。

边（edges）：有向连接，from → to，可附带条件（condition）实现条件路由。

入口（entry）：从哪个节点开始执行。

ScheduleGraph 的结构
{
  entry: 'dive-concepts',           // 从这里开始
  nodes: [
    { kind: 'agent', id: 'dive-concepts', role: '核心概念研究员', ... },
    { kind: 'agent', id: 'dive-patterns', role: '实践模式研究员', ... },
  ],
  edges: [
    { from: 'dive-concepts', to: 'dive-patterns' },
  ],
}

下图是三种节点类型的完整参考，包括每个字段的含义和实际代码示例：

🤖 Agent最简单的节点。一个 Agent + 一个 task 提示词。适合独立任务。
🌲 Batch并行 fan-out：多个 branches 同时跑，可选的 converge 整合结果，可选的 gate 质量门禁。
🔄 Map滚动窗口：从数据源（itemFrom）取条目，对每条执行同一个 worker。maxConcurrency 控制并发。
🧩 组合模式串行链（agent→edge→agent）、扇出+收敛、滚动窗口、带门禁的批量——4 种常见组合。
记住
  • 三种节点 = Agent（单兵）/ Batch（组团）/ Map（流水线），覆盖所有调度模式
  • 边可以加 condition（条件路由），比如"如果质量检测不通过就跳到降级处理"

三种节点 — 从示例看懂怎么用

小白须知

这一章从 demo skill 里取出三个真实的例子，分别展示 agent、batch、map 在实际项目中长什么样。

示例 1：单 Agent（Step 00）

stepParse 的调度图 — 最简单的形式
schedule: {
  nodes: [ agent({ id: 'parse', label: '主题解析', role: '调研框架构建者', task: `...` }) ],
  edges: [],
  entry: 'parse',
}

就是单个节点，没有边。适合简单的"给一个任务让 LLM 做"。

示例 2：Batch + 收敛 + 门禁（Step 01）

stepExplore 的调度图 — 最复杂的例子
schedule: {
  nodes: [
    batch('explore-batch', '多维度并行探索', [
      agent({ id: 'explore-docs', role: '文档分析师', ... }),
      agent({ id: 'explore-practice', role: '实践分析师', ... }),
      agent({ id: 'explore-ecosystem', role: '生态分析师', ... }),
    ], {
      converge: agent({ id: 'explore-converge', role: '收敛者', ... }),
      gate: {
        rule: '3 agents completed or at most 1 degraded',
        onPass: 'converge',   // 门禁通过 → 启动收敛 Agent
        onFail: 'degrade',     // 不通过 → 降级
      },
    }),
  ],
}

3 个 Agent 并行调研文档、实践、生态。全部完成后（或最多 1 个失败），收敛者合并结果。门禁规则由你定义。

示例 3：Map — 滚动窗口（Step 04）

stepReport 的调度图 — 流水线模式
schedule: {
  nodes: [
    mapWork('report-map', '章节报告生成',
      '{workDir}/.meta/organized.json#categories',  // 数据来源
      agent({ id: 'report-worker', role: '报告撰写员', ... }),
      3  // 最大并发槽位
    ),
  ],
}

从 organized.json 里取出 categories 数组，对每个 category 启动一个 Agent 写报告。最多 3 个并发。

记住
  • Agent 最简单，Batch 最强大（并行 + 收敛 + 门禁），Map 适合数据驱动
  • 写 skill 时优先用 Agent 组合 + 边，不够用了再上 Batch/Map

配置 + CLI — 从文件到命令行的旅程

小白须知

写好了 skill.ts，怎么把它变成最终的 Markdown？这就是 skillpack.config.ts 和 CLI 的工作。

配置文件和 Vue 的 vite.config.ts 类似——它声明"我用哪个 Skill 文件"和"输出到哪里"：

demos/skillpack.config.ts
import { defineConfig } from 'skillpack-build';

export default defineConfig({
  skill: './skill.ts',      // ← 参数链接到 skill 定义
  outputDir: '../dist',    // ← 产物输出到根目录 dist/
});

CLI（cli.ts）读取这个配置，按以下步骤执行：

packages/skillpack-build/src/bin/cli.ts — 流程
1. 解析命令行参数
   // skillpack build skillpack.config.ts

2. 读取 skillpack.config.ts
   // → { skill: './skill.ts', outputDir: '../dist' }

3. 解析 skill 路径并动态 import
   // → resolve('./skill.ts') → import → { name, steps }

4. 合并 meta（config 可覆盖 skill 的 name/description）

5. 调用 buildPipeline(steps, outputDir, meta)
   // → 校验 → 拓扑排序 → 渲染 → 写文件

构建流水线 — 三阶段详解

小白须知

buildPipeline 是 skillpack 的"发动机"。它分三个阶段，像工厂流水线一样：先检查原料（校验），再排工序（排序），最后包装出厂（渲染）。

第一阶段：校验（Validate）

三步检查，任何一步出错就停止：

1. validateStep(step)           // 每个步骤字段是否完整
   // - id / title / description 非空
   // - schedule 有节点且 entry 存在
   // - writes 非空
   // - barrier 有 clarifyPrompt

2. validateDependencyRefs(steps) // dependsOn 是否指向真实 step id
   // 如果 stepB.dependsOn 写了 "stepC" 但不存在 → 报错

3. validateBarrierContinuity(steps) // barrier 连续性
   // 最后一个 barrier 之前的所有步骤都必须有 barrier

第二阶段：拓扑排序（Resolve）

核心算法：Kahn 算法。思路很简单：

  • 找到所有"没有前置依赖"的步骤（入度为 0）→ 排在最前面
  • 它们排完后，检查哪些步骤的依赖已经全部排完 → 再排它们
  • 重复直到所有步骤排完
  • 如果还有步骤没排但已经没有入度为 0 的了 → 循环依赖错误

排序完为每个步骤分配 seq 编号（00, 01, 02...），同时解析构建变量 {{stepSeq}} → "00"、{{stepId}} → "topic-parse"。

第三阶段：渲染（Render）

  • 对每个步骤调用 renderStep(step) → 生成 Markdown → 写入 dist/processes/{seq}-{id}.md
  • 调用 renderSkillMd(pipeline, meta) → 生成 dist/SKILL.md
  • renderStep 内部会遍历步骤的 schedule 图，根据节点类型（agent/batch/map）调用不同的渲染器
构建结果：5 个步骤的拓扑顺序
00: topic-parse        // 无依赖 → 最先
01: multi-dim-explore  // 依赖 topic-parse
02: organize           // 依赖 multi-dim-explore
03: deep-dive          // 依赖 organize
04: report             // 依赖 deep-dive → 最后
记住
  • 校验保证你不会产出错误文件
  • Kahn 排序保证步骤永远按正确顺序执行（即使你写错了 steps 数组顺序）
  • 渲染把 TypeScript 对象变成可读的 Markdown 文档

开发循环 — 怎么改代码、怎么跑

小白须知

日常开发有 3 种场景，对应 3 种不同长度的循环。记住：改 skill 不用构建，改框架需要构建。

Loop A — 改框架代码（最慢）

修改 packages/*/src/ 下的源码后，需要：

npm run build    // tsc -b 重新编译框架包（增量，只重编改了的包）
npm run typecheck // 可选，快速类型检查（tsc -b --noEmit）
npm run demo     // 重新生成 dist/

Loop B — 改 Skill 定义（最快，零构建）

修改 demos/skill.ts 或 demos/skillpack.config.ts 后：

npm run demo     // 够了！tsx 直接加载 .ts 源文件，无需编译

注意：框架包至少需要构建过一次（npm run build），否则 tsx 找不到编译后的 dist/*.js。

Loop C — 全量重建

npm run clean    // 删除所有 dist/ + tsbuildinfo
npm run build    // 完整编译（从头开始）
npm run demo     // 验证

命令速查：

命令实际执行什么时候用
npm run buildtsc -b改完 packages/*/src/ 后
npm run demotsx CLI → buildPipeline → dist/查看构建结果
npm run typechecktsc -b --noEmit快速检查类型错误
npm run cleanrm -rf packages/*/dist dist重建前清理
记住
  • Loop B 是你最常用的——改 skill.ts → npm run demo，秒级反馈
  • 只有改框架包（packages/）才需要 npm run build
  • npm run clean + build + demo 是最稳妥的"从头来一遍"

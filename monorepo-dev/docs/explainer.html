🔧 skillpack monorepo

一套用于定义和构建 LLM Markdown Skill 管线的开发框架。
本文档以 6 张结构图完整讲述 monorepo-dev 的架构、数据流、类型系统和开发操作。

1

全局鸟瞰 — 项目物理结构

给老板的话

这张图是你应该记住的"地图"。整个 monorepo 只有 18 个源文件，分三层：框架库（packages/）、示例项目（demos/）、构建产物（dist/）。你不需要记每个文件，只需要知道"改框架去 packages/，改技能去 demos/，看结果去 dist/"。

📦 Framework Layer4 个 skillpack 包，单向依赖链：types → common → {build, validate}
🎯 Demo Layerskillpack.config.ts（配置） + skill.ts（技能定义），通过 import 链接
📄 Output Layerdist/SKILL.md + processes/ 5 个步骤文件，全量由 buildPipeline 生成
🔗 跨层连接config import { defineConfig } from build; CLI 读取 config + skill → buildPipeline → 输出到 dist
2

包依赖拓扑 — 谁依赖谁

给老板的话

这是项目的"承重墙"。tsconfig 的 project references 保证了构建顺序总是 types → common → build+validate。npm workspaces 让每个包互相可见。记住：types 是基石，common 是逻辑核心，build/validate 是消费者。绝对不允许反向依赖。

tsconfig project referencestsc -b 按引用拓扑依次构建，只重建变更的包
npm workspace deps每个 package.json 声明 workspace: * 依赖，npm 自动 hoist
构建顺序Phase 1: types → Phase 2: common → Phase 3: build + validate（并行）
demo 特殊demos/ 不在 tsconfig references 中，通过 tsx 运行时加载 .ts 源文件
3

构建流水线 — 输入到输出的完整路径

给老板的话

这是整个 skillpack 的"发动机"。从 skillpack.config.ts 声明配置 → CLI 动态加载 → buildPipeline 三阶段处理（校验→拓扑排序→渲染）→ 最终写入磁盘。三次校验确保不会产出错误文件，Kahn 排序保证步骤永远按正确的依赖顺序执行。

📥 输入层skillpack.config.ts（配置） + skill.ts（技能定义），通过 CLI 动态 import 加载
🖥 CLI 层cli.ts 解析参数 → 读取 config → 解析 skill 路径 → 合并 meta → 调用 buildPipeline
✅ 校验阶段3 步：validateStep（字段完整性）→ validateDependencyRefs（引用完整性）→ validateBarrierContinuity（屏障连续性）
🔢 解析阶段Kahn 拓扑排序 → 分配 seq（00-04）→ 解析 {{stepSeq}}/{{stepId}} 构建变量
🎨 渲染阶段renderStep × N → processes/{seq}-{id}.md + renderSkillMd → SKILL.md
📁 输出层dist/ 目录下 6 个文件：1 个 SKILL.md + 5 个步骤文件

节点类型参考 — 三种调度原语

给老板的话

调度是 skill 的核心复杂度。我们只暴露 3 种节点：Agent（单兵）、Batch（组团并行）、Map（流水线）。用这 3 种原语 + 有向边，你可以组合出任意调度模式。这张图就是你的"API 速查表"——写 skill.ts 时对照着看。

🤖 AgentWork单 Agent 节点：id、label、role、task、timeout。用于独立任务
🌲 BatchWork扇出并行：branches[]（WC） + converge（可选收敛者） + gate（可选质量门禁）
🔄 MapWork滚动窗口：itemFrom 数据源 + worker 每条目执行 + maxConcurrency 并发槽位
🧩 组合模式串行链（agent→edge→agent）、扇出+收敛、滚动窗口、带门禁的批量

技能定义解剖 — createSkill 内部结构

给老板的话

这张图是你写 skill.ts 时的"参考图纸"。createSkill 包裹整个定义，StepDefinition 是每一步的蓝本。注意每个字段的 required/optional 标记——id/dependsOn/schedule/writes 是必须的，barrier/reuse/degrade 是按需的生命周期钩子。红色标注的就是最常见的错误来源。

📦 createSkill 包裹单 export：{ name, title, description, steps[] }，和 Vue 3 createApp 同构
📋 StepDefinition11 个字段：id(必) / title(必) / description(必) / dependsOn(必) / schedule(必) / reads / writes / barrier / reuse / degrade / plugins
🗺 ScheduleGraph每个步骤内部的调度子图：node[] + edges[] + entry，LangGraph 风格
🔔 Lifecycle hooksbarrier = 检查点 / reuse = 文件级增量跳过 / degrade = 失败降级

开发者循环 — 三种工作模式

给老板的话

这张图教你"如何在这个项目里干活"。有三种场景，对应三个不同长度的循环：改框架（最慢，需 rebuild）、改 skill（最快，零构建）、全量重建（最干净）。日常 90% 的工作在 Loop B 里——你只需改 skill.ts 然后 npm run demo，秒级反馈。

🛠 Loop A: 框架开发改 packages/*/src → npm run build → npm run demo → 验证 dist/
⚡ Loop B: Skill 开发改 demos/skill.ts → npm run demo（零构建！tsx 直接加载 .ts）→ 验证
🧹 Loop C: 全量重建npm run clean → npm run build → npm run demo → 从头到尾完整验证
📜 命令速查build（tsc -b）/ demo（CLI → config → skill → buildPipeline）/ typecheck（tsc -b --noEmit）/ clean（清所有产物）

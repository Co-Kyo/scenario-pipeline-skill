# 核心能力：原子能力图谱（Capability Graph）

---

## 一、提取方法

### Step 1：从分词结果中提取原子能力

对命题的分词标注（[通用内核] + [特化层]）进行逐层拆解：

```
命题："Vue3 虚拟列表：万级数据渲染"

分词标注：
  [通用内核]
    · 算法层：虚拟化算法（前缀和 vs 双游标）
    · 浏览器层：渲染管线与 DOM 节点数的关系
    · 内存层：DOM 节点回收与 GC 策略
    · 交互层：滚动锚定、惯性滚动

  [Vue 特化]
    · 框架层：Composition API 响应式联动
    · 生态层：@tanstack/vue-virtual

提取的原子能力：
  A1-浏览器渲染管线  A2-DOM节点生命周期  A3-V8 GC机制
  A4-IntersectionObserver  A5-虚拟化算法(前缀和/双游标)
  A6-事件循环与rAF调度
  [Vue特化] V1-Proxy响应式追踪  V2-Composition API作用域
```

### Step 2：原子能力命名规范

```
格式：<技术域>-<能力描述>
示例：
  ✓ A1-浏览器渲染管线（CRP→Layout→Paint→Composite）
  ✓ A5-虚拟化算法（前缀和/双游标/二分定位）
  ✗ 渲染（太模糊）
  ✗ 浏览器相关（无具体能力描述）
```

### Step 3：能力依赖关系

标注原子能力之间的 **前置依赖**：

```
A1-浏览器渲染管线
  依赖：（无，基础能力）

A2-DOM节点生命周期
  依赖：A1-浏览器渲染管线（理解 DOM 如何被渲染）

A3-V8 GC机制
  依赖：A2-DOM节点生命周期（理解 DOM 节点如何被回收）

A4-IntersectionObserver
  依赖：A1-浏览器渲染管线（理解观察的触发时机）
```

## 二、扇出度计算

**扇出度（Fan-out）**= 该原子能力被多少个命题共享。

### 计算规则

1. 对所有已提取的原子能力，统计它出现在多少个命题的能力集合中
2. 框架特化能力（V1, V2...）和通用能力（A1, A2...）分开统计
3. 同一能力在不同限定词变体中出现时，只计一次

### 扇出度等级

| 等级 | 扇出度 | 含义 |
|------|--------|------|
| 🔴 核心 | ≥ 50% 命题覆盖 | 战略高地，必须掌握 |
| 🟡 高价值 | 30-50% 命题覆盖 | 重要能力，优先学习 |
| 🟢 专项 | 10-30% 命题覆盖 | 特定场景需要 |
| ⚪ 低频 | < 10% 命题覆盖 | 按需学习 |

## 三、限定词注入分析

### 4.1 限定词作为能力注入器

限定词向命题注入框架特化原子能力集（见下表）：

| 限定词类型 | 注入的能力集 | 影响范围 |
|----------|------------|---------|
| 框架限定词（Vue/React/Angular） | 响应式系统、组件模型、生命周期管理、状态管理模式 | 改变特化层全部能力 |
| 构建工具限定词（Webpack/Vite/Rollup） | 模块解析策略、缓存机制、插件体系、HMR 实现 | 改变构建层能力 |
| 架构模式限定词（SPA/SSR/微前端） | 路由策略、渲染模式、沙箱机制、隔离方案 | 改变架构层能力 |
| 宽泛限定词（前端网络工程/渲染性能） | 无特定框架注入，命题完全由通用内核驱动 | 不改变能力组成 |

### 4.2 限定词对战略高地的影响

同一命题内核 + 不同限定词 = 战略高地优先级不同：

```
内核："虚拟列表"
限定词 = React → 注入 Fiber 调度能力 → "事件循环与异步调度"优先级上升
限定词 = Vue   → 注入 Proxy 响应式 → "响应式系统粒度控制"优先级上升
限定词 = 无    → 全部重量在通用内核 → 扇出度 = 实际战略价值
```

### 4.3 限定词耦合度评分

每个原子能力标注其被限定词绑定的程度：

| 耦合度 | 含义 | 示例 |
|--------|------|------|
| 1 | 完全框架无关，换任何框架都直接复用 | 浏览器渲染管线、V8 GC、HTTP 缓存 |
| 2 | 部分框架相关，但核心原理跨框架通用 | 事件循环（框架调度策略不同但底层相同） |
| 3 | 高度框架绑定，换框架需重新学习 | Vue Proxy 响应式、React Fiber 调度 |

# 核心能力：战略高地识别（Strategic Highground）

---

## 一、识别方法

### Step 1：计算战略价值分数

```
战略价值 = fanout.count × (1 / 限定词耦合度)
```

- **fanout.count**：该能力被多少个命题引用
- **限定词耦合度**：该能力被框架/工具绑定的程度（1=无关, 2=部分相关, 3=高度绑定）


### Step 2：排序与分级

| 等级 | 战略价值分 | 含义 |
|------|----------|------|
| 🏔️ 一级高地 | ≥ 4.0 | 必须攻克，解锁大面积命题 |
| ⛰️ 二级高地 | 2.0 - 3.9 | 重要补充，深化特定领域理解 |
| 🏕️ 三级营地 | 1.0 - 1.9 | 专项能力，按需学习 |
| ⚪ 基础设施 | < 1.0 | 工具/环境层，自动获得 |

### Step 3：验证高地独立性

检查高地之间是否存在依赖关系。如果高地 A 依赖高地 B，则 B 的实际战略价值 += A 的战略价值。

```
A1-浏览器渲染管线（价值 6.0）
├── A2-DOM生命周期（价值 5.0）← 依赖 A1
│   └── A3-V8 GC（价值 3.0）← 依赖 A2
└── A4-事件循环（价值 4.0）← 依赖 A1

A1 的实际价值 = 6.0 + 5.0 + 3.0 + 4.0 = 18.0（累积依赖）
```

---

## 二、限定词耦合度分析框架

### 3.1 通用高地 vs 特化高地

| 类型 | 限定词耦合度 | 特征 | 学习策略 |
|------|------------|------|---------|
| 通用高地 | 1 | 浏览器/JS引擎/协议层能力，框架无关 | 一次学习，终身复用 |
| 混合高地 | 2 | 核心原理通用，但实现细节框架相关 | 学原理，查文档 |
| 特化高地 | 3 | 高度绑定特定框架/工具 | 随框架版本更新需重新学习 |

### 3.2 限定词对高地优先级的影响

同一命题 + 不同限定词 = 高地排序不同：

```
命题内核："虚拟列表"

限定词 = React 时的高地排序：
  1. A1-浏览器渲染管线（通用高地，不变）
  2. R1-Fiber调度（特化高地，React 特有）
  3. A4-事件循环（混合高地，Fiber 调度依赖事件循环）

限定词 = Vue 时的高地排序：
  1. A1-浏览器渲染管线（通用高地，不变）
  2. V1-Proxy响应式（特化高地，Vue 特有）
  3. A4-事件循环（混合高地，nextTick 依赖微任务）

限定词 = 无时的高地排序：
  1. A1-浏览器渲染管线（通用高地）
  2. A4-事件循环（通用高地）
  3. A5-虚拟化算法（通用高地）
```


### 3.3 宽泛限定词的战略价值

当限定词为宽泛类型（"前端网络工程""渲染性能"）时：
- 命题完全由通用内核驱动，无特化层干扰
- 战略高地 = 纯通用高地（耦合度全部 = 1）
- 学到的能力可以平移到任何框架/工具

## 三、修炼路径生成

### 4.1 依赖拓扑排序

将战略高地按依赖关系进行拓扑排序，生成最优学习顺序：

```
输入：
  A1（无依赖）
  A2（依赖 A1）
  A3（依赖 A2）
  A4（依赖 A1）
  A5（依赖 A1, A6）
  A6（无依赖）

拓扑排序输出：
  第 1 层：A1, A6（无依赖，并行学习）
  第 2 层：A2, A4（依赖 A1）
  第 3 层：A3（依赖 A2）
  第 4 层：A5（依赖 A1 + A6）
```

### 4.2 最大覆盖优先

在同层能力中，优先学习扇出度最高的：

```
第 1 层：A1（扇出 6）先于 A6（扇出 3）
第 2 层：A2（扇出 5）先于 A4（扇出 4）
```

### 4.3 输出格式

```markdown
## 修炼路径

### 第 1 步：A1-浏览器渲染管线（覆盖 6/7 命题）
- 学习内容：CRP、Layout/Paint/Composite 触发条件、性能瓶颈定位
- 工具：Chrome DevTools Performance 面板
- 验证：能独立分析一个万级 DOM 页面的渲染瓶颈

### 第 2 步：A2-DOM生命周期（覆盖 5/7 命题）
- 前置：A1
- 学习内容：DOM 节点创建/挂载/卸载/回收、Detached DOM
- 工具：Chrome DevTools Memory 面板
- 验证：能用 Heap Snapshot 定位 DOM 泄漏
```

## 四、战略高地与后处理的衔接

战略高地影响后处理的研究策略：

### 5.1 高地驱动的研究优先级

```
如果命题 X 包含 2+ 个一级高地 → 优先研究 X（一次研究修炼多个高地）
如果命题 X 只包含三级营地   → 延后研究（专项知识，按需学习）
```

### 5.2 高地驱动的研究深度

对包含战略高地的命题，四象限研究应加重对应象限的深度：

| 高地 | 对应象限 | 研究重点 |
|------|---------|---------|
| 浏览器渲染管线 | 象限 I 链路解构 | 从 CRP 到 Composite 的完整链路 |
| DOM 生命周期 | 象限 II 极致场景 | Detached DOM、GC 压力的边界 Case |
| 事件循环 | 象限 III Trade-off | rAF vs setTimeout vs requestIdleCallback |
| HTTP 缓存 | 象限 I 链路解构 | 三层缓存（浏览器+CDN+SW）的协作链路 |

### 5.3 通用高地 vs 特化高地的内容比例

后处理产出中：
- **通用高地内容** ≥ 70%（原理层，框架无关，长期有效）
- **特化高地内容** ≤ 30%（实现层，框架相关，随版本变化）

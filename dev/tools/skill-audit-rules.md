# Skill 文件审查规则

> 本文件定义 skill 文件的结构化审查规则。Agent 可读取本文件对任意 skill 进行自动化审查。

---

## 规则 1：执行文件只含 what & how，不含 why

### 原则

skill 的执行文件（processes/、SKILL.md、meta/、core/、plugins/）必须只包含 **what（做什么）** 和 **how（怎么做）**。所有 **why（为什么这样做）** 的内容必须放在 design/ 目录下。

**理由**：why 泄漏到执行文件会影响 LLM 的预测机制——它会开始"思考设计"而不是"执行指令"。

### "why" 的判定标准

以下类型的内容属于 why，不应出现在执行文件中：

| 类型 | 特征 | 示例 |
|------|------|------|
| 设计哲学 | 解释设计理念的宣言式语句 | "不是读完就懂，而是做到才算" |
| 设计理由 | 解释为什么某个参数/规则是这样 | "选择 W=5 是因为 LLM 的上下文窗口限制" |
| 替代方案比较 | 讨论为什么不用另一种方案 | "原方案是 X，改为 Y 是因为..." |
| 性能基准 | 展示不同方案的性能对比数据 | "旧方案 15min，新方案 10min" |
| 架构演进历史 | 说明某个设计从什么变为什么 | "原术语叫滑动窗口，改为并发池" |
| 修辞性反问 | 用反问强化设计意图 | "该命题是否真的如此高价值？" |
| 自我定位声明 | 解释文件/模块的角色定位 | "本 plugin 定义的是正常研究模式" |

### 豁免条件

以下内容**不算** why 泄漏，可以保留在执行文件中：

| 类型 | 理由 | 示例 |
|------|------|------|
| _trace 字段 | 记录决策凭据是 what（做什么） | `priority_trace: "总分10..."` |
| 异常处理 | 告诉 agent 遇到问题怎么做 | "文件不存在 → 标注缺失并跳过" |
| 约束说明 | 告诉 agent 操作的边界 | "每组上限 5 个能力" |
| 格式说明 | 告诉 agent 输出格式 | "按以下 JSON 结构产出" |
| 示例代码/JSON | 给 agent 看的参考格式 | output-contracts.md 中的示例 |

### 审查方法

```
1. 遍历执行层文件（processes/*.md、SKILL.md、meta/*.md、core/*.md、plugins/*.md）
2. 对每段内容，判断：这段在告诉 agent "做什么/怎么做" 还是 "为什么这样做"？
3. 如果是 why：标记文件名、行号、泄漏类型、建议处理方式
4. 输出审查报告
```

### 处理方式

| 泄漏类型 | 处理 |
|---------|------|
| 设计哲学 | 删除（agent 不需要知道） |
| 设计理由 | 移至 design/（人类需要知道） |
| 替代方案比较 | 移至 design/（人类需要知道） |
| 性能基准 | 移至 design/（人类需要知道） |
| 架构演进历史 | 移至 design/（人类需要知道） |
| 修辞性反问 | 改写为纯 what 陈述句 |
| 自我定位声明 | 删除（agent 不需要知道） |

## 规则 2：_trace 字段规范

### 原则

关键决策（影响后续产出、可能导致内容丢失的决策）必须在产物中保留决策凭据。

### 规则

1. **命名**：在决策字段旁添加同名 `_trace` 后缀字段
2. **内容**：记录做出该决策时的原始依据，包含：
   - 输入数据是什么
   - 判定标准是什么
   - 为什么选了这个值而不是其他值
3. **粒度**：只给关键决策字段加 `_trace`，客观事实字段和可重算字段不加
4. **不影响消费方**：`_trace` 字段是可选的，下游步骤不依赖它

### 审查方法

```
1. 读取 00-shared.md 中的 _trace 清单
2. 对清单中每个步骤/字段，检查对应 process 文件：
   a. 执行步骤中是否要求写入 _trace
   b. 校验清单中是否检查 _trace 的存在
3. 读取产物（如果存在），检查 _trace 字段：
   a. 是否存在
   b. 是否非空
   c. 是否包含三要素（输入/标准/选择）
4. 输出审查报告
```

## 审查报告模板

```markdown
# Skill 审查报告

**审查对象**：{skill 名称}
**审查时间**：{日期}
**审查规则**：what&how / _trace 规范

## what&how 审查

| # | 文件 | 行号 | 泄漏内容 | 泄漏类型 | 建议处理 |
|---|------|------|---------|---------|---------|
| 1 | ... | ... | ... | 设计哲学 | 删除 |

**统计**：N 处泄漏，涉及 M 个文件

## _trace 审查

| 步骤 | 字段 | 执行步骤要求 | 校验清单检查 | 产物存在 | 质量 |
|------|------|------------|------------|---------|------|
| ① scan | source_tier_trace | ✅ | ✅ | N/A | N/A |

**统计**：N/M 项通过

## 综合评估

- what&how 纯净度：X%
- _trace 覆盖率：X%
- 整体评级：...
```

# 核心方法论设计 rationale

> 本文档存放从 `core/` 迁出的概念定义与设计原理，供人类阅读理解设计意图。
> Agent 执行时不需要读此文件。

---

## 架构分词（Architecture Decomposition）

### 概念定义

**架构分词**——对一个技术命题，识别其内部的"通用工程层"和"框架特化层"的结构关系。

### 核心原则

**分词是标注，不是拆分。**

一个命题经过分词后仍然是一个命题，只是内部结构变得清晰：

```
分词前：Vue3 虚拟列表（一个模糊的整体）
分词后：Vue3 虚拟列表
        ├── 通用内核：算法+浏览器渲染+内存管理（深度来源）
        └── Vue 特化：Composition API 实现（框架载体）
```

通用内核和 Vue 特化不是两个独立命题，而是**同一个命题的两个层次**。

### 为什么需要分词

不做分词的后果：
- **跨栈耦合被低估**：框架标签遮蔽了底层技术深度（"Vue 虚拟列表"看起来只跨 2 层，实际跨 4+ 层）
- **后处理深度失焦**：四象限研究在框架层打转，没有触及真正的工程难点
- **产出比例失调**：不知道通用内容和框架特化内容应该各占多少

### 分词对评分的影响

架构分词直接影响四维评分中的**跨栈耦合**维度：

| 场景 | 未分词评分 | 分词后评分 | 原因 |
|------|----------|----------|------|
| "Vue3 虚拟列表" | 跨栈耦合 2 | 跨栈耦合 3 | 通用内核跨算法+浏览器+内存+交互 4 层 |
| "Vue 微前端" | 跨栈耦合 3 | 跨栈耦合 3 | 微前端本身就是跨栈场景，不因 Vue 标签改变 |
| "Vue keep-alive" | 跨栈耦合 2 | 跨栈耦合 2 | 通用层仅跨组件+路由+生命周期，无更深层内核 |

**规则**：分词后如果发现被限定词遮蔽的通用工程层，跨栈耦合分数应上调。命题本身不拆分，只是评分更准确。

### 分词对后处理产出的影响

分词后，四象限研究和系列文章的结构应遵循：

```
开篇（10-15%）：
  从限定词（Vue）开发者的真实痛点切入
  "你用 Vue 做过万级列表吗？遇到过滚动卡顿吗？"
  → 引出问题，建立读者共鸣

主体（70-80%）：
  通用工程内核的深度展开
  · 算法原理（不是 Vue 特定的）
  · 浏览器渲染机制（不是 Vue 特定的）
  · 内存管理策略（不是 Vue 特定的）
  · 通用避坑清单
  → 这是核心价值所在

收尾（10-15%）：
  回到限定词（Vue）的生态实践
  · Vue 生态中的最佳工具选择
  · Vue 特定的 API 使用技巧
  · 完整 Vue 实现代码
  → 让读者知道"在 Vue 中怎么落地"
```

### 与传统技术文章的区别

| 传统 Vue 文章 | 分词后的文章 |
|-------------|------------|
| 80% 讲 Vue API | 80% 讲通用工程 |
| "Vue 虚拟列表怎么用" | "虚拟列表的原理是什么 + Vue 怎么实现" |
| 换个框架就过时 | 框架换了内核还在 |
| 适合 Vue 初学者 | 适合想深入理解机制的 3-5 年开发者 |

## 原子能力图谱（Capability Graph）

### 概念定义

**原子能力（Atomic Capability）**——一个命题内部不可再分的独立技术能力单元。

特征：
- **独立可学习**：可以脱离具体命题单独理解和掌握
- **跨命题复用**：同一个原子能力出现在多个不同命题中
- **限定词无关**：大部分原子能力不依赖特定框架/工具（少数特化层能力除外）

### 与架构分词的关系

```
架构分词：命题 → [通用内核] + [特化层]        （标注结构）
原子能力图谱：[通用内核] + [特化层] → {能力集合}  （提取原子）
```

分词是"看清楚"，能力图谱是"拆出来"。

### 扇出度设计理由

扇出度被用作学习优先级指标。被引用越多，说明它是越底层、越通用的"基础能力"，掌握它的 ROI 最高。使用绝对计数（fanout.count）而非百分比，是因为绝对数直接反映该能力能解锁的命题数量。

### 为什么要提取

- **战略高地识别**：扇出度最高的原子能力 = 修炼 ROI 最高的能力
- **修炼路径规划**：基于能力依赖关系生成最优学习顺序
- **限定词耦合度分析**：区分"框架无关能力"和"框架绑定能力"

## 战略高地（Strategic Highground）

### 概念定义

**战略高地**——被多个命题共享的原子能力，掌握它能以最小成本解锁最多命题的理解。

与四维评估矩阵的区别：
- 四维矩阵评估**单个命题**的价值（值不值得研究）
- 战略高地评估**原子能力**的价值（值不值得修炼）

一个高分命题可能不包含任何战略高地（命题价值高但能力都是专项的）。
一个中分命题可能包含多个战略高地（命题本身一般但内含通用能力）。

### 战略价值计算

战略价值 = fanout.count × (1 / 限定词耦合度)

使用 fanout.count（绝对计数）而非百分比，是因为绝对数直接反映该能力能解锁的命题数量，更直观地衡量学习 ROI。

## 复合工程场景（Composite Engineering Scenarios）

### 概念定义

**复合工程场景**——跨技术层、无单一权威文档、需真实项目经验才能深入理解的工程组合问题与 Trade-off 模式。

特征：
- **跨栈耦合**：问题链路横跨 2+ 技术层，单独理解任何一层无法形成有效解法
- **文档真空**：不存在单一权威文档覆盖全链路（属于"工程组合"而非"技术规范"）
- **经验依赖**：必须通过真实项目实践才能深入理解，凭空学习效率极低

## 设计优势

**结构化而非堆砌** — 不是给你一堆链接，而是经过扫描→分级→分词→研究→组装的完整管线产出。每个命题有链路解构、坑点提取、方案对比、可运行实验，四个维度交叉验证。

**能力图谱驱动** — 原子能力之间的依赖关系决定了学习顺序，不是拍脑袋排的。扇出度和耦合度量化了每个能力的战略价值。

**并行且可控** — 后处理阶段用滑动窗口并行（默认窗口=4），每个 agent 只负责一个文件，失败可精确重跑，不影响其他。

**学习阶梯不是教程** — 不重写内容，而是给读者一个"入口"和"抓手"。带着问题去读现有产出，比从头到尾读一遍更高效。

**增量复用** — 已研究过的能力会跳过，已生成的 briefing 会复用，支持断点续跑。

**零依赖** — 纯 Markdown，无需构建、无需服务器进程、无需安装任何东西。

## 风险与局限

### 产物质量依赖 agent 能力

核心逻辑是"编排 agent 去做研究"。研究深度、信源质量、代码正确性都受限于执行 agent 的模型能力。不同模型、不同上下文窗口下，产出质量可能差异显著。

### 信源获取不稳定

前处理的 `capability-extract` 阶段需要大量 web_fetch 验证信源 URL。网络不通、反爬策略、信源失效都会导致信源缺失。后处理有 fallback 搜索机制，但不保证覆盖。

### 实验代码未经充分验证

`experiment/` 目录下的代码由 agent 生成，可能存在 bug、兼容性问题或过时的 API 调用。请在隔离环境中运行，不要直接用于生产。

### 上下文窗口限制

单 agent 的上下文消耗在 50-80KB 量级。对于特别复杂的命题（涉及 10+ 原子能力），可能需要分批处理或降低研究深度。

### 平台依赖

依赖平台的多 agent spawn 能力。在不支持 spawn 的平台上，后处理将降级为单线程执行，失去并行优势。

## 经验年限 — 命题颗粒度匹配原理

### 设计决策

经验年限不直接映射命题难度，而是映射**命题的颗粒度和社区常见度**。

- 年限越短 → 命题越粗、出现频率越高、社区讨论越充分
- 年限越长 → 命题越细、出现频率越低、越需要主动挖掘

### 理由

后处理的四象限框架负责深度放大，前处理只需按规则筛选命题颗粒度。难度由能力依赖链深度决定（Step ⑤），不由经验年限决定。

# 待定设计：动态并发调整 W

> 状态：挂起。待有线上超时数据后再决策。

## 问题背景

当前所有后处理步骤的并发槽位 W 硬编码为 5（`core/shared-conventions.md` §全局参数）。

线上实际运行时，4 个头脑风暴维度 Agent 同时 spawn 可能触发 LLM API 并发限流（非 agent 并发上限，而是 API provider 的速率限制），导致部分 agent 超时。

## 当前已修复的兜底

`processes/00-brainstorm.md` §3.1.3.1（2026-05-29）：
超时后先检查磁盘文件是否完整，完整则保留，不完整则补发一次。
这解决了"超时后错误丢弃有效产出"的问题，但没有解决"为什么会超时"。

## 待定方案：动态 W 调整

### 核心思路

W 不再是固定常量，而是根据超时信号动态调整：

```
初始 W = 5
检测到 ≥1 个 agent 超时 → 下一轮 W = max(2, W - 2)
连续 N 轮无超时 → W 回升 1（上限 5）
W 降到 2 仍超时 → 串行执行（W=1）+ 告警用户
```

### 需要解决的设计问题

**1. 反馈环的写入点**

W 的状态需要在步骤间传递。三种选型：

| 方案 | 位置 | 优点 | 缺点 |
|------|------|------|------|
| A. 共享约定接口 | `shared-conventions.md` 定义 W 的读写规则，每步开始时读、结束时更新 | 集中管理，一步改全局 | 所有 steps 都要遵守新协议 |
| B. Step-local | 每个 step 自己检测超时并调整 W | 无跨步依赖，改动局部 | 逻辑重复，W 在步骤间不传递 |
| C. 串行降级 | 不改 W，检测到超时后剩余任务串行 | 最简单，无状态管理 | 丧失并行优势 |

**2. 调度模式差异**

- 简单窗口（⑤⑥⑦）：W 直接控制 spawn 数量，调整 W 即可
- DAG 调度（④）：W 控制每批次的最大并行数，调整 W 影响批次粒度
- 头脑风暴（⓪）：W=4 一次性全部 spawn，不涉及动态调整

**3. 验证困难**

- 正常运行时 W=5 无超时 → 看不出动态调整的价值
- 需要模拟 API rate limit → 依赖外部环境，不可控
- 间接验证：对比"有/无动态调整"在限流条件下的成功率 → 需要多次实验

# 前处理——命题捕获与筛选

> 本文件定义前处理流程。后处理流程见 `deepResearch.md`。
> 共享评估框架见 `core/scenario-matrix.md`。

---

## 一、流程总览

```
信息源（互联网）
    │
    ▼
┌─────────────┐
│  广域扫描    │  Agent 爬取/监控多个技术信息源
└──────┬──────┘
       │ raw topics
       ▼
┌─────────────┐
│  命题提取    │  从原始素材中识别出独立的技术命题
└──────┬──────┘
       │ extracted topics
       ▼
┌─────────────┐
│  资质筛选    │  比对四维评估标准（见 core/scenario-matrix.md §二）
└──────┬──────┘
       │ qualified candidates
       ▼
┌─────────────┐
│  入池归档    │  写入候选池，等待后处理消费
└─────────────┘
```

## 二、触发方式

```
扫描：<信息源描述>
```

或：

```
deep scan：<信息源描述>
```

**示例：**
- `扫描：掘金/SegmentFault 近一周前端热门文章`
- `扫描：GitHub trending 近期 JavaScript/TypeScript 仓库`
- `扫描：前端面试题相关的新帖（V2EX、牛客、知乎）`
- `扫描：Hacker News 前端相关讨论`

**可选参数：**

| 参数 | 说明 |
|------|------|
| `--source=<url>` | 直接指定一个具体 URL 进行分析 |
| `--feed=<topic>` | 按主题订阅式扫描（如 `前端性能`、`跨端方案`） |
| `--digest` | 对已入池的候选做一次摘要汇报（不扫描新内容） |

## 三、信息源分级

> 信源可信度直接影响后续知识资产的质量。本节按 Tier 分级，Agent 必须在扫描和提取阶段标注来源层级。

| Tier | 级别 | 典型来源 | 用途 | 可信度权重 |
|------|------|---------|------|-----------|
| T1 | 权威/底层 | W3C/WHATWG 规范、Chromium 官方博客、Node.js Release Notes、框架官方文档与高赞 Issue | 事实来源，后处理交叉验证的**强制引用源** | 1.0 |
| T2 | 高质量工程实践 | Cloudflare/Vercel/字节/美团/阿里等技术团队博客、InfoQ 深度文章 | 事实来源的补充验证 | 0.8 |
| T3 | 社区讨论 | 掘金、SegmentFault、知乎、StackOverflow、V2EX | 仅作为**热点风向标**，不作为核心事实来源 | 0.3 |

### A. 社区动态类（T3）
掘金、SegmentFault、V2EX 前端节点、牛客面经区、知乎前端话题

### B. 开源动态类（T2-T1）
GitHub Trending（JS/TS/Dart/Rust）、GitHub Discussions、npm 下载量飙升包、框架官方 Issue 区高赞回答（T1）

### C. 面试与招聘类（T3）
牛客/力扣讨论区面试经验、招聘 JD 中反复出现的技术关键词

### D. 技术博客与深度内容（T1-T2）
W3C/Chromium 官方博客（T1）、大厂技术团队博客（T2）、个人技术博客（T2-T3）

### E. 议题驱动类
用户直接指定的 URL 或主题（`--source` 模式），Agent 需根据内容质量自行判定 Tier

## 四、命题提取规则

从原始素材中提取命题，遵循 `core/scenario-matrix.md §四` 的命名规范与粒度原则。

若扫描指令包含经验年限约束（如"3-5年经验""高级""资深"），按 `plugins/year-granularity.md` 中对应阶梯的筛选规则和命题粒度进行过滤。

## 四·五、架构分词（必选步骤）

**命题提取后、资质筛选前**，必须对每个候选命题执行架构分词。

执行方法见 `core/architecture-decomposition.md`。核心步骤：

1. **识别限定词与技术栈关键词**：限定词 = 指定框架/平台的词（Vue/React/小程序），技术栈关键词 = 指向底层工程机制的词（虚拟列表/缓存/沙箱）
2. **向上展开依赖链**：技术栈关键词依赖什么底层能力？
3. **向下展开被依赖链**：技术栈关键词被哪些上层场景使用？
4. **分层标注**：标注 [通用内核] 和 [特化层]，**不拆分命题**，保持为一个完整命题
5. **确定内容权重**：通用内核占比 ≥ 80% / 60-80% / 40-60% / < 40%

**分词对评分的影响**：
- 如果分词后发现被限定词遮蔽的通用工程层，**跨栈耦合分数应上调**
- 如果分词后通用层为空或极浅，按框架特化命题处理，跨栈耦合 ≤ 1

**关键约束**：分词是标注，不是拆分。一个命题分词后仍然是一个命题，只是内部结构（通用内核 + 特化层）变得清晰。候选池中不出现"通用版"和"特化版"两个独立条目。

**示例**：
```
原始命题："Vue3 虚拟列表"
限定词：Vue3 | 技术栈关键词：虚拟列表
分词标注：
  [通用内核] 算法+浏览器渲染+内存管理+滚动交互（4 层）
  [Vue 特化] Composition API + @tanstack/vue-virtual
内容权重：通用 ≥ 80%
评分修正：跨栈耦合 2 → 3
入池：一个命题（不拆分），带结构标注
```

## 五、资质筛选

使用 `core/scenario-matrix.md §二` 的四维评估矩阵和入池阈值进行判定。

### 交叉验证规则（强制）

1. 前处理扫描可使用 T3 信源识别热点
2. 任何进入候选池的命题，Agent 必须在提取阶段标注：
   - 已找到的 T1/T2 佐证来源（URL 或来源名）
   - 若无 T1/T2 佐证，命题标注为 `⚠️ 待验证`，优先级上限为 medium
3. 后处理执行时，必须引用至少 1 个 T1 或 2 个 T2 来源
   - 无法满足时，在 `references.md` 中标注 `⚠️ 信源不足，结论可信度受限`

## 六、候选池结构

所有通过筛选的命题，写入 `workflow/research/candidates.md`：

```markdown

| # | 命题 | 来源 | 信源佐证 | 四维评分 | 优先级 | 捕获时间 | 备注 |
|---|------|------|---------|---------|--------|---------|------|
| 1 | ... | T3-掘金 | T1-Chromium Blog ✓ | 9/12 | high | ... | ... |
| 2 | ... | T3-知乎 | 无 T1/T2 ⚠️ | 6/12 | medium | ... | 待验证 |

## 已研究 (Done)

| # | 命题 | 研究目录 | 完成时间 |
|---|------|---------|---------|
| ... | ... | workflow/research/<slug>/ | ... |

## 已淘汰 (Rejected)

| # | 命题 | 扫描时间 | 淘汰理由 |
|---|------|---------|---------|
| ... | ... | ... | ... |
```

## 七、后处理衔接

候选池中的命题，由以下方式触发后处理：

| 方式 | 指令 | 说明 |
|------|------|------|
| 手动 | `研究：<命题名称>` | 调用 deepResearch.md 流程 |
| 批量 | `研究：--batch=pending` | 按优先级依次处理所有 pending |
| 条件 | `研究：--filter="优先级=high"` | 按条件筛选后批量执行 |

## 八、完整流水线示例

```
用户：扫描：掘金近一周前端热门文章
        │
        ▼

        │

        │
        ▼

        │
        ▼
用户：研究：AI 聊天窗口的虚拟长列表渲染
        │
        ▼
调用 deepResearch → 产出写入 workflow/research/ai-chat-virtual-list/
candidates.md 状态更新：pending → done
```
